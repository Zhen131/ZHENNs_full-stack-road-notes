# 账本项目上下文启动系统设计

日期：2026-06-25  
状态：待用户书面审核  
目标：让新聊天框通过一句任务描述自动获得必要上下文，减少用户重复说明文件位置、当前进度和工作方式的成本。

## 结论

采用 `AGENTS.md + 项目级 Skill + 当前开发状态 + AI 指南修订` 的四件套。

`AGENTS.md` 保存始终生效的项目规则；Skill 根据任务模式按需加载材料；当前开发状态提供跨聊天框的最小进度快照；AI 指南记录长期目录语义和维护规则。

用户以后主要使用：

```text
账本写代码：<今天的任务>
账本 Git：<今天的任务>
账本讨论：<今天的任务>
账本继续：<可选补充>
账本日志：<今天的记录目标>
```

也允许用户直接描述任务。AI 应从语义判断模式，不要求严格口令。

## 动机

选择分层上下文入口，因为单一说明文档会过时，单一 Skill 的触发不够稳定，而四件套可以同时保证自动加载、按需读取和进度延续。

## 当前目录语义

| 路径 | 性质 | 默认处理 |
| --- | --- | --- |
| `00frappe-books-typescript/` | 外部参考项目，不是用户账本源码 | 默认不读，仅在架构参考或用户明确要求时读取 |
| `00开始前的一些准备文件/` | 用户的长期规划、需求和项目说明 | 按任务选择性读取 |
| `01一些进度/产出/LocalFirstTradingLedger/` | 用户账本源码，且是独立 Git 仓库 | 写代码和源码 Git 任务的主要工作区 |
| `01一些进度/日志/` | 周计划、设计、验收和开发日志 | 优先读取当前状态与当前周文件 |
| `02_NLP/` | NLP 课程及后期扩展材料 | 默认不读，NLP 任务才读取 |
| `99_tool/` | 工具和 Skill 记录 | 工具维护任务才读取 |
| `00开始前的一些准备文件/99-私人网络环境说明.md` | 私人资料 | 永不主动读取、引用或提交 |
| `*.canvas` | 图形化架构材料 | 仅在架构图或相关设计任务中读取 |

根目录与 `LocalFirstTradingLedger/` 各自是 Git 仓库。Git 任务必须先识别目标仓库，不得混用状态、分支或提交。

## 系统组成

### 1. 根目录 `AGENTS.md`

职责：

- 声明整个工作区围绕账本项目展开。
- 解释主要目录的性质和默认读取边界。
- 指定源码仓库和文档仓库的区别。
- 要求先判断任务模式，再读取有限上下文。
- 保护私人文件。
- 规定默认沟通风格、验证要求和进度收尾动作。

边界：

- 保持简短，只放始终适用的规则。
- 不复制周计划、需求细节或长篇架构说明。
- 不保存容易频繁变化的当前进度。

### 2. `.agents/skills/ledger-workflow/SKILL.md`

职责：

- 在用户提到账本开发、账本 Git、账本讨论、继续账本或账本日志时触发。
- 将任务归类为一种主要模式。
- 按模式选择文件和工具。
- 规定开始、执行、验证和收尾流程。

模式路由：

| 模式 | 默认入口 | 按需扩展 |
| --- | --- | --- |
| 写代码 | 当前状态、源码 README、源码 Git 状态、相关代码与测试 | 当前周计划、相关设计文档 |
| Git | 目标仓库 Git 状态、分支、近期提交、diff | 当前状态、任务相关文档 |
| 讨论 | 当前状态、相关需求或设计文档 | 相关代码、Frappe Books、Canvas |
| 继续 | 当前状态、当前周计划、README、两个仓库状态 | 最近日志和相关代码 |
| 日志 | 当前状态、当前周计划、实际 Git 变更 | 文档产出规范、测试结果 |

任务无法明确分类时，AI 应依据用户描述选择最接近的模式；只有当不同选择会导致明显不同结果或写入范围时才提问。

### 3. `01一些进度/日志/00-当前开发状态.md`

职责：

- 提供跨聊天框的最小进度快照。
- 让“账本继续”不依赖历史聊天记录。
- 为代码、讨论和日志任务提供统一当前入口。

固定内容：

```text
当前日期与阶段
当前里程碑
已完成
正在进行
下一步
阻塞或风险
关键入口文件
最近验证结果
最后更新时间
```

维护规则：

- 只写当前有效状态，不累积成长日志。
- 实质进展、里程碑变化或下一步变化后更新。
- 普通讨论、只读检查或无状态变化时不更新。
- 历史过程仍写入对应 Week 日志。

### 4. AI 使用指南修订

职责：

- 保留长期目标、范围判断和项目原则。
- 将固定阅读顺序改为任务驱动路由。
- 明确外部参考源码、用户源码、日志和私人文件的边界。
- 指向 `AGENTS.md`、Skill 和当前状态文件，避免重复维护同一规则。

## 启动数据流

```text
用户一句任务
→ Codex 自动读取 AGENTS.md
→ Skill 根据描述显式或隐式触发
→ 识别工作模式和目标 Git 仓库
→ 读取 00-当前开发状态.md
→ 读取该模式的最小入口材料
→ 搜索并读取任务直接相关文件
→ 执行讨论、代码、Git 或日志工作
→ 验证结果
→ 必要时更新当前状态
```

## 上下文预算规则

默认禁止：

- 启动时遍历整个工作区。
- 一次读取全部历史日志。
- 默认读取 Frappe Books 全仓库。
- 因为存在 Canvas 就自动解析所有 Canvas。
- 为了“了解背景”读取 NLP、选课或私人资料。

默认允许：

- 查看目录的有限层级以定位文件。
- 读取当前状态、源码 README、当前周计划。
- 查看目标仓库的 Git 状态和近期提交。
- 使用文件搜索定位与用户任务直接相关的代码或文档。

扩展读取条件：

- 当前入口信息不足以安全执行。
- 用户明确要求比较历史方案或外部参考。
- 测试失败、代码依赖或架构关系要求继续追踪。

## 异常与过期处理

| 情况 | 处理 |
| --- | --- |
| 当前状态文件不存在 | 从 README、当前周计划和 Git 状态重建，并提示将创建状态文件 |
| 当前状态与 Git/代码冲突 | 以可验证的 Git 和代码事实为准，修正状态文件 |
| 用户口令和具体任务冲突 | 以具体任务语义为准，并简短说明采用的模式 |
| 无法确认目标 Git 仓库 | 先检查两个仓库；只有仍有歧义时询问用户 |
| Skill 未隐式触发 | `AGENTS.md` 仍要求遵循基本路由；用户也可显式说“账本……” |
| 历史文档与当前计划冲突 | 当前状态和当前周计划优先，长期总纲用于范围校验 |

## 实施边界

本次实施包括：

- 新建根目录 `AGENTS.md`。
- 新建项目级 `ledger-workflow` Skill。
- 新建并初始化当前开发状态文件。
- 修订现有 AI 使用指南。
- 验证文件位置、触发描述和路由规则。

本次不包括：

- 重命名或搬迁现有目录。
- 整理全部历史日志。
- 修改账本业务代码。
- 修改两个 Git 仓库的结构。
- 创建自动定时任务。
- 将 Skill 发布成插件。

## 验收标准

| 场景 | 通过标准 |
| --- | --- |
| `账本写代码：完成未实现盈亏` | AI 先看当前状态和源码入口，再定位计算器与测试；不读取全部日志 |
| `账本 Git：整理当前版本` | AI 先确认目标仓库，再检查状态、diff 和提交历史 |
| `账本讨论：IndexedDB 放在哪层` | AI 读取相关架构材料；仅在需要比较时读取 Frappe Books |
| `账本继续` | AI 能从当前状态、当前周计划和 Git 事实给出下一步 |
| 直接说具体任务 | AI 能语义分类，不强迫用户使用固定格式 |
| 无关资料保护 | 默认不读取 NLP、选课、私人网络资料和无关 Canvas |
| 状态维护 | 实质进展后更新当前状态；无进展时不制造日志 |

## 测试方法

实施后进行静态和场景验证：

1. 检查 `AGENTS.md` 是否简短且没有复制动态进度。
2. 检查 Skill 的名称、描述、触发词和模式路由是否明确。
3. 检查所有引用路径实际存在。
4. 用五类示例任务模拟读取路径，确认不会默认扫描全仓库。
5. 检查当前状态是否与源码 README、当前周计划和 Git 历史一致。
6. 检查私人文件保护规则在 `AGENTS.md` 和 Skill 中一致。

## 用户日常用法

推荐格式：

```text
账本写代码：今天实现未实现盈亏
账本 Git：检查当前改动并整理提交
账本讨论：页面内存状态应该怎么组织
账本继续
账本日志：总结今天完成的 validator 和测试
```

用户不需要再说明：

- 源码在哪个目录。
- 当前日志在哪个目录。
- Frappe Books 是外部参考。
- 应该先看哪个总纲。
- Git 仓库如何区分。
- 哪些私人文件不能读取。

用户仍需说明当天真正想完成的目标；系统负责定位上下文，不替用户决定产品方向。
