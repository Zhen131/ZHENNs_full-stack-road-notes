对比对象：

- 旧版：`01一些进度/日志/Z99Archive/000-账本核心架构与数据流图-横向泳道版.canvas`

## 先结论

新版不是把旧图改成 Week 4 的局部图，而是在保留总项目视角的基础上，把 Week 4 以后真正要走的状态链路补清楚了。

这次新版的核心进步是：

```text
外部输入
-> Draft
-> normalize / validator / service
-> UI result handling
-> ledger actions
-> ledgerReducer
-> LedgerData
-> positionService / Repository / 后续持久化
```

旧版已经能表达账本长期架构的大方向，但有几个问题：

- `React state / ledgerReducer` 还是一个概括节点，没有把 `initialLedgerData`、action、UI 成功/失败处理拆开。
- 外部输入来源不够清楚，容易误解成表单、NLP、导入文件可以直接生成 `Trade`。
- 价格输入链路不够清楚，`PriceProvider`、`PriceSnapshotDraft` 和未来 API 价格源没有独立表达。
- 读取链路里有“安全读取路径”和“读取账本”两个说明节点，表达有重复。
- 图没有颜色语义，节点很多时只能靠文字读，阅读成本高。

新版解决的是总路线图的表达问题，不代表 Week 4 源码已经完成。源码验收仍然以 `01B_W4-useReducer状态地基执行与验收标准.md` 和 `00_W4-Week4-Checklist.md` 为准。

## 新版新增的关键表达

### 1. 补清了外部输入入口

新版新增 `TradeInputSource / parser`：

```text
manual form / future NLP / future CSV / exchange import / future Agent draft
-> TradeDraft
```

意义：

- 表单只是输入来源之一，不是唯一来源。
- NLP、CSV、交易所导入、Agent 草稿都被放进未来同一入口。
- 任何来源都不能直接生成正式 `Trade`，必须先进入 Draft 和校验链路。

这比旧版只画“components / app -> 输入表单 -> Draft models”更准确。

### 2. 补清了价格来源入口

新版新增 `PriceProvider / price source`：

```text
manual input / future API
-> PriceSnapshotDraft / future PriceQuote
```

意义：

- 手动价格和未来 API 价格都不直接写入 `LedgerData`。
- 外部价格必须先转换为系统内部价格对象，再进入 validator / service。
- `PriceSnapshot` 和未来实时行情源之间有了边界，不会混成一件事。

### 3. 把 Week 4 的状态地基拆成可执行链路

旧版只有 `React state / ledgerReducer` 一个节点。新版拆成：

```text
initialLedgerData
-> ledgerReducer

UI result handling
-> ledger actions
-> ledgerReducer
-> LedgerData
```

新增节点：

- `initialLedgerData`
- `ledger actions`
- `Runtime state / ledgerReducer`
- `UI result handling`

意义：

- `initialLedgerData` 只提供页面运行态初始账本，不替代 Repository / IndexedDB。
- `ledger actions` 只表达“要改账本什么地方”。
- `ledgerReducer` 只负责状态变化，不校验、不计算、不直接读写存储。
- Service 返回成功/失败后，由 UI 决定是否 dispatch，失败不会污染 `LedgerData`。

这正好对应 Week 4 的核心方向：`useReducer + LedgerData`。

### 4. 把 `LedgerData` 的事实边界讲得更准

旧版 `LedgerData` 写的是“本地保存的整坨账本数据”。新版改成：

```text
账本事实容器。
运行时由 ledgerReducer 管理；
持久化时由 Repository / StorageAdapter 保存。
Position 不在这里保存。
```

意义：

- 同一个 `LedgerData` 同时服务运行态和持久化态，但管理者不同。
- 页面内存态由 `ledgerReducer` 管。
- 长期保存由 Repository / StorageAdapter 管。
- `Position[]` 仍然只是派生结果，不进入事实数据。

这是新版比旧版更接近最终形态的关键点。

### 5. 把 Repository 和运行态状态分开

旧版 Repository 节点主要说明“账本读写入口”。新版补充：

```text
读写 LedgerData，但不管理页面运行态。
懂账本动作，不懂 IndexedDB 细节；不直接加密 / 解密。
```

意义：

- Repository 不是 React state。
- Repository 不替代 reducer。
- Repository 不直接碰 IndexedDB 细节和加密细节。
- 未来 Week 5 接 IndexedDB 时，不会把页面状态、Repository、StorageAdapter、EncryptionService 混成一团。

### 6. 把导入和读取后的状态回流画进总图

新版新增或强化了这些动作线：

```text
Repository -> ledger actions: ledger/load
导入覆盖确认 -> ledger actions: ledger/import
ledger actions -> ledgerReducer: 状态变更
ledger actions -> Repository: 持久化动作
```

意义：

- 不是只有表单新增交易会改变账本。
- 读取旧账本、导入账本、后续价格快照写入，都可以统一进入 action/reducer 语义。
- 这让总图更接近未来真实应用，而不是只服务当前页面 demo。

## 新版纠正的问题

### 1. 纠正“单一事实来源”表达不够清楚的问题

旧版的 `React state / ledgerReducer` 能看出方向，但不够明确。

新版明确为：

```text
initialLedgerData
ledger actions
Runtime state / ledgerReducer
LedgerData
```

纠正后的规则：

- 页面运行时的当前账本事实是 `LedgerData`。
- 改账本必须走 action。
- action 进入 `ledgerReducer` 后返回新的 `LedgerData`。
- `Position[]` 不进 state。

### 2. 纠正外部来源直接入账的风险

旧版输入链路更像：

```text
输入表单 -> Draft models -> normalize / validator
```

新版把输入来源扩成：

```text
TradeInputSource / parser -> Draft models
PriceProvider / price source -> Draft models
```

纠正后的规则：

- 输入来源可以很多，但正式入账路径只有一条。
- `TradeDraft` 和 `PriceSnapshotDraft` 是候选数据，不是账本事实。
- 任何外部数据都不能绕过 validator / service 直接写 `LedgerData`。

### 3. 纠正 Service 和 reducer 的职责边界

新版新增 `UI result handling`，把这条边界写清：

```text
Service 返回 success / errors
成功：UI dispatch ledger action
失败：显示 errors，不 dispatch，不污染 LedgerData
```

纠正后的规则：

- Service 不直接改 reducer。
- reducer 不做校验。
- UI 不自己改账本对象，只 dispatch action。

### 4. 纠正读取链路的重复表达

旧版有两个容易重复的说明节点：

- `安全读取路径`
- `读取账本`

新版删除了这两个独立节点，把含义收束到：

- `StorageAdapter`
- `EncryptionService`
- `EncryptedEnvelope`
- `schemaValidator / migrator`
- `LedgerRepository`
- `ledger/load`

这样读图时不会在两个“读取说明节点”之间来回跳，链路更像真实实现路径。

### 5. 纠正 `Trade -> FeeRule` 的关系表达

旧版模型关系里，`PriceSnapshot -> FeeRule` 的连接更像“费率规则另存”，不够贴近实际类型关系。

新版改成：

```text
Trade -> FeeRule: feeRuleId
```

这更准确：

- 交易可以引用某个手续费规则。
- 手续费规则不是价格快照的下游。
- `FeeRule` 是账本事实模型的一部分，而不是价格链路的一部分。

### 6. 把范围收束从临时决策改成未来边界

旧版是 `Scope decision`，说明 `position-adjustment` 暂不进入核心。

新版改成 `Future work boundary`，更适合总路线图：

- 当前核心仍然只做 `buy / sell`。
- `position-adjustment` 进入 future work。
- 总图不应该保留太多一次性会议决策口吻。

## 新版的阅读优势

### 1. 颜色让责任域一眼可见

旧版完全没有颜色，所有节点只能逐字读。

新版用 8 类颜色区分责任域：

- 输入来源看蓝色。
- 校验规范化看橙色。
- reducer / action / 运行态看紫色。
- 事实模型看绿色。
- 计算展示看青绿色。
- 持久化安全看红色。
- 导入导出看青蓝色。
- 说明和测试看灰色。

这让图从“文字堆叠”变成“可扫描的架构地图”。

### 2. 排版更适合人眼追链路

新版把链路重新排过：

- 左侧更偏输入和写入。
- 中间更偏 reducer / LedgerData / service。
- 右侧更偏持久化、安全和 IndexedDB。
- 下方更偏模型、计算和展示。

这不是逻辑新增，但显著降低了阅读负担。

### 3. 总图仍然保留全局视角

新版不是 Week 4 附属图。

它仍然覆盖：

- 输入来源。
- Draft / normalize / validator。
- service。
- reducer / LedgerData。
- Repository / StorageAdapter / Encryption / IndexedDB。
- Import / Export。
- Asset / Trade / PriceSnapshot / FeeRule / Position。
- decimalMath / positionCalculator / display。
- future boundary / tests / performance path。

所以它适合作为根目录下的总路线图。

## 对 Week 4 Day 1 的意义

今天没有完成 Week 4 的源码验收标准。

但今天完成了 Week 4 开始前很重要的一步：把总路线图从“长期架构草图”升级成“能指导 Week 4 状态地基开发的总图”。

它对后续代码工作的帮助是：

- 写 `initialLedgerData.ts` 时，知道它只是运行态初始账本，不是存储层。
- 写 `ledgerReducer.ts` 时，知道 reducer 只处理 action，不做校验、不算持仓、不碰存储。
- 写 `positionService.ts` 时，知道它只读 `LedgerData.trades` 和 `LedgerData.priceSnapshots`，然后调用 `calculatePositions(...)`。
- 写 `tradeService.ts` 时，知道校验失败不能 dispatch，成功后才交给 UI 触发 action。
- Week 5 接 IndexedDB 时，知道 Repository、StorageAdapter、EncryptionService 的边界不能混。
