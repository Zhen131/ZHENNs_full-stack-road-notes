## 先结论

这不是开发日志，也不是周报。

本文件只记录一件事：

```text
账本源码这棵树，到目前为止先长出了什么、后长出了什么、为什么必须按这个顺序长。
```

当前源码主线是：

```text
页面空壳
-> src 分层骨架
-> models/types.ts 定义账本名词
-> utils/decimalMath.ts 统一小数计算
-> calculators/positionCalculator.ts 从 Trade[] 算 Position[]

-> Vitest 测试体系
-> state/initialLedgerData.ts 建立内存账本初始值
-> state/ledgerReducer.ts 建立账本状态变更入口

```

核心认知：

- `Trade` 是原始事实，要保存。
- `PriceSnapshot` 是价格事实，要保存。
- `Position` 是计算结果，不保存。
- `LedgerData` 是当前内存账本的总结构。
- `decimalMath.ts` 是所有精确小数计算的统一入口。
- `positionCalculator.ts` 只负责计算，不保存、不读页面。
- `tradeValidator.ts` 只负责校验不可信输入，不生成正式持仓。
- `ledgerReducer.ts` 只负责状态变化，不校验、不计算、不持久化。

### `src/models/types.ts`

这个文件是账本源码的词典。


为什么它要先长出来：

- 没有 `Trade`，后面无法写 `positionCalculator`。
- 没有 `TradeDraft`，后面无法写 `tradeValidator`。
- 没有 `Position`，后面无法表达计算结果。
- 没有 `LedgerData`，后面无法表达整本账本的状态。
- 没有 `PriceSnapshot`，后面无法计算市值和未实现盈亏。

它不负责：

- 不计算。
- 不校验。
- 不保存。
- 不渲染页面。

### `src/models/index.ts`

这个文件是类型出口。

它让其他模块可以从 `../models` 统一导入核心类型，而不是到处写 `../models/types`。

### `src/utils/decimalMath.ts`

这个文件是精确小数计算入口。

它封装了：

```text
toDecimal
toDecimalString
add
subtract
multiply
divide
compare
isZero
isPositive
isNegative
isGreaterThan
isLessThan
isEqual
absolute
isWithinTolerance
formatDecimal
```

为什么它必须在 calculator 和 validator 之前长出来：

- 加密资产数量很小，不能用 JavaScript 原生 `number` 裸算。
- `quantity * price` 会涉及小数精度。
- 平均成本、剩余成本、已实现盈亏都必须稳定。
- Validator 的成交金额容差也要用同一套小数规则。

它不负责：

- 不知道什么是 BTC / ETH / ADA。
- 不知道什么是买入 / 卖出。
- 不判断交易是否合法。
- 不保存 Decimal 对象到账本。

这一层长出来以后，后面的源码才有共同的数字地基。

## Week 2 Day 3-4：长出 positionCalculator

### `src/calculators/positionCalculator.ts`

这个文件是第一版账本的计算核心。

它的主函数是：

```ts
calculatePositions(trades, priceSnapshots)
```

输入：

```text
Trade[]
PriceSnapshot[]
```

输出：

```text
Position[]
```

它解决的问题：

- 按交易时间排序。
- 按资产聚合。
- 买入时增加数量和成本。
- 卖出时按卖出前平均成本结转成本。
- 计算剩余 `quantity`。
- 计算剩余 `costBasis`。
- 计算 `averageCost`。
- 计算 `realizedPnl`。
- 根据最新 `PriceSnapshot` 推导 `latestPrice`。
- 推导 `marketValue`。
- 推导 `unrealizedPnl`。

为什么它要在 `types.ts` 和 `decimalMath.ts` 后面长出来：

```text
types.ts 定义 Trade / PriceSnapshot / Position
decimalMath.ts 提供精确加减乘除
positionCalculator.ts 才能从 Trade[] 推导 Position[]
```

它不负责：

- 不读取 UI 表单。
- 不校验 TradeDraft。
- 不保存 Position。
- 不读写 IndexedDB。
- 不格式化展示文案。

重要边界：

```text
Position[] 是派生结果，不进入 LedgerData 保存。
```

这条边界决定了后面 `ledgerReducer` 里不能保存 `Position[]`。

### `src/calculators/positionCalculator.test.ts`

这个测试文件让计算器不是“看起来能算”，而是有固定结果证明。

它用 Week 2 的 5 条样例验证：

- BTC 持仓。
- ETH 持仓。
- ADA 多次买入。
- ADA 部分卖出。
- 成本结转。
- 已实现盈亏。
- 价格快照。
- 市值。
- 未实现盈亏。
- 无价格快照时不制造假字段。
- 超卖防御。

## Week 2 Day 5-8：长出 tradeValidator 和核心测试体系

### `src/validators/tradeValidator.ts`

这个文件是账本入口的守门员。

它的公开主函数是：

```ts
validateTradeDraft(input, context)
```

输入为什么是 `unknown`：

- 表单数据不可信。
- JSON 导入不可信。
- 未来 NLP 解析结果也不可信。
- TypeScript 类型不能证明运行时输入一定合法。

它校验的是 `TradeDraft`，不是 `Trade`：

- `TradeDraft` 是用户还没入账的草稿。
- `Trade` 是校验通过后生成的正式事实，带 `id`、`createdAt`、`updatedAt`。

它当前负责：

- `type` 必须是 `buy` 或 `sell`。
- `assetSymbol` 必须存在于合法 `Asset[]` 中。
- `quantity`、`price`、`totalValue` 必须是合法正数。
- `fee` 缺省为 `"0"`，存在时不能小于 0。
- `quantity * price` 和 `totalValue` 的差异必须在容差内。
- 卖出不能超过已有持仓。
- 普通校验失败返回结构化 `errors`，不靠 throw 表达业务失败。

它依赖：

```text
models/types.ts
utils/decimalMath.ts
```

它不负责：

- 不生成正式 `Trade.id`。
- 不保存交易。
- 不计算正式 `Position[]`。
- 不改 `LedgerData`。
- 不操作 React state。

它和 calculator 的关系：

```text
tradeValidator.ts
  负责让非法 TradeDraft 不进入账本

positionCalculator.ts
  负责用已经结构化的 Trade[] 计算 Position[]
```

### `src/validators/index.ts`

Validator 的统一出口。

以后页面、service 或导入流程不需要知道内部文件怎么拆，只需要从 validators 层拿公开能力。

### `src/validators/tradeValidator.test.ts`

这个测试文件证明守门员真的拦得住脏数据。

它覆盖：

- 非法类型。
- 资产不存在。
- 数量、价格、总额、手续费非法。
- 成交金额容差。
- 无持仓卖出。
- 超卖。
- 等量清仓。
- 多次买入后部分卖出。
- Validator 失败后不进入 calculator。

### `src/test/fixtures.ts`

这个文件是测试样例地基。

它固定了 Week 2 的 golden 输入：

```text
BTC buy
ETH buy
ADA buy
ADA buy
ADA sell
```

为什么它要长出来：

- `positionCalculator.test.ts` 和 `tradeValidator.test.ts` 要用同一套样例。
- 避免每个测试复制一遍长对象。
- 避免不同测试偷偷使用不同版本的 ADA 数据。

它不应该被生产代码导入。

### Vitest 迁移

Week 2 后期把核心测试统一到 Vitest。

这意味着当前核心源码不只是文件存在，而且有自动化证明：

```text
decimalMath.test.ts
positionCalculator.test.ts
tradeValidator.test.ts
ledgerReducer.test.ts
```

测试体系这根枝条的意义：

- 源码每长一层，就有对应测试确认边界。
- 不靠肉眼判断计算是否正确。
- 后续接页面和 service 时，可以保护 Week 2 的核心规则不被改坏。

## Week 4 Day 1：确定 useReducer + LedgerData 路线

Week 4 的核心目标是：

```text
页面不能继续靠写死数组展示账本。
```

当时确认的目标链路：

```text
initialLedgerData
-> useReducer(ledgerReducer)
-> LedgerData
-> positionService
-> calculatePositions(...)
-> Position[]
-> DashboardShell 展示
```

新增交易计划链路：

```text
表单输入
-> TradeDraft
-> tradeService
-> validateTradeDraft
-> Trade
-> dispatch({ type: "trade/add" })
-> ledgerReducer 写入 LedgerData.trades
```

这一阶段的关键判断：

- 当前不用 IndexedDB。
- 当前不用 localStorage。
- 当前不用 Redux / MobX。
- 当前先用 React `useReducer`。
- reducer 只负责状态变化。
- service 负责业务流程编排。
- calculator 继续只负责计算。
- validator 继续只负责校验。

## Week 4 Gate 1：长出 state 层

### `src/state/initialLedgerData.ts`

这个文件提供内存账本初始值。

当前实际实现是空账本：

```text
schemaVersion: 1
assets: []
trades: []
priceSnapshots: []
feeRules: []
```

为什么它要先长出来：

- `useReducer` 需要初始 state。
- `LedgerData` 需要一个明确、可复用的初始结构。
- 后续页面、测试、reset 都要有同一个起点。

为什么当前是空账本，而不是预置 BTC / ETH / ADA：

- Gate 1 只建立状态地基。
- 资产初始化策略可以后续再决定。
- 空数组能更清楚地证明 reducer 不夹带测试数据或假数据。

它不负责：

- 不导入 fixtures。
- 不保存到浏览器。
- 不生成价格。
- 不生成交易。

### `src/state/ledgerReducer.ts`

这个文件是账本状态变化入口。

当前支持三类 action：

```text
trade/add
trade/delete
ledger/reset
```

它的职责：

- 接收当前 `LedgerData`。
- 接收 `LedgerAction`。
- 返回新的 `LedgerData`。
- 保持不可变更新，不直接修改旧 state。

为什么它要在 `LedgerData` 后面长出来：

```text
LedgerData 定义账本长什么样
ledgerReducer 才能定义账本怎么变
```

它为什么不能早于 `tradeValidator` 和 `positionCalculator` 乱长：

- 如果 reducer 直接校验交易，就会抢 validator 的职责。
- 如果 reducer 直接计算持仓，就会抢 calculator 的职责。
- 如果 reducer 保存 `Position[]`，就会破坏“Position 是派生结果”的原则。

它不负责：

- 不调用 `validateTradeDraft(...)`。
- 不调用 `calculatePositions(...)`。
- 不保存 `Position[]`。
- 不读写 `localStorage`。
- 不读写 IndexedDB。
- 不接 UI 表单。

### `src/state/ledgerReducer.test.ts`

这个测试文件证明 state 层守住了边界。

它覆盖：

- 初始账本结构正确。
- 每次创建初始账本都有独立数组引用。
- `trade/add` 不修改旧 state。
- `trade/add` 不做交易业务校验。
- `trade/delete` 按 `trade.id` 删除。
- 删除不存在 id 时保持原 state。
- `ledger/reset` 返回新的空账本。

这组测试尤其重要，因为它证明：

```text
reducer 是状态管家，不是业务法官，也不是计算器。
```

## 已经剪掉的早期枝条

源码生长不是只增加。

早期曾经有这些文件：

```text
src/models/ledger.ts
src/models/tradeDraft.ts
src/repositories/ledgerRepository.ts
src/repositories/storageAdapter.ts
src/services/ledgerService.ts
src/utils/constants.ts
```

后来它们被删除，原因可以理解为：

```text
当时真实计算和校验还没长稳，repository / service / storageAdapter 提前实体化会让项目看起来完整，但实际核心账本能力还不可靠。
```

修剪后的路线更清楚：

```text
先算对
-> 再校验住
-> 再管理内存状态
-> 再接页面
-> 再持久化
```

这个顺序比“先把所有层都写一个空壳”更适合当前学习和原型阶段。

## 推荐阅读顺序

如果要按源码生长顺序理解当前项目，建议这样看：

```text
1. src/models/types.ts
2. src/utils/decimalMath.ts
3. src/test/fixtures.ts
4. src/calculators/positionCalculator.ts
5. src/validators/tradeValidator.ts
6. src/state/initialLedgerData.ts
7. src/state/ledgerReducer.ts
8. src/components/dashboard/DashboardShell.tsx
9. src/services/README.md
10. src/repositories/README.md
11. src/adapters/README.md
```

不要按文件夹字母顺序读。

原因：

- 先看 `types.ts`，知道账本有哪些名词。
- 再看 `decimalMath.ts`，知道数字怎么算。
- 再看 fixture，知道项目用哪批交易当标准输入。
- 再看 calculator，知道账本怎么从交易算出持仓。
- 再看 validator，知道交易怎么先被拦住。
- 再看 state，知道 React 内存账本怎么变化。
- 最后看页面和未来 service / storage 边界。
