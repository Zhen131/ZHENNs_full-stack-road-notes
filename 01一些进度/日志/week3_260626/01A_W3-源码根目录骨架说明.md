## 结论

拿到一个 Next.js / TypeScript 项目时，先不要从业务代码硬读。第一步先分清三类东西：

1. `README.md`、`package.json`、`src/`：优先看。
2. 各种 config 文件：知道用途，暂时不深挖。
3. `.git/`、`.next/`、`node_modules/`：基本别读，通常也别手改。

## 第一轮阅读优先级

| 条目 | 第一判断 | 一句话作用 | 现在怎么处理 |
| --- | --- | --- | --- |
| `README.md` | 先看 | 项目说明书，告诉人这个项目做什么、怎么运行、当前进度是什么 | 先读 |
| `package.json` | 先看 | 项目身份证、命令菜单、依赖清单 | 先读 |
| `src/` | 先看 | 真正的业务代码和测试代码所在位置 | 先读目录骨架，不急着读细节 |
| `.eslintrc.json` | 知道即可 | ESLint 代码检查规则 | 跑 lint 出问题时再看 |
| `.gitignore` | 知道即可 | 告诉 Git 哪些文件不要提交 | 新增文件提交异常时再看 |
| `next-env.d.ts` | 知道即可 | Next.js 给 TypeScript 用的环境类型声明 | 不手改 |
| `next.config.mjs` | 知道即可 | Next.js 框架配置 | 页面 / 构建异常时再看 |
| `package-lock.json` | 知道即可 | 锁定依赖的精确版本 | 不读，但通常要提交 |
| `postcss.config.mjs` | 知道即可 | CSS 处理链配置，通常服务 Tailwind | 样式构建异常时再看 |
| `tailwind.config.ts` | 知道即可 | Tailwind CSS 配置 | 做 UI 样式时再看 |
| `tsconfig.json` | 知道即可 | TypeScript 编译和类型检查配置 | 路径 / 类型检查异常时再看 |
| `vitest.config.ts` | 知道即可 | Vitest 测试配置 | 测试运行异常时再看 |
| `docs/` | 暂时放边上 | 项目辅助文档 | 只在相关主题需要时看 |
| `.git/` | 基本别读 | Git 内部数据库，记录提交、分支、索引 | 不手改 |
| `.next/` | 基本别读 | Next.js 运行 / 构建后的生成物和缓存 | 不读、不提交、不手改 |
| `node_modules/` | 基本别读 | npm 安装下来的第三方库实体 | 不读、不提交、不手改 |

## 基本别读的三类

### `.git/`

`.git/` 是 Git 自己的内部数据库。

它保存：

- 当前分支指向哪里。
- commit 历史。
- 暂存区索引。
- 远端仓库信息。

学习项目时不需要打开它。正常工程师通过 `git status`、`git log`、`git diff` 这些命令和 Git 交互，而不是直接改 `.git/` 里的文件。

边界：

- 可以知道它存在。
- 不要手动编辑。
- 出问题先用 Git 命令诊断，不直接碰内部文件。

### `.next/`

`.next/` 是 Next.js 自动生成的构建 / 开发缓存目录。

它通常在执行这些命令后出现：

```bash
npm run dev
npm run build
```

它保存的是框架编译后的中间结果，不是人写的源码。

边界：

- 不读。
- 不手改。
- 不提交。
- 构建缓存异常时可以删除后重新生成，但这属于工具维护，不是业务学习重点。

### `node_modules/`

`node_modules/` 是 `npm install` 下载出来的第三方依赖库。

例如本项目的 `package.json` 里写了：

```text
next
react
react-dom
decimal.js
typescript
vitest
tailwindcss
eslint
```

这些库的实际代码会被安装到 `node_modules/` 里面。

边界：

- 正常不读。
- 正常不手改。
- 正常不提交。
- 想知道项目用了什么库，看 `package.json`，不是翻 `node_modules/`。

## 第二轮再逐个理解的配置文件

这些文件不属于业务逻辑，但决定项目怎么运行、怎么检查、怎么构建。

| 文件 | 属于哪类工具 | 触发它的命令 / 场景 |
| --- | --- | --- |
| `.eslintrc.json` | ESLint | `npm run lint` |
| `next.config.mjs` | Next.js | `npm run dev`、`npm run build` |
| `postcss.config.mjs` | PostCSS / CSS 构建 | 页面样式构建 |
| `tailwind.config.ts` | Tailwind CSS | 写 UI 样式 |
| `tsconfig.json` | TypeScript | 类型检查、编辑器提示、构建 |
| `vitest.config.ts` | Vitest | `npm test`、`npm run test:watch` |
| `next-env.d.ts` | Next.js + TypeScript | Next 自动维护类型环境 |
| `package-lock.json` | npm | `npm install` |

## 当前学习顺序

第一步只需要建立这个判断：

```text
README.md      看项目说明
package.json   看命令和依赖
src/           看业务代码
配置文件        知道它们服务于工具链
.git/          Git 内部数据，不读
.next/         Next 生成物，不读
node_modules/  第三方依赖实体，不读
```

## 自检表

| 问题 | 通过线 |
| --- | --- |
| 看到 `node_modules/` 会不会想进去逐个读？ | 不会，知道它是依赖实体 |
| 看到 `.next/` 会不会把它当源码？ | 不会，知道它是构建产物 |
| 看到 `.git/` 会不会手动改文件？ | 不会，知道应该用 Git 命令 |
| 想知道项目能跑什么命令时看哪里？ | `package.json` |
| 想知道业务代码在哪里看哪里？ | `src/` |
| 想知道项目整体目标先看哪里？ | `README.md` |
