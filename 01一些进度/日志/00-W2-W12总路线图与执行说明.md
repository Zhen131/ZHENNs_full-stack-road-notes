## 2. 范围判断（必做 / 推迟 / 砍序）

**核心目标（一句话）**：做出一个浏览器内、本地优先、加密存储、算得对、能导入导出、有图表、有性能数据、能支撑论文 System Design / Implementation / Evaluation 三章的个人交易账本原型。

**必做（不可砍，P0）**：数据模型、核心计算（持仓/均价/盈亏/超卖）、React 内存账本状态地基、IndexedDB 持久化、导入导出、加密、benchmark、论文素材。

**推迟到核心版本之后（不进核心）**：NLP（W12，可砍）、Agent（W13，可砍）、实时价格 API、K 线、技术指标、策略回测、插件、多端同步、Tauri/Capacitor 打包——暑假期间只预留接口、不写实现。

## 3. 节奏与时间模板

**每周 6 天**：Day1 计划设计 / Day2–4 开发主战场（每天只攻一件事，写完即测）/ Day5 测试修复（**缓冲日，吸收延期**）/ Day6 文档复盘（攒论文素材）/ Day7 休息。

**每天 4 小时**：0.5h 启动（回看昨天、明确今天只做一件事）+ 2h 深度开发/学习 + 1h 测试验证 + 0.5h 记录。

## 4. 风险、优先级与压缩预案

**优先级**：P0 命脉（绝不砍）= 模型/计算/React 状态地基/IndexedDB/导入导出/加密/benchmark/论文素材；P1（能压缩）= 第二三个图表、100k 量级、UI 美化；P2（可整块砍）= NLP、Agent、其余 backlog。

**落后三级处理**：
1. **Day 5 缓冲**：当周没做完的，Day 5 先补再测。
2. **跳票周**：某周结束 ≥4 项未完成，下一周不做新主题、专补 P0，后续顺延一周。每月最多 1 次；8/7 后不再启动。
3. **砍功能**（按序）：先砍 NLP(W12)+Agent(W13) → 再砍第三个图表 → 再砍 benchmark 的 100k → 再缩 NLP/Agent 到最小 → **绝不砍**核心计算/React 状态地基/IndexedDB/加密/导入导出/benchmark 基本盘/论文素材。

**如果还是太满 → 压缩核心**：W6 导入只做整库覆盖；W8 只做一张饼图；W10 三章只写提纲+图；W12–13 直接不做。最差情况下仍交付"算得对+内存态可用+存得住+锁得住+导得出+一张图+benchmark+论文素材"，**毕业论文照样成立**。
