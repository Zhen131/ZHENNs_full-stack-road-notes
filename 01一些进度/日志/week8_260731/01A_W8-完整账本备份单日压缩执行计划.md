## 4. 固定备份契约

### 4.1 类型

在 `src/backup/backupEnvelope.ts` 定义：

```ts
type BackupEnvelopeV1 = {
  backupFormatVersion: 1;
  appVersion: string;
  exportedAt: string;
  ledgerSchemaVersion: 1;
  ledgerData: LedgerData;
};
```

固定规则：

- 顶层字段顺序严格为 `backupFormatVersion / appVersion / exportedAt / ledgerSchemaVersion / ledgerData`。
- `backupFormatVersion` 是备份格式兼容 Gate；未知版本直接拒绝。
- `ledgerSchemaVersion` 必须为 `1`，且必须等于 `ledgerData.schemaVersion`。
- `appVersion` 从 `package.json` 的非空 `version` 注入，只记录来源；应用补丁版本不同不得拒绝导入。
- `exportedAt` 由依赖注入，必须是带时区的严格 ISO datetime；生产使用 `new Date().toISOString()`。
- `ledgerData` 必须来自 `validateLedgerData(...)` 成功结果的 `value`。
- 不加入 persistence status、IndexedDB key、加密参数、`Position[]`、UI 状态或设备信息。

### 4.2 纯函数

固定导出 API：

```text
createBackupEnvelope(ledgerData, metadata)
-> 再次运行 validateLedgerData
-> 使用重建后的 immutable snapshot
-> 返回 BackupEnvelopeV1 或结构化错误

serializeBackupEnvelope(envelope)
-> JSON.stringify(envelope, null, 2) + "\n"
-> 不排序 assets / trades / priceSnapshots / feeRules
-> 相同 envelope 必须产生完全相同字节

parseBackupJson(text)
-> parse 前执行 UTF-8 文件字节检查
-> 坏 JSON 返回结构化错误，不 throw 普通输入错误
-> JSON.parse 成功后交给 validateBackupEnvelope

validateBackupEnvelope(value)
-> 重建固定顶层字段
-> 校验 backup/app/time/schema 元数据
-> 复用 validateLedgerData
-> 复用 evaluateLedgerResourcePolicy
-> 返回脱离原始输入引用的完整候选
```

顶层未知字段不进入成功结果；数组业务顺序保持原样。不得使用递归 key 排序改变实体字段、数组顺序或业务语义。

### 4.3 稳定错误

backup 模块只拥有 envelope 和 JSON 语法错误：

```text
BACKUP_BAD_JSON
BACKUP_INVALID_ENVELOPE
BACKUP_UNSUPPORTED_FORMAT_VERSION
BACKUP_INVALID_APP_VERSION
BACKUP_INVALID_EXPORTED_AT
BACKUP_SCHEMA_VERSION_MISMATCH
```

错误对象至少包含：

```ts
type BackupError = {
  code: string;
  path: string;
  message: string;
};
```

归属规则：

- 账本字段错误保留既有 `LEDGER_DATA_*` code 和 path；backup 模块只做聚合，不复制一套字段校验。
- 文件、集合和字符串超限保留既有 `LEDGER_RESOURCE_*` code、path、limit、actual。
- Repository 写失败保留 `LEDGER_REPOSITORY_WRITE_FAILED`。
- UI 只把 code 映射为中文，不创建业务错误码。

## 5. 文件与资源边界

导入文件执行两次字节防线：

1. 选择文件后先检查 `File.size`，超过 8 MiB 时禁止调用 `file.text()`。
2. 得到文本后、`JSON.parse` 前调用 `evaluateLedgerJsonResourcePolicy(text)`，按 UTF-8 实际字节复核。

若现有 ResourcePolicy 缺少接收 `File.size` 的纯函数，只在 `resourcePolicy.ts` 增加按实际 byte length 评估的入口；复用 `LEDGER_RESOURCE_FILE_TOO_LARGE`，不得在组件中手写阈值或错误对象。

导入成功候选必须同时通过：

```text
file bytes <= 8 MiB
assets <= 500
trades <= 25,000
priceSnapshots <= 5,000
feeRules <= 500
ID / symbol / currency / name / platform / note / rawText 长度上限
完整 LedgerData Validator
```

导出分两类，二者都受 8 MiB 文件硬上限约束：

- 常规备份：完整校验和资源策略均通过；生成的 JSON 不超过文件上限，保证当前版本可重新导入。
- 救援备份：已只读 hydrate 的旧账本通过 `validateLedgerData`，且序列化后的完整 envelope 不超过 8 MiB；允许导出，但 UI 必须显示“救援备份可能因集合或字符串超限而无法由当前版本重新导入”。

序列化结果超过 8 MiB 时禁止创建 Blob，显示“当前 v1 无法安全导出该超大账本”。本周不新增更高阈值，不实现流式导出。救援导出只保证受控范围内的数据可携带，不得绕过导入 ResourcePolicy。

## 8. 批次 C：原子整账导入

不要给 Repository 新增 `replace`。当前 `save` 已在单个 IndexedDB transaction 中 whole-blob `put`，即本次需要的存储原子替换。新增同义方法只会制造第二套持久化语义。

修改 `usePersistentLedger.ts`：

```ts
type PersistenceOperation = "idle" | "clearing" | "importing";

type ImportLedgerMode = "normal" | "recovery";

const LEDGER_IMPORT_ERROR_CODES = {
  NOT_ALLOWED: "LEDGER_IMPORT_NOT_ALLOWED",
  INVALID_CANDIDATE: "LEDGER_IMPORT_INVALID_CANDIDATE",
  RESOURCE_POLICY_REJECTED: "LEDGER_IMPORT_RESOURCE_POLICY_REJECTED",
} as const;

type ImportLedgerResult =
  | { ok: true }
  | {
      ok: false;
      code:
        | (typeof LEDGER_IMPORT_ERROR_CODES)[keyof typeof LEDGER_IMPORT_ERROR_CODES]
        | typeof LEDGER_REPOSITORY_ERROR_CODES.WRITE_FAILED;
    };

replaceLedgerFromBackup(candidate: LedgerData): Promise<ImportLedgerResult>
```

固定顺序：

```text
同步判断 normal ready 或 hydration error recovery
-> normal 模式检查 !readOnly
-> 检查 active Repository / idle
-> 在 Hook 边界重新 validateLedgerData(candidate)
-> evaluateLedgerResourcePolicy(sanitized candidate)
-> 同步把 operationRef 置为 importing
-> 冻结 operation Repository、generation、token 和 candidate snapshot
-> 延迟 requested Repository 切换
-> 接到现有 writeQueueRef 尾部
-> repository.save(candidate snapshot) 一次
-> save 成功后才 reducerDispatch(ledger/replace)
-> 同步 ledgerDataRef、lastPersistedSnapshot、version、error、retry、hydration、read-only refs
-> 标记新账本 clean，禁止 effect 再自动保存一次
-> operation 回到 idle
-> 再按正常规则处理期间积压的 requested Repository
```

允许进入的状态必须写成显式分支：

```text
normal:
hydrationStatus === "ready"
&& hydratedRepositoryRef.current === activeRepository
&& readOnlyRef.current === false

recovery:
hydrationStatus === "error"
&& hydrationErrorRepositoryRef.current === activeRepository
```

`hydrationStatus === "loading"`、ready + read-only、Repository 不匹配或 operation 非 idle
统一返回 `LEDGER_IMPORT_NOT_ALLOWED`，不得调用 `repository.save(...)`。

失败规则：

- 不先 `clear()`，不逐条 dispatch，不逐条 save。
- 校验或资源失败：零 Repository 写入，旧 UI 不变。
- normal 模式 Repository save 失败：旧页面 state 不变；import write 开始前的最后成功 record 不变。
- recovery 模式 Repository save 失败：保持 `hydrationStatus === "error"`；旧损坏 record 不变，不伪装为已恢复。
- recovery 模式 save 成功：直接整账 replace，清除 hydration error，建立 clean snapshot，再进入 `ready`；不得先 clear。
- ready + read-only 状态禁止 import；不能把 import 当成绕过 S-07 的 mutation/clear 入口。
- `persistenceStatus === "saving" | "error"` 时允许用户进入导入确认，但必须提示当前未落盘页面数据将被覆盖，并提供先导出救援备份的入口。
- 前置普通 save 成功或失败后，import 都按队列顺序执行；不得与旧 save 交叉覆盖。
- import 期间拒绝普通 action、retry、clear 和第二次 import。
- requested Repository 在 importing 期间只记录为待处理切换；不得修改 `activeRepositoryRef`、开始新 hydration 或让旧 import completion 作用于新 Repository。
- operation 结束后再执行正常 Repository 切换；import 始终只写确认时固定的 operation Repository。
- 重复点击不得产生第二次写；可共享同一 import Promise 或确定性返回 busy。
- 组件卸载、Repository 更换或 generation 失效后，旧 Promise 不得改写新页面 state。
- 成功后 `mutationVersion === persistedVersion`、`isDirty === false`、`persistenceStatus === "saved"`。

rollback 基准按队列状态分别定义：

```text
无前置 save:
import 失败后 record === 用户确认导入前的 record

前置 queued save 成功:
import 失败后 record === queued save 成功后的 record

前置 queued save 失败:
import 失败后 record === queued save 之前最后一次成功的 record

三种情况:
页面 state 均保持 import 开始前的页面账本
```

必须测试：

- deferred save 未完成时页面仍显示旧账本，operation 为 importing。
- save 成功后页面一次性变为候选，Repository 只写一次，不触发第二次自动保存。
- save 失败后旧 state、版本、dirty 语义和旧存储保持正确。
- 前置 queued save、前置 save failure、重复 import、import/clear 互斥、import/action 互斥。
- normal import、hydration error recovery import 成功和失败。
- ready + read-only import 返回 NOT_ALLOWED、零写入、旧 record 不变。
- 无效候选和超限候选零写入。
- deferred import 期间请求切 Repository：旧 operation Repository 被固定，新 Repository 不提前 hydration；operation 结束后才切换。
- Repository 切换、卸载和 stale completion 不覆盖新实例。
- 三种 rollback 基准分别断言页面 state 和实际 record。
- 使用保存了旧 envelope 的故障 `StorageAdapter` 驱动真实 `DefaultLedgerRepository` write failure；不得只用会直接 throw 的 mock Repository 代替旧 record 证明。
- fake IndexedDB 真实组装链写入后重挂载恢复完整候选。

- 成功后清空文件 input、候选和旧错误；页面交易、价格、资产和持仓立即来自导入账本。
- 重置 input value，允许再次选择同一个文件。
- importing 时禁用交易、价格、删除、clear、export、重复 import 和 Repository 切换动作。
- 删除区原“Week 8 备份尚未实现”文案改为当前事实，不再声称不可恢复。

组件测试：

- 选文件后只解析，不写 Repository。
- 错文件显示结构化错误；确认按钮不存在。
- 取消不写入；重选清除旧候选和错误。
- A/B 两次文件读取乱序完成时只有 B 能成为候选；取消或卸载后的旧读取不得更新状态。
- 合法文件确认后调用一次原子导入。
- 写失败显示“原账本未更改”；成功显示“备份已导入并保存到本地”。
- saving/error/read-only 的导出警告和可用性正确。
- hydration error recovery 成功进入 ready；失败保持 error 和恢复入口。
- ready + read-only import 始终禁用，Repository 零写入。
- loading/clearing/importing 时所有入口门禁正确。

## 10. 批次 E：round-trip / rollback 自动化 Gate

建立一个完整固定账本，至少包含：

```text
BTC / ETH / ADA 内置资产
1 个自定义资产
5 条 golden trades
至少 1 条带 note / rawText / feeRuleId 的交易
至少 2 条 priceSnapshots
至少 1 条 feeRule
全部可选字段中的代表值
```

不得把 `Position[]` 写入 fixture。fixture 只进入测试或备份测试数据，不得被生产代码导入。

必须关闭以下场景：

| 场景 | 通过线 |
| --- | --- |
| canonical create/serialize/parse | 相同输入和 metadata 字节一致 |
| export -> parse | 完整 `LedgerData` 逐字段一致 |
| export -> clear -> import -> remount | 刷新后逐字段一致 |
| canonical re-export | 规范化 `ledgerData` 一致；忽略新 `exportedAt` |
| bad JSON | 零写入 |
| unknown backup version | 零写入 |
| schema mismatch | 零写入 |
| missing collection/entity field | 零写入 |
| duplicate ID/symbol | 零写入 |
| unknown asset/dangling feeRule | 零写入 |
| bad Decimal/date/timeline | 零写入 |
| file/collection/string over limit | 零写入 |
| normal import write failure | 旧页面不变；record 等于 import write 前最后成功版本 |
| hydration error recovery success | 不先 clear；写成功后整账显示并进入 ready |
| hydration error recovery failure | 保持 error；损坏 record 不变 |
| ready + read-only import | NOT_ALLOWED；零写入；只允许救援导出 |
| pending save + import | 严格按队列顺序，最终只有导入账本 |
| pending save + failed import | 按三种 rollback 基准核对最终 record |
| repeated import | 不重复写 |
| import + clear/action/retry | 互斥，无交叉覆盖 |
| import + Repository switch | operation Repository 固定；切换延迟到 operation 结束 |
| file A/B 读取乱序 | 只有最后选择的文件可成为候选 |
| rescue export bytes | `<= 8 MiB` 可生成；超一字节不创建 Blob |

测试层级：

```text
纯函数单元测试
-> Hook 状态与并发测试
-> BackupControls 交互测试
-> Dashboard 集成测试
-> fake IndexedDB 真实组装链重挂载测试
```

先运行定向测试；每个红灯执行：

```text
保存失败输入与断言
-> 定位职责层
-> 最小修复
-> 定向复验
-> 全量复验
```

不得为了让测试通过放宽 Validator、ResourcePolicy 或原子替换顺序。

## 11. 批次 F：production 浏览器验收

使用 production build、独立端口、隔离 origin、单标签页。不得删除默认 `localhost:3000` 或用户浏览器现有数据。

成功主链：

1. 启动空库，等待 hydration ready。
2. 通过真实 UI 新增固定 BTC / ETH 交易和 BTC 价格，等待“已保存到本地”。
3. 点击导出；记录文件名、字节数、envelope 版本、appVersion、exportedAt、四个集合数量和无 `Position[]`。
4. 执行固定文本 clear；直接确认 `ledger:v1` 不存在。
5. 选择刚导出的文件；确认前核对摘要并确认页面仍为空。
6. 确认覆盖；等待“备份已导入并保存到本地”。
7. 直接读取 `ledger:v1`，确认 record 已一次性恢复。
8. 刷新；核对交易、价格、持仓和完整账本字段。
9. 再次导出；比较两份 envelope 的规范化 `ledgerData` 完全一致。
10. 控制台 warning/error 必须为 0。

完整字段主链：使用通过自动化校验的完整备份 fixture，从 UI 导入含自定义资产、5 条交易、价格和 feeRule 的账本；刷新后再导出并由 parser 复核逐字段一致。fixture 不得通过 DevTools 直接写 IndexedDB。

hydration error 恢复主链仅在同一隔离测试 origin 执行：

1. 先导出一份已通过 parser 的合法备份。
2. 在 DevTools 中把该测试 origin 的 `ledger:v1.encryptedPayload` 改为坏 JSON，刷新并确认 hydration error。
3. 不执行 clear，直接从“从备份恢复”入口导入合法备份。
4. 确认成功后进入 ready，直接读取 record 并刷新复验完整账本。
5. 故障注入下的 recovery write failure 仍由确定性自动化证明，不在生产代码增加失败开关。

人工拒绝场景至少执行：

- 坏 JSON。
- `backupFormatVersion = 2`。
- `ledgerSchemaVersion` 与 payload 不一致。
- 一项引用错误。
- 超过 8 MiB 的文件。

人工验收负责真实浏览器成功路径和错误展示；Repository write 故障、竞态、卸载和回滚由确定性自动化证明。不得在生产代码加入故障按钮或 debug 后门。

## 15. 禁止范围与停止条件

- 为验收增加生产 debug 开关、故障按钮或测试依赖入口。

出现以下情况立即停止扩大实现，保留失败测试并报告：

- 必须修改 `LedgerData.schemaVersion` 才能继续。
- 必须改变 IndexedDB schema、record key 或 Adapter transaction 才能保证原子性。
- 必须放宽现有完整 Validator 或 ResourcePolicy 才能接受 fixture。
- 必须允许 ready + read-only import 才能继续。
- 无法在 8 MiB 已批准上限内完成救援导出，且需要新增更高阈值或流式导出。
- 无法证明 write failure 保留旧 record。
- import 无法进入现有 write queue，或成功前无法阻止 UI state 替换。
- 需要新增生产依赖、执行强制依赖升级或进入 Week 9 范围。

最终汇报只包含：Gate Go/No-Go、源码改动、round-trip/rollback 结果、test/lint/build、production 实际值、源码 commit、双仓库状态和 Week 9 是否获准进入。
