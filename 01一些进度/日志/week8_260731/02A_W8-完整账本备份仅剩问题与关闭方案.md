## P1-01：production 证据链不完整

### 问题现象

`01E` 已记录页面级结果：

- BTC 交易和 BTC 价格保存。
- 导出成功提示。
- clear 后页面为空。
- 完整 BTC / ETH fixture 导入与刷新。
- 五类无效文件被 UI 拒绝。
- console warning/error 为 0。

以下固定 Gate 没有证据：

- ETH 交易由生产 UI 新增并等待 saved。
- 实际下载文件的文件名、字节数、版本、时间、四组集合数量。
- 实际下载文件确认不包含 `Position[]`。
- clear 后直接确认 `ledger:v1` record 不存在。
- 导入使用刚刚实际导出的文件，不是重新生成的测试 fixture。
- import 后直接确认 `ledger:v1` 一次性恢复。
- 再导出文件经 parser 复核后，规范化 `ledgerData` 与首次导出一致。
- production hydration error recovery。
- 五类拒绝场景同时证明页面和 record 均未变化。

### 出现原因

1. 浏览器验收只记录页面可见状态，没有保存下载文件和 record 级证据。
2. 下载事件未提供可读取的文件路径后，执行过程改用人工生成 fixture，实际导出文件没有进入回导链。
3. 为避免直接读取或改写浏览器存储，production recovery 和 record 核对改由 fake IndexedDB 测试替代。
4. 原固定 Gate 没有被正式变更，执行过程却把“production + 自动化混合证据”等同于完整 production Gate。

### 风险

- 页面显示“导出成功”不能证明下载文件内容、文件名和字节数正确。
- 页面 clear 不能证明 record 已删除。
- 页面 import 成功不能证明 record 只写入一次且内容完整。
- 使用生成 fixture 回导不能证明真实导出文件可回导。
- UI 拒绝不能单独证明 IndexedDB record 零变化。
- 自动化 recovery 不能替代 production origin 的 recovery 接线证据。

### 答案

保留现有源码，不新增测试后门。使用 production build、隔离 origin、真实下载文件和允许检查 IndexedDB 的人工 Chrome DevTools 验收关闭该问题。

## P1-02：8 MiB 精确边界测试不真实

### 问题现象

当前测试分成三部分：

1. 把一个很小的合法 `File` 的 `size` 强制改成 8 MiB。
2. 使用 8 MiB 的 `"x"` 验证 ResourcePolicy 接受字节数。
3. 使用超大 `rawText` 验证超限导出不调用 `URL.createObjectURL`。

这些测试没有证明：

- 一个内容真实为 8 MiB 的合法 `BackupEnvelopeV1` 可以读取、parse 和进入确认。
- 同一个真实序列化备份增加 1 byte 后被拒绝。
- 超限导出没有构造 `Blob`。

### 出现原因

1. `File.size` 与 `File.text()` 被分别伪造，二者不代表同一个真实文件。
2. ResourcePolicy 测试只验证数值边界，不验证合法 JSON 与 parser 主链。
3. 超限导出测试只监控 `URL.createObjectURL`，没有直接监控 `Blob` 构造。
4. 三个局部断言被合并解释为一个端到端精确边界 Gate。

### 风险

- UTF-8 字节计算的 off-by-one 可能未被发现。
- `File.size` 通过但文本复核失败的边界行为没有真实样例。
- parser 对带大量尾随空白的合法 JSON 是否通过没有被锁定。
- 未来重构可能先创建大 Blob、后检查上限，而现有测试仍通过。

### 答案

构造真实合法 envelope，在序列化 JSON 后追加 JSON 允许的 ASCII 尾随空白，使实际 UTF-8 字节数精确达到 8 MiB。所有断言必须使用同一字符串生成真实 `File`。

### 测试实现

#### A. 公共测试辅助函数

```ts
function padSerializedBackupToBytes(
  serialized: string,
  targetBytes: number,
): string {
  const encoder = new TextEncoder();
  const currentBytes = encoder.encode(serialized).byteLength;

  if (currentBytes > targetBytes) {
    throw new Error("Serialized fixture already exceeds target");
  }

  return `${serialized}${" ".repeat(targetBytes - currentBytes)}`;
}
```

尾随空格属于合法 JSON whitespace。禁止修改 envelope 字段、放宽字符串上限或加入未知业务字段。

#### B. 恰好 8 MiB 导入

1. 创建合法 `BackupEnvelopeV1`。
2. 使用固定 metadata 序列化。
3. 填充到 `8 * 1024 * 1024` bytes。
4. 断言 `TextEncoder().encode(serialized).byteLength` 精确等于上限。
5. 使用该字符串创建真实 `File`。
6. 不覆盖 `file.size`。
7. `File.text()` 返回同一字符串。
8. UI 上传后必须出现候选摘要和“确认恢复备份”。
9. `parseBackupJson(serialized)` 必须成功。

#### C. 超 1 byte 导入

1. 在 B 的同一字符串末尾追加一个 ASCII 空格。
2. 断言实际字节数为 `8 MiB + 1`。
3. 创建真实 `File`，不覆盖 `file.size`。
4. UI 必须在 `File.text()` 前拒绝。
5. parser 必须返回 `LEDGER_RESOURCE_FILE_TOO_LARGE`。
6. 不出现确认恢复入口。

#### D. 精确导出边界与 Blob

1. 固定系统时间，保证 `exportedAt` 和 envelope 开销稳定。
2. 构造 read-only 测试账本，通过可控 `rawText` 长度把实际序列化结果调到：
   - 恰好 8 MiB。
   - 恰好 8 MiB + 1 byte。
3. 恰好 8 MiB：
   - 允许导出。
   - `Blob` 构造一次。
   - `URL.createObjectURL` 调用一次。
4. 8 MiB + 1：
   - 显示“当前 v1 无法安全导出该超大账本；未创建备份文件”。
   - `Blob` 构造 0 次。
   - `URL.createObjectURL` 调用 0 次。

该测试只验证文件字节边界。read-only 风险文案和字符串 ResourcePolicy 继续由现有测试负责。

### 必须修改的测试

- `src/components/backup/BackupControls.test.tsx`
- `src/validators/resourcePolicy.test.ts`

生产实现原则上不改。只有真实精确边界测试失败时，才修改：

- `src/components/backup/BackupControls.tsx`

### 通过线

- 合法备份实际内容恰好 8 MiB，parse 与 UI 候选确认通过。
- 同一合法备份增加 1 byte，读取前与 parse 前双边界均拒绝。
- 精确导出边界证明 8 MiB 创建 Blob，`8 MiB + 1` 不创建 Blob。
- 不伪造与内容不一致的 `File.size`。
- 不使用非法 JSON 字符串代替合法备份。
