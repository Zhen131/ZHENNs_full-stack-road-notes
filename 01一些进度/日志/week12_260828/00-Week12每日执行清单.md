# Week 12 每日执行清单

时间：2026-08-28 至 2026-09-03
主题：论文证据整理 + P0 发布门

> 本周不再做原计划中的 NLP。NLP 已降级为暑期路线外延伸，不占用论文 P0 时间。
> 本周只做两件事形成的一条主线：把已经实现的系统整理成可引用证据，并把候选版本推到可运行 Week 13 final benchmark 的状态。
> Week 11 只完成 benchmark 协议、harness 与 pilot；本周不得提前把 pilot 写成 final 结果。

---

## 本周起点

进入本周前，必须已经满足：

- Week 10 两张保留图表通过验收。
- Week 11 benchmark 协议已冻结。
- 100 / 1k / 10k 三个量级的 pilot 已跑通。
- benchmark harness 能保存原始数据与环境信息。
- 100,000 条仍只是 stretch，不是论文硬通过线。

如果上述任一条件不满足，本周先用对应日期补齐前序 Gate，不新增功能。

---

## Day 1：8月28日，整理 System Design 证据

今天只做一件事：按源码真实结构整理系统设计证据。

要做：

- 固定当前实际存在的层次：模型、纯计算、校验、React 状态、服务、repository / adapter、加密、备份恢复、图表。
- 画出或更新实际数据流：用户操作 → 校验/服务 → reducer → repository → IndexedDB。
- 标注浏览器启动时的恢复路径：读取 → 解密 → 校验 → hydrate → 页面 ready。
- 对每一层附上真实文件路径或测试路径。
- 把“源码已经实现”和“原方案设想但未实现”分栏记录。

产出：

- System Design 章节证据包。
- 一份实际架构与数据流清单。

完成标准：

- 每个设计结论都能回到源码或测试。
- 不把 placeholder、计划接口或已降级功能写成实现成果。
- 不再沿用与当前源码不一致的旧架构描述。

---

## Day 2：8月29日，整理 Implementation 证据

今天只做一件事：整理核心功能的实现证据。

要做：

- 为核心计算与交易校验整理实现文件、关键测试和 golden 样例。
- 为 React 状态地基整理 reducer、初始数据、页面接线和交互证据。
- 为 IndexedDB、导入导出、加密和两张图表整理对应实现与验收证据。
- 记录关键设计取舍，不重复抄源码。
- 标出任何仍未关闭的 P0 缺口。

产出：

- Implementation 章节证据包。
- P0 实现—测试—验收映射表。

完成标准：

- 论文中的每项“已实现”都有源码、测试或手动验收证据。
- 不把 Week 1 的设计概念、service README 占位说明当作已实现。
- NLP、Agent、第三张图和 100k 不进入核心实现成果。

---

## Day 3：8月30日，整理 Evaluation 方法证据

今天只做一件事：冻结论文评价方法及其边界。

要做：

- 把 Week 11 benchmark 协议整理成论文可复现的方法说明。
- 明确三个硬量级、四类指标、10 次 final 重复、median + IQR。
- 说明数据预置与正式测量分离，避免把造数时间混入指标。
- 整理威胁模型、加密边界与 hard-offline 验收证据。
- 对照 local-first 原则记录支持项、限制项和范围偏差。

产出：

- Evaluation Method 初稿。
- 安全与 local-first 证据清单。

完成标准：

- 任何人按方法说明都知道测什么、如何测、如何汇总。
- pilot 与 final 被明确区分。
- 不虚构用户研究、服务器同步或尚未执行的实验结果。

---

## Day 4：8月31日，执行 P0 全量回归

今天只做一件事：对候选版本执行一次完整 P0 回归。

自动验证：

- 运行全部测试。
- 运行 lint。
- 运行 production build。

手动验证：

- 新增、删除、价格更新与汇总联动正确。
- 刷新后 IndexedDB 数据恢复正确。
- 全量导出、导入与失败回滚正确。
- 首次加密迁移、锁定、正确/错误口令行为正确。
- hard-offline 下核心流程可用。
- 两张保留图表与账本数据一致。
- benchmark pilot 仍能由统一入口运行。

产出：

- 一份带日期、命令和结果的 P0 回归记录。
- 按 P0 / P1 / P2 分级的问题清单。

完成标准：

- 所有验证都有明确的通过或失败记录。
- 失败项不得用“基本可用”代替结论。
- 当天只盘点，不顺手扩展新功能。

---

## Day 5：9月1日，关闭 P0 发布阻塞项

今天只做一件事：只修复 Day 4 暴露的 P0 阻塞项，直到发布门转绿。

执行规则：

- 只处理会破坏数据正确性、持久化、备份恢复、加密、核心图表、benchmark 可运行性或构建的 P0 问题。
- 每修一个问题，都补相称的自动测试或回归证据。
- P1 / P2 只记录，不在本日扩张范围。
- 如果一天无法关闭全部 P0，Week 13 final benchmark 延后，不得带红灯进入 final。

产出：

- P0 修复与复验记录。
- 更新后的发布门状态。

完成标准：

- P0 阻塞项全部关闭。
- 测试、lint、build 重新通过。
- 关键手动路径针对修复点完成复验。

## Day 6：9月2日，完成候选版本可复现审计

今天只做一件事：证明候选版本可从干净环境按文档复现。

要做：

- 按 README 从干净依赖安装开始完成运行、测试和 production build。
- 按 benchmark 文档完成一次非 final 的最小 dry-run。
- 检查实验命令、数据目录、环境记录和结果格式是否完整。
- 记录准备交给 Week 13 的候选版本范围与已知限制。
- 明确 final benchmark 只能针对锁定后的候选提交运行。

产出：

- 可复现审计记录。
- Week 13 候选版本交接清单。

完成标准：

- 新环境按文档无需口头补充即可启动、验证和 dry-run。
- P0 发布门保持全绿。
- 本日不跑 final benchmark，也不提前宣称最终性能结果。
