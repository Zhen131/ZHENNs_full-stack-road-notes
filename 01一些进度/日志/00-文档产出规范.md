# 文档产出规范

状态：默认写作规则

用途：约束后续 Week 日志、设计说明、技术选型、验收标准等文档的产出风格。

---

## 总原则

文档要像工程卡片，不像教程文章。

核心要求：

- 先给结论，再给依据。
- 动机必须有，但用一句话打中核心差异。
- 信息密度高，少铺垫，少重复。
- 多写边界、规则、通过线、自检表。
- 用表格和清单承载结构。
- 读者默认有编程基础，不解释常识。

推荐长度：

```text
普通说明：1000-1500 字左右
决策记录：800-1200 字左右
验收标准：1200-1800 字左右
```

超过长度时，优先删背景、比喻、重复解释，不删结论和边界。

---

## 动机写法

动机不能消失，但必须短。

推荐格式：

```text
选择 A，因为 B 无法满足 X，而 A 能守住 Y。
```

例：

```text
选择 decimal.js，因为 big.js 更轻但能力更窄，而本项目需要长期稳定支撑均价、盈亏和误差校验。
```

不要写成：

```text
我们为什么要做这个？在软件开发中，很多时候……
```

动机的作用是解释取舍，不是写背景作文。

---

## 通用结构

以后文档优先使用这个骨架：

```text
# 标题

日期 / 状态 / 目标

## 结论
直接说本文件决定了什么。

## 动机
一句话说明为什么这么做。

## 核心规则
列出真正会约束后续开发的规则。

## 边界
说明做什么、不做什么。

## 自检表
用表格让以后快速判断是否符合规范。
```

不是每份文档都必须完整套用，但结论、规则、边界、自检至少要有其中三项。

---

## 不同文档模板

### 架构说明

重点：职责和边界。

推荐结构：

```text
结论
目录 / 模块骨架
各层职责
禁止事项
自检表
```

少写“为什么要做架构”，多写“什么代码放哪里”。

### 技术选型

重点：选择理由和使用边界。

推荐结构：

```text
选型结论
一句话动机
候选对比
使用边界
落地顺序
自检表
```

必须回答：

- 选了什么。
- 为什么不是另一个。
- 谁可以直接使用它。
- 谁不能直接使用它。

### 验收标准

重点：通过线。

推荐结构：

```text
验收结论
固定样例
手算 / 预期基准
必须通过项
测试清单
不在本次验收范围
不可推迟项
```

验收标准必须可测试，不能只写“功能正常”“体验良好”。

### 每日计划

重点：今天只做什么。

推荐结构：

```text
今日目标
产出物
完成标准
不做事项
风险 / 可砍项
```

每日计划要窄，不要把一周任务都塞进去。

### 开发日志

重点：当天完成了什么、为什么这样处理。

推荐结构：

```text
今日完成
关键决策
遇到的问题
明天入口
```

开发日志可以有一点叙述，但不能变成流水账。

---

## 禁止事项

以后生成文档时避免：

- 大段解释基础概念。
- 重复说明同一个动机。
- 用很多“为了更好地”“非常重要”“有助于”但不落到规则。
- 写太多未来幻想，不写当前边界。
- 把验收标准写成愿望。
- 把工具层、业务层、页面层职责混在一起。
- 为了显得完整而写无用章节。

---

## 推荐表达

优先写：

```text
页面不直接计算，不直接保存，不直接碰浏览器 API。
```

少写：

```text
页面层在整个系统中承担着非常重要的作用，它不仅关系到用户体验，也会影响未来项目的可维护性……
```

优先写：

```text
如果一个文件同时承担两行以上职责，优先拆开。
```

少写：

```text
我们应该尽量遵循单一职责原则，因为单一职责原则是软件工程中非常重要的原则……
```

---

## 自检表

| 问题 | 通过标准 |
| --- | --- |
| 开头是否有结论 | 第一屏能看到决定了什么 |
| 动机是否过长 | 一句话能说清核心取舍 |
| 是否有边界 | 明确写了做什么 / 不做什么 |
| 是否能指导后续开发 | 有规则、职责或通过线 |
| 是否有废话 | 删除一段后不影响执行的内容应删 |
| 是否像教程 | 如果像讲课而不是工程记录，需要压缩 |

---

## 默认要求

后续新建日志文档时，默认按本规范写。

如果某份文档需要写长，必须满足：

```text
长是因为规则更多，不是因为解释更多。
```
