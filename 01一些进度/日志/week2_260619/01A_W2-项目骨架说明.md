# Day 1 项目骨架说明

日期：2026-06-19

状态：Day1 骨架草案

目标：定下 Week2 以后代码的基本分层。今天不写真实计算，不改页面，只规定“什么代码应该长在哪里”。

---

## 总原则

项目骨架的价值：先把骨骼比例定准，再往上填计算、校验、保存和页面。工程上就是让 UI、领域模型、计算、校验、流程、存储彼此分离，后续扩展时加模块，不拆页面。

核心依赖方向：

```text
app / components
  -> services

services
  -> validators
  -> calculators
  -> repositories

validators
  -> models
  -> utils/decimalMath

calculators
  -> models
  -> utils/decimalMath

repositories
  -> models
  -> adapters

adapters
  -> IndexedDB / Web Crypto / JSON / File APIs
```

页面不直接计算，不直接保存，不直接碰浏览器 API。

---

## 目录骨架

```text
src/
  app/             页面入口和路由
  components/      UI 展示和输入组件
  models/          核心数据结构
  validators/      输入和交易合法性校验
  calculators/     持仓、均价、盈亏计算
  services/        业务流程编排
  repositories/    账本读写入口
  adapters/        外部系统适配
  utils/           通用工具
```

---

## 各层职责

### app / components

页面层。只负责展示、输入、触发动作。

允许：渲染表格、表单、按钮、错误提示，调用 service。

禁止：直接 `quantity * price`，直接操作 IndexedDB / `localStorage`，直接解析 NLP，直接决定交易是否合法。

### models

数据模型层。定义项目里的核心名词。

第一版对象：`DecimalString`、`Asset`、`Trade`、`TradeDraft`、`PriceSnapshot`、`FeeRule`、`Position`、`LedgerData`。

边界：只定义结构，不写计算，不写保存，不写 UI 文案。

### validators

校验层。判断输入能不能进入账本。

第一版校验：买卖类型、资产存在、数量大于 0、价格大于 0、总金额大于 0、手续费不小于 0、卖出不超持仓、`quantity * price` 与 `totalValue` 误差合理。

边界：不保存交易，不生成报表，不信任表单、NLP、CSV、导入文件等任何来源。

### calculators

计算层。账本的计算大脑。

第一版计算：持仓数量、成本基准、加权平均成本、已实现盈亏、市值、未实现盈亏。

边界：只接收已结构化数据，只输出计算结果；不读页面，不保存数据，不格式化展示文案。

### services

业务流程层。把多个模块串成一条业务动作。

例：新增交易流程：

```text
TradeDraft
  -> validateTradeDraft
  -> calculate current position if needed
  -> create Trade
  -> repository.saveTrade
  -> return success / error
```

它解决的是“先校验、再计算、再保存、再返回结果”的顺序问题。

边界：不直接碰 IndexedDB / `localStorage`，不把复杂计算写在自己里面，不定义核心类型。

### repositories

账本读写入口。给 service 提供业务语义的读写方法。

典型方法：`getLedgerData`、`listTrades`、`saveTrade`、`savePriceSnapshot`、`clearAll`、`exportData`、`importData`。

边界：Repository 懂“账本动作”，不懂底层存储细节；不负责交易是否合法，也不负责怎么算盈亏。

### adapters

外部适配层。把浏览器、文件、导入导出、未来 API 等外部细节包起来。

第一版可能有：`IndexedDBStorageAdapter`、`JsonImportAdapter`、`JsonExportAdapter`、`NoopEncryptionService`。未来可换成加密适配器、文件适配器、交易所导入适配器。

边界：Adapter 懂外部 API 怎么用，但不能让外部细节污染 service 和页面。

### utils

通用工具层。放跨业务复用的小工具。

Week2 核心工具：`decimalMath.ts`。

职责：DecimalString 转换、加减乘除、比较、误差判断、格式化入口。

边界：不是杂物间；凡是有明确业务归属的代码，不放这里。

## 骨架自检

后续写代码时，用这张表判断位置：

| 问题 | 放哪里 |
| --- | --- |
| 交易长什么样 | `models` |
| 交易是否合法 | `validators` |
| 均价和盈亏怎么算 | `calculators` |
| 新增交易的完整流程 | `services` |
| 账本怎么读写 | `repositories` |
| IndexedDB / JSON / 加密 / 文件 API 怎么接 | `adapters` |
| Decimal 怎么统一计算 | `utils/decimalMath.ts` |

如果一个文件同时承担两行以上职责，优先拆开。
