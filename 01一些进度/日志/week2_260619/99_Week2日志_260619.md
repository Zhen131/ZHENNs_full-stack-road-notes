# Day 1 记录：确定 Week2 骨架、Decimal 方案与验收标准

日期：2026-06-19

## 详细版

今天正式进入 Week2。Week2 的主题是“先让账本算得对”，但 Day1 没有急着写真实计算代码，而是先完成三件地基工作：项目骨架、Decimal 库选型、本周验收标准。今天的判断是：如果一开始就写 `positionCalculator`，很容易把页面、计算、校验、保存混在一起；先把骨架和通过线定清楚，后面 Day2 到 Day5 的实现才有明确落点。

第一，完成并精修了《01A_W2-项目骨架说明》。这份文档把 Week2 以后的代码分成 `app / components / models / validators / calculators / services / repositories / adapters / utils` 几层，并明确依赖方向：页面只负责展示和输入，service 负责编排业务流程，validator 负责判断输入是否合法，calculator 负责持仓、均价和盈亏计算，repository 负责账本读写入口，adapter 负责 IndexedDB、JSON、加密、文件 API 等外部细节，utils 只放跨业务复用的小工具。经过多轮压缩后，文档从长篇说明改成了工程卡片式说明，重点放在职责、边界和自检表上。

第二，完成并精修了《01B_W2-Decimal库选型说明》。本项目正式选定 `decimal.js`，数字策略定为三段式：保存用 `DecimalString`，计算用 `decimal.js Decimal`，展示用格式化字符串。选择 `decimal.js` 的理由是：`big.js` 更轻，但能力更窄；本项目后面要长期支撑均价、盈亏、误差校验、IndexedDB、导入导出和 benchmark，因此更需要一个完整稳定的十进制计算核心。文档同时约束了使用边界：Decimal 操作集中放在 `utils/decimalMath.ts`，页面、repository、adapter 不直接接触 Decimal。

第三，完成并精修了《01C_W2-Week2验收标准》。这份文档把“账本能算对”具体化为可测试的通过线：不用打开页面，只运行测试，就能证明核心账本计算正确。Week2 固定使用 Week1 的 5 条交易作为 golden test，并锁定手算基准：BTC 持仓 `0.00016388`、成本 `11`、均价约 `67122.28459848669`；ETH 持仓 `0.004854`、成本 `10`、均价约 `2060.1565718994643`；ADA 持仓 `251.2006`、成本 `62`、均价约 `0.24681469709865342`。本周不验收 UI、不验收存储、不验收加密，只验收模型、DecimalMath、计算器、校验器和 Vitest 测试。

今天还顺手统一了 Week1 到 Week4 的日志文件命名规则，全部改为带周标识的形式，例如 `01A_W2-项目骨架说明.md`。这样以后看文件名就能直接知道它属于第几周、第几天、第几个产出，避免不同周的 01、02、03 文件混在一起。同时新增了《00-文档产出规范》，把后续文档风格固定为“结论先行、动机一句话、边界清楚、自检表收尾”，避免以后每次生成文档都写得太长、太教学化，再反复删改。

另一个重要校准来自对前期文档和论文方案的反推：项目主线从“完整 8 大模块系统”收束为“论文核心账本原型”。保存层路线也正式统一为 `IndexedDBStorageAdapter + LedgerRepository + NoopEncryptionService`，不再走 `localStorage` 中间方案；Week4 直接落地 IndexedDB，Week6 再接 AES 真加密。Week3 的任务也被明确为删除 `DashboardShell` 中的硬编码视觉占位，改为渲染 Week2 的真实计算结果和 service 输出数据。

今天的成果不是功能代码，而是把 Week2 的施工图画清楚。下一步 Day2 将进入代码层：安装 `decimal.js`，落地 `models` 类型和 `utils/decimalMath.ts`，让纸面上的 `Trade / Asset / PriceSnapshot / Position / LedgerData` 开始变成真正可测试的 TypeScript 结构。

## 短版更新

今天完成 Week2 Day1 设计收口：确定项目分层骨架、选定 `decimal.js` 与 `DecimalString → Decimal → 格式化展示` 策略，写好 Week2 验收标准，并统一日志命名和文档产出规范。项目路线同步收束为论文核心账本原型，保存层正式改为 IndexedDB 先行。

# Day 2 记录：落地核心类型与 DecimalMath

日期：2026-06-20

## 详细版

今天完成 Week2 Day2 的核心代码落地：把 Week1 写在文档里的 `Trade / Asset / PriceSnapshot / FeeRule / Position / LedgerData / TradeDraft` 第一次变成 TypeScript 类型，并把十进制计算集中封装到 `src/utils/decimalMath.ts`。

第一，正式在 `src/models` 中建立了账本核心类型。`DecimalString` 继续作为保存层的小数表示，`Trade` 记录原始交易事实，`PriceSnapshot` 记录手动或未来 API 价格事实，`FeeRule` 记录手续费规则，`Position` 只作为计算结果，不作为原始事实保存，`LedgerData` 继续作为第一版账本整体数据结构。这个落地结果与 Week1 的核心数据结构设计保持一致，也为 Day3 的持仓计算器提供了稳定输入输出。

第二，正式安装并接入 `decimal.js`。本项目继续执行 Day1 定下的三段式数字策略：保存用 `DecimalString`，计算用 `decimal.js Decimal`，展示用格式化字符串。`decimalMath.ts` 集中提供转换、加减乘除、比较、正负零判断、绝对值、误差容忍判断和展示格式化入口。这样后续 `calculators` 和 `validators` 不需要到处 `new Decimal()`，也不应该在业务代码里裸写 `quantity * price`。

第三，补了一个轻量的 `npm run test:decimal` 验证脚本。今天没有提前引入完整 Vitest golden tests，因为 Day5 才是本周专门写 Vitest 的时间；但为了保证 DecimalMath 不是“只写了没验”，今天先用 TypeScript 编译加 Node assert 覆盖基础行为：`0.1 + 0.2 = 0.3`、小数字符串乘法、成本除以数量得到均价、数字比较不走字符串排序、`quantity * price` 与 `totalValue` 的误差容忍判断。

第四，按真实 GitHub 流程完成了一次完整分支开发。新建分支 `zhennn/w2-day2-types-decimalmath`，提交 commit `98913c4 feat: add ledger types and decimal math`，推送到 GitHub，创建并合并 Pull Request #2。合并后已把本地 `main` 快进到远端最新状态，并删除了已合并的本地和远端功能分支。今天也确认了工程习惯：短期 feature branch 合并后通常删除，历史保留在 PR 和 main 的提交记录里。

今天的验证结果：

npm run lint
npm run build

三项均通过。`test:decimal` 覆盖 8 条 DecimalMath 基础行为，`next lint` 无 warning / error，`next build` 成功完成静态构建。
