## 4. 一次性执行流程

1. **双仓库基线**：分别检查文档仓库和源码仓库的 `status / diff / diff-check / main...origin/main`；保留无关用户改动。未 fetch 时只能表述为“本地跟踪引用差异”。
2. **源码基线**：运行 `npm test && npm run lint && npm run build`。
3. **实现与定向测试**：完成第 2 节改动，先运行 `npm test -- DashboardShell`。
4. **人工验收**：按第 3 节完成 golden、价格、超卖、两类删除和宽窄屏检查。
5. **缺陷闭环**：`保存失败输入 -> 定位层级 -> 先写失败测试 -> 最小修复 -> 定向与全量复验 -> 重做人工场景`。
6. **收尾**：全量验证、边界复审、文档同步、Gate 结论。

可当场修复 UI 映射、表格、测试稳定性和已有 Week 6 链路回归。若需改变 golden 结果、手续费/币种/盈亏口径、schema、Repository / hydration 或 Week 7+ 范围，立即停止并请求确认。

## 5. 生产浏览器安全验收

- 禁止删除默认 `http://localhost:3000` 的 `local-first-trading-ledger`，也不清理用户浏览器数据。
- 运行 `npm run build`，然后用新端口（如 `npm run start -- -p 3106`）和全新/专用浏览器环境。页面非空时改用其他端口或环境，不删数据。
- 等待 hydration 结束；记录 URL、实际值和通过/失败，不得只写“正常”。验收后停止 `next start`。
- hydration 失败且表单不可用：Week 6 No-Go。仅自动保存失败：记为 Week 7 存储阻断，不得宣布存储 Gate 通过。
