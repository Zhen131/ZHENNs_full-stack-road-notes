## 6. 缺陷修复闭环

自动化或人工验收发现问题时，不得直接改代码碰运气，也不得只写 No-Go 后结束。

固定处理流程：

```text
保存失败输入和实际结果
-> 确认问题位于 UI、Service、Validator、Reducer 还是 Calculator
-> 先写能稳定复现的失败测试
-> 只修复造成失败的最小范围
-> 运行相关定向测试
-> 运行全量 test / lint / build
-> 在隔离浏览器中重做对应人工场景
-> 更新缺陷结论和 Gate 状态
```

允许当场修复：

- 表头、字段映射、`colSpan`、横向滚动和页面接线错误。
- 新增的 golden 交互测试本身的稳定性问题。
- 已有 Week 6 交易、价格、删除链路中有失败证据的回归缺陷。

必须停止并请求确认：

- 需要改变 Week 2 golden 数学结果。
- 需要改变手续费、币种、成本或盈亏口径。
- 需要改动 schema、导入导出、加密、图表或 benchmark。
- 需要重构 Repository、Adapter、hydration 或其他 Week 7+ 边界。
- 需要破坏性清理任何本地数据。

### 10.2 Week 7 存储 Gate 边界

Week 6 Go 不等于 Week 7 人工存储 Gate 已通过。

下一入口是：

```text
已实现的 IndexedDB 与 hydration 链路
-> 隔离浏览器刷新恢复
-> clear 契约
-> IndexedDB envelope 结构
-> 失败场景
-> Week 7 存储 Gate Go / No-Go
```

禁止把“进入 Week 7”理解为重新实现 IndexedDB、Repository 或 hydration。

### 10.3 No-Go 条件

任一情况发生，Week 6 React Gate 必须保持 No-Go：

- golden 数学结果与 Week 2 基线不一致。
- 表单成功文案出现，但列表或持仓没有同步。
- 超卖或不安全删除污染了 `LedgerData`。
- UI 重写了 Calculator 公式。
- 生产浏览器人工验收缺少实际结果。
- test、lint、build 任一失败。
- 存在未说明的越界改动或未阅读的未跟踪文件。

No-Go 时必须写明：失败场景、实际结果、影响范围、已尝试修复和下一步。
