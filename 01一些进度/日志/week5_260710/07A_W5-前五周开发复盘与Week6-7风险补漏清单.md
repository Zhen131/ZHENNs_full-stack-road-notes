## 1. 新增交易页面闭环尚未实现

**等级：P0**

**证据：**

- `DashboardShell` 只取出 `ledgerData`，没有使用 `dispatch`。
- “保存交易”按钮是普通 `type="button"`，没有提交处理。
- `tradeService` 已实现，但尚未被页面调用。

**问题：**

当前页面只能读取和展示账本，不能通过生产 UI 新增交易。表单字段到 `TradeDraft` 的映射、Service 错误展示和成功后的 dispatch 都还不存在。

**结论：**

这是计划内的未完成功能，不是已实现代码的回归；但它是 Week 6 Gate 4 的核心阻断项。未关闭前不能把页面描述成“可录入交易”。

**建议处理：**

Week 6 先完成真实的 `表单 -> tradeService -> dispatch -> 列表/持仓更新`，再进行后续功能。

## 2. 直接删除交易可能破坏后续持仓时间线

**等级：P0**

**证据：**

- `trade/delete` 会直接按 ID 过滤交易，不校验删除后的时间线。
- `positionCalculator` 遇到卖出数量超过当时持仓时会抛错。
- Week 6 当前计划准备让 UI 直接使用既有 `trade/delete`。

**问题：**

后续卖出可能依赖前面的买入。如果删除被依赖的买入，剩余交易会形成超卖时间线。页面重新计算持仓时可能抛错。

示例：

```text
买入 41.58
买入 126.6825
卖出 82.9381
```

删除第二笔买入后，第一笔买入不足以支撑卖出。

**结论：**

这是本次复盘发现的真实潜在缺陷。Reducer 不做业务校验是正确边界，但 UI 不能无条件直接 dispatch 删除。

**建议处理：**

Week 6 接删除功能时，先验证删除后的完整交易时间线；会产生超卖时应阻止删除。

## 3. 价格录入生产链路尚未实现

**等级：P0**

**证据：**

- 页面“保存价格”按钮没有事件处理。
- Reducer 没有 `priceSnapshot/add` action。
- 源码只有 `PriceSnapshot` 类型、Calculator 支持和测试 fixture。
- 当前没有生产价格创建 Service。

**问题：**

系统能根据已有 `PriceSnapshot[]` 计算市值和未实现盈亏，但用户无法从生产页面创建价格事实。

**结论：**

这是 Week 6 已知的计划内缺口，不是 Calculator 的问题。没有完成价格 Service、Reducer action 和 UI dispatch 前，Gate 5 不能通过。

**建议处理：**

Week 6 按既定顺序补齐价格创建入口和页面写入链，并用 BTC `70000 USD` 场景验收。

## 4. hydrate 前缺少完整 LedgerData 运行时校验

**等级：P0**

**证据：**

- 当前没有 IndexedDB、Repository 或 hydrate 实现。
- 现有 `tradeValidator` 校验候选交易及其持仓时间线，但不校验任意完整 `LedgerData`。
- 源码没有完整 `LedgerData` 的运行时 Validator。
- TypeScript 类型不能校验从 IndexedDB 或 JSON 读取的真实数据。

**问题：**

未来保存数据可能包含未知 schema、重复 ID、坏 Decimal、非法类型、坏资产引用或不合法交易时间线。如果直接替换 reducer state，可能污染计算和页面。

**结论：**

该风险目前不可触发，因为持久化尚未实现；但它是 Week 7 的数据安全前置条件。只检查 `schemaVersion` 不足以关闭风险。

**建议处理：**

任何保存数据进入 reducer 前，必须先完成整账运行时校验；失败时不得替换当前 state，也不得回写存储。

## 5. 缺少真实 Dashboard 交互回归

**等级：P1**

**证据：**

- 当前 Dashboard 测试使用 Node 环境和静态 HTML 渲染。
- 测试能证明纯视图和 Reducer 输出契约。
- 当前没有 jsdom、Testing Library、user-event 或等价交互测试环境。
- 现有测试明确不证明真实 UI dispatch。

**问题：**

静态测试无法证明用户填写表单、点击保存、Service 返回、dispatch 和页面更新这一整条链真正可用。

**结论：**

原文将其列为 P0 偏高。它本身不会破坏账本数据，因此调整为 P1；但如果没有交互回归，Week 6 可能出现“测试全绿、按钮实际不可用”的假通过。

**建议处理：**

在关闭 Gate 4 时增加真实用户操作测试，至少证明合法交易同时更新列表和持仓，非法交易不改变账本。

## 6. 持久化接口和 hydration 时序尚未最终统一

**等级：P1**

**证据：**

- Week 1 设计中，StorageAdapter 直接读写 `LedgerData`，空库返回默认账本，并由 Adapter 调用 EncryptionService。
- Week 7 新计划要求空库返回明确“无数据”，Repository 依赖 StorageAdapter 与 EncryptionService。
- Week 7 已提出 hydration 前禁止写、连续写有序、失败保留旧数据，但尚无最终代码契约。

**问题：**

旧设计和新计划对空库语义、加密调用位置和数据载荷存在差异。如果开工前不统一，Repository、Adapter 和 Noop Encryption 可能重复负责或遗漏职责。

同时，若 hydration 和自动保存时序处理错误，初始空账本可能覆盖旧数据，连续异步写也可能发生旧状态覆盖新状态。

**结论：**

这不是当前代码缺陷，因为持久化还没有实现；原文列为 P0 偏高，调整为 P1 架构前置风险。Week 7 Day 1 未拍板前不应开始 IndexedDB 实现。

**建议处理：**

Week 7 Day 1 只统一接口、空库语义、加密位置和 hydration 写入规则，再进入 Adapter 与 Repository。
