# Day 2 核心数据结构设计草稿

日期：2026-06-13

状态：讨论草稿

目标：定义第一版账本的数据长什么样，让交易记录、价格记录和持仓计算都有稳定依据。

---

## 设计原则

第一版先把数据分成三类：

1. 原始事实：用户手动输入并保存的数据，例如交易和价格。
2. 基础配置：系统用来识别资产的数据，例如 BTC、ETH、ADA。
3. 计算结果：从原始事实推导出来的数据，例如持仓、均价和盈亏。

其中 `Trade`、`Asset`、`PriceSnapshot` 应该保存。

`Position` 第一版建议先作为计算结果，不手动保存。这样可以避免交易修改后，持仓数据和交易数据对不上。

第一版遵循单一职责原则：

- `Trade` 只记录交易事实。
- `PriceSnapshot` 只记录价格事实。
- `FeeRule` 只描述手续费规则。
- `FeeCalculator` 只负责计算手续费。
- `PositionCalculator` 只负责计算持仓和盈亏。
- 页面只负责展示和输入，不直接承担计算和保存逻辑。

---

## 数据总结构

第一版本地保存的数据可以长这样：

```ts
type LedgerData = {
  schemaVersion: 1;
  assets: Asset[];
  trades: Trade[];
  priceSnapshots: PriceSnapshot[];
  feeRules: FeeRule[];
};
```

说明：

- `schemaVersion` 用来标记当前数据结构版本。
- `assets` 保存资产基础信息。
- `trades` 保存所有交易记录。
- `priceSnapshots` 保存手动输入的价格记录。
- `feeRules` 保存手续费规则，第一版可以只配置一个手动百分比规则。
- `positions` 暂时不保存，需要展示时从 `trades` 和 `priceSnapshots` 计算。

## DecimalString：高精度数字字符串

加密资产数量可能非常小，例如 `0.00016388` BTC。为了避免 JavaScript 普通 `number` 带来的小数误差，第一版建议把需要精确保存和计算的小数保存成字符串。

```ts
type DecimalString = string;
```

示例：

```ts
const quantity: DecimalString = "0.00016388";
const price: DecimalString = "67121.7";
const totalValue: DecimalString = "11";
```

规则：

- 保存层保存十进制字符串。
- 计算层统一通过 `DecimalMath` 或未来的 decimal 库处理。
- 页面展示时再格式化成用户容易看的数字。
- 不在业务代码里到处直接写 `quantity * price`。

参考 Frappe Books 的思路：

- Frappe Books 对金额使用 `pesa` 的 `Money` 类型，而不是到处裸算金额。
- 它区分内部计算精度和展示精度：默认内部精度是 11，展示精度是 2。
- 它的数量字段仍然多处使用普通 `number`，但我们的项目面对加密资产数量，数量本身也需要更谨慎。

## Asset：资产

`Asset` 用来说明一个资产是什么。

```ts
type Asset = {
  id: string;
  symbol: string;
  name: string;
  quoteCurrency: string;
  createdAt: string;
  updatedAt: string;
};
```

字段说明：

| 字段              | 含义      | 第一版要求             |
| --------------- | ------- | ----------------- |
| `id`            | 资产唯一 ID | 必填，例如 `asset-btc` |
| `symbol`        | 资产代码    | 必填，例如 `BTC`       |
| `name`          | 资产名称    | 必填，例如 `Bitcoin`   |
| `quoteCurrency` | 默认计价货币  | 必填，第一版默认 `USD`    |
| `createdAt`     | 创建时间    | 系统生成              |
| `updatedAt`     | 更新时间    | 系统生成              |

第一版样例：

```ts
const btc: Asset = {
  id: "asset-btc",
  symbol: "BTC",
  name: "Bitcoin",
  quoteCurrency: "USD",
  createdAt: "2026-06-13T00:00:00Z",
  updatedAt: "2026-06-13T00:00:00Z",
};
```

## Trade：交易

`Trade` 是第一版最重要的原始数据。每一笔买入或卖出都应该保存为一条交易。

```ts
type Trade = {
  id: string;
  occurredAt: string;
  timePrecision: "day" | "minute" | "second";
  type: "buy" | "sell";
  assetSymbol: string;
  quantity: DecimalString;
  quantitySortKey?: DecimalString;
  price: DecimalString;
  totalValue: DecimalString;
  totalValueSortKey?: DecimalString;
  currency: string;
  fee: DecimalString;
  feeCurrency: string;
  feeRuleId?: string;
  note?: string;
  rawText?: string;
  createdAt: string;
  updatedAt: string;
};
```

字段说明：

| 字段              | 含义      | 第一版要求                      |
| --------------- | ------- | -------------------------- |
| `id`            | 交易唯一 ID | 必填                         |
| `occurredAt`    | 交易发生时间  | 必填，例如 `2026-04-02`         |
| `timePrecision` | 时间精度    | 必填，第一版主要用 `day`            |
| `type`          | 交易类型    | 必填，`buy` 或 `sell`          |
| `assetSymbol`   | 资产代码    | 必填，例如 `BTC`                |
| `quantity`      | 交易数量    | 必填，必须大于 0                  |
| `quantitySortKey` | 数量查询键 | 可选，给数据库排序和筛选使用             |
| `price`         | 成交均价    | 必填，必须大于 0                  |
| `totalValue`    | 交易总金额   | 必填，通常等于 `quantity * price` |
| `totalValueSortKey` | 总金额查询键 | 可选，给数据库排序和筛选使用           |
| `currency`      | 交易计价货币  | 必填，第一版默认 `USD`             |
| `fee`           | 手续费金额   | 必填，无手续费时为 0                |
| `feeCurrency`   | 手续费货币   | 必填，第一版默认和 `currency` 一致    |
| `feeRuleId`     | 手续费规则 ID | 可选，用来说明手续费由哪个规则算出          |
| `note`          | 用户备注    | 可选                         |
| `rawText`       | 原始输入文本  | 可选，建议保留                    |
| `createdAt`     | 记录创建时间  | 系统生成                       |
| `updatedAt`     | 记录更新时间  | 系统生成                       |

第一版校验规则：

- `type` 只能是 `buy` 或 `sell`。
- `quantity`、`price`、`totalValue` 必须大于 0。
- `fee` 不能小于 0。
- `assetSymbol` 必须能在 `assets` 里找到。
- 卖出时不能超过当前持仓数量。
- `totalValue` 可以和 `quantity * price` 有轻微误差，因为真实交易可能有四舍五入。

手续费说明：

- 第一版手续费可以由用户填写一个百分比，然后系统自动计算。
- 计算出的手续费金额保存在 `Trade.fee`。
- 第一版手续费只记录，不计入平均成本和盈亏计算。
- 未来可以增加 OKX、Binance、股票券商等不同规则，不改变交易主结构。

## 查询和排序策略

`DecimalString` 适合精确保存，但不能直接当作数字排序。

例如字符串排序可能得到：

```text
"10" < "2"
```

但数字意义上应该是：

```text
2 < 10
```

所以第一版先定下原则：

- 保存层保留原始十进制字符串，例如 `quantity: "0.00016388"`。
- 页面少量数据排序时，可以在内存中用 decimal 库比较。
- 未来进入数据库查询时，可以增加查询用字段，例如 `quantitySortKey`、`totalValueSortKey`。
- 查询用字段是派生字段，不是用户原始输入；如果丢失，可以从原始 DecimalString 重新生成。

查询用字段有两种候选方案：

### 方案 A：规范化十进制字符串

把数字标准化成适合字典序比较的字符串，例如统一整数位和小数位长度。

适合：

- 数量级不夸张的本地查询。
- 不想太早引入每个资产独立 atomic 精度。

风险：

- 要设计固定长度和负数规则。
- 不如数据库原生数字索引自然。

### 方案 B：atomic integer

为每个资产定义精度，把数量转成最小单位整数。

例如 BTC 精度为 8：

```text
0.01 BTC = 1,000,000 satoshi
```

这个方案适合处理资产数量，尤其适合 BTC、ETH、SOL 这类有明确最小单位的资产。

风险：

- 不同资产精度不同，需要 `Asset.decimals`。
- 股票、基金、价格、汇率不一定天然适合这种模型。
- 价格和均价经常是 USD/BTC 这类比例，仍然需要 decimal 计算。

第一版结论：

- 不把 atomic integer 作为第一版主结构。
- 可以在 `Asset` 里预留 `decimals`，未来需要时再生成 `quantityAtomic`。
- 第一版重点仍然是 `DecimalString + DecimalMath`。

## PriceSnapshot：价格记录

`PriceSnapshot` 用来保存某个时间点的资产价格。第一版由用户手动输入，未来可以接实时价格 API。

```ts
type PriceSnapshot = {
  id: string;
  assetSymbol: string;
  price: DecimalString;
  currency: string;
  recordedAt: string;
  source: "manual" | "api";
  note?: string;
  createdAt: string;
  updatedAt: string;
};
```

字段说明：

| 字段 | 含义 | 第一版要求 |
| --- | --- | --- |
| `id` | 价格记录唯一 ID | 必填 |
| `assetSymbol` | 资产代码 | 必填，例如 `BTC` |
| `price` | 当前价格 | 必填，必须大于 0 |
| `currency` | 价格计价货币 | 必填，第一版默认 `USD` |
| `recordedAt` | 价格对应的时间 | 必填 |
| `source` | 价格来源 | 必填，第一版为 `manual` |
| `note` | 备注 | 可选 |
| `createdAt` | 记录创建时间 | 系统生成 |
| `updatedAt` | 记录更新时间 | 系统生成 |

为什么要有 `source`：

- 第一版手动输入价格时，`source` 是 `manual`。
- 未来接 API 时，`source` 可以是 `api`。
- 数据结构不用推翻，只需要增加来源类型或提供商字段。

---

## FeeRule：手续费规则

`FeeRule` 用来描述某个平台或券商的手续费计算方式。第一版先支持一个手动百分比规则。

```ts
type FeeRule = {
  id: string;
  name: string;
  platform: string;
  type: "percentage";
  rate: DecimalString;
  currency: string;
  createdAt: string;
  updatedAt: string;
};
```

第一版样例：

```ts
const okxManualFeeRule: FeeRule = {
  id: "fee-rule-okx-manual",
  name: "OKX manual fee",
  platform: "OKX",
  type: "percentage",
  rate: "0.001",
  currency: "USD",
  createdAt: "2026-06-13T00:00:00Z",
  updatedAt: "2026-06-13T00:00:00Z",
};
```

计算逻辑：

```text
fee = totalValue * rate
```

例如：

```text
totalValue = 100 USD
rate = 0.001
fee = 0.1 USD
```

## Position：持仓

`Position` 第一版建议作为计算结果，不直接保存。

```ts
type Position = {
  assetSymbol: string;
  quantity: DecimalString;
  averageCost: DecimalString;
  costBasis: DecimalString;
  latestPrice?: DecimalString;
  marketValue?: DecimalString;
  realizedPnl: DecimalString;
  unrealizedPnl?: DecimalString;
  currency: string;
};
```

字段说明：

| 字段              | 含义     | 来源                        |
| --------------- | ------ | ------------------------- |
| `assetSymbol`   | 资产代码   | 来自 `Asset` / `Trade`      |
| `quantity`      | 当前持仓数量 | 从交易计算                     |
| `averageCost`   | 当前平均成本 | 从买入和卖出计算                  |
| `costBasis`     | 当前持仓成本 | 从交易计算                     |
| `latestPrice`   | 最新价格   | 从 `PriceSnapshot` 取最新一条   |
| `marketValue`   | 当前市值   | `quantity * latestPrice`  |
| `realizedPnl`   | 已实现盈亏  | 卖出时计算                     |
| `unrealizedPnl` | 未实现盈亏  | `marketValue - costBasis` |
| `currency`      | 计价货币   | 第一版默认 `USD`               |

第一版暂定计算口径：

- 买入会增加持仓数量和持仓成本。
- 卖出会减少持仓数量，并按当前平均成本计算已实现盈亏。
- 平均成本法使用加权平均成本，不做 FIFO。
- 手续费第一版自动计算并记录，但不计入平均成本和盈亏。

加权平均成本说明：

```text
平均成本 = 当前总持仓成本 / 当前持仓数量
```

采用这个方案的原因：

- 计算简单。
- 数据量少。
- 出错概率低。
- 原始交易仍然完整保留，未来可以继续做 FIFO、单笔收益回溯、最佳交易分析等扩展功能。

## 汇总缓存策略

`Position` 第一版作为计算结果，不属于用户原始数据。

但这不代表未来每次打开页面都必须从第一条交易重新计算到最后一条交易。

为了避免十万、百万级交易记录导致刷新变慢，未来可以增加派生缓存：

```ts
type PositionCache = {
  assetSymbol: string;
  quantity: DecimalString;
  averageCost: DecimalString;
  costBasis: DecimalString;
  realizedPnl: DecimalString;
  currency: string;
  lastTradeId: string;
  lastCalculatedAt: string;
  schemaVersion: 1;
};
```

缓存规则：

- `Trade` 和 `PriceSnapshot` 是原始数据。
- `PositionCache` 是派生数据，可以保存，但不能当作真相来源。
- 新增交易时，只用新交易更新对应资产的缓存。
- 修改或删除历史交易时，标记缓存失效，然后从受影响时间点重新计算。
- 如果缓存丢失，可以从全部交易重新生成。

这相当于：

```text
原始交易 = 小票
PositionCache = 已经算好的汇总结果
```

第一版实现可以先全量计算，因为样例数据很少。

但第一版设计文档要承认未来路线：

- 小数据量：打开页面时全量计算。
- 中等数据量：按资产增量更新缓存。
- 大数据量：按资产和时间分段生成快照，例如月度快照或每 1000 笔交易一个 checkpoint。

这样可以避免每次刷新都重新计算全部历史记录。

## 已知风险

### 1. 数字精度风险

加密资产数量可能很小，例如 `0.00016388` BTC。JavaScript 的普通 `number` 可能出现小数误差。

第一版保存层使用 `DecimalString`，计算层统一通过 `DecimalMath` 处理。未来正式写代码时，可以接入 decimal 库，避免浮点数误差扩散到业务代码里。

### 2. 手续费口径风险

手续费可能用 USD、资产本身或其他币种支付。第一版先记录 `fee` 和 `feeCurrency`，但暂时不做复杂成本处理。

第一版只支持手动百分比规则，例如 `0.1%` 或 `0.2%`。未来再扩展 OKX、Binance、股票券商等平台规则。

### 3. 平均成本口径风险

不同投资账本可能使用加权平均、FIFO 或其他方式。第一版先使用加权平均成本，并在文档中写清楚。

### 4. 多币种风险

第一版只支持 USD。如果未来加入 CNY、USDT、EUR，需要增加汇率或统一计价规则。

### 5. 修改历史风险

如果用户修改旧交易，所有后续持仓和盈亏都需要重新计算。第一版先不保存 `Position`，可以降低这个风险。

### 6. 价格来源风险

手动价格和 API 价格可能不一致。`PriceSnapshot.source` 先保留来源字段，未来可以继续加 `provider`。

### 7. 字符串排序和筛选风险

`DecimalString` 不能直接用普通字符串排序和筛选。第一版少量数据可以用 decimal 库在内存中比较。未来数据库查询需要增加查询用字段或 atomic integer 字段。

### 8. 大数据量重算风险

如果交易记录达到十万、百万级，每次刷新都从头计算 `Position` 会变慢。未来需要 `PositionCache`、按资产增量计算、时间分段快照等机制。

---

## 已决定的问题

1. 第一版只支持 `USD` 计价。
2. 资产代码先直接使用 `BTC`、`ETH`、`ADA` 这类 symbol。
3. 卖出成本计算使用加权平均成本。
4. `Position` 只作为计算结果，不写入本地保存。
5. 本地保存只聚焦交易记录和价格记录。
6. 手续费第一版用手动百分比规则自动计算，保存最终手续费金额。
7. 手续费第一版不计入平均成本和盈亏。
8. 第一版保存格式继续使用 `DecimalString`。
9. atomic integer 暂时不作为第一版主结构，只作为未来优化方向。
