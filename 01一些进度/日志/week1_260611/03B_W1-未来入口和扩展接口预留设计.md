# 未来入口和扩展接口预留设计

日期：2026-06-14

状态：扩展设计草稿

目标：提前标出未来功能要接入的位置，避免第一版把页面、计算、保存、行情、图表、AI 输入等逻辑写死。第一版不实现复杂功能，但代码结构要允许未来像搭乐高一样逐块接入。

---

## 一句话结论

Local-First Trading Ledger 第一版可以很简单，但结构不能乱。

第一版要优先遵守几个核心设计原则：

- SRP：Single Responsibility Principle，单一职责原则。
- OCP：Open Closed Principle，开闭原则。
- LSP：Liskov Substitution Principle，里氏替换原则。
- DIP：Dependency Inversion Principle，依赖倒置原则。

也就是：

```text
每个模块只负责一件事。
未来新增功能时，优先新增模块，而不是大面积修改旧模块。
同一类接口的不同实现，应该可以互相替换。
高层业务不直接依赖低层实现，而是依赖抽象接口。
```

第一版真正要守住的底线：

> 页面只负责展示和收集输入，不直接负责计算、不直接负责保存、不直接负责解析自然语言、不直接调用行情 API。

- 自然语言输入交易

## 核心原则

### 1. SRP：单一职责原则

每个模块只负责一个明确职责。

### 不好的写法

```text
Dashboard 页面：
- 直接读取浏览器存储
- 解析交易
- 校验表单
- 计算持仓
- 调 API 获取价格
- 画图
- 保存数据
- 处理 AI 问答
```

这样会导致 Dashboard 变成一个巨大的混乱文件。

### 好的拆法

```text
页面组件：负责展示和用户输入
TradeValidator：负责校验交易
TradeRepository：负责保存和读取交易
PositionCalculator：负责计算持仓和盈亏
PriceProvider：负责提供价格
PriceSnapshotRepository：负责保存手动价格
ChartDataService：负责把数据整理成图表需要的格式
NaturalLanguageTradeParser：负责把自然语言转成交易草稿
AgentQueryService：负责未来问答查询
```

页面只调用这些服务，不自己承担所有逻辑。

---

### 2. OCP：开闭原则

系统应该对扩展开放，对修改关闭。

意思不是永远不改旧代码，而是未来新增功能时，尽量通过增加新实现来扩展。

### 示例：价格来源

第一版只有手动价格：

```text
ManualPriceProvider
```

未来接 CoinGecko：

```text
CoinGeckoPriceProvider
```

未来接 Binance：

```text
BinancePriceProvider
```

页面不应该关心价格到底来自哪里。页面只问：

```text
PriceService：给我 BTC 当前价格。
```

### 示例：交易输入来源

第一版只有表单输入：

```text
ManualTradeInput
```

未来可以新增：

```text
NaturalLanguageTradeInput
VoiceTradeInput
CSVImportTradeInput
ExchangeImportTradeInput
```

但它们最终都应该转成同一种 `TradeDraft`，再交给统一的校验和保存流程。

---

### 3. LSP：里氏替换原则

LSP 全称是 Liskov Substitution Principle，中文常叫里氏替换原则。

这个原则来自 Barbara Liskov。它不是黑天鹅原则，重点是“子类型必须能够替换父类型”。

用更直白的话说：

```text
如果一个模块依赖某个接口，那么这个接口的任何实现都应该可以替换进去，并且不破坏原来的程序行为。
```

用本项目的话说：

```text
Dashboard 不应该只适配 ManualPriceProvider。
如果未来换成 CoinGeckoPriceProvider、BinancePriceProvider 或 OKXPriceProvider，
只要它们都实现 PriceProvider 接口，Dashboard 和 PriceService 就不应该崩。
```

### 示例：价格来源可以替换

所有价格来源都应该遵守同一个接口：

```ts
interface PriceProvider {
  name: string;
  getLatestPrice(assetSymbol: string, currency: string): Promise<PriceQuote>;
}
```

那么这些实现都应该可以被替换：

```text
ManualPriceProvider
CoinGeckoPriceProvider
BinancePriceProvider
OKXPriceProvider
```

不好的设计：

```text
ManualPriceProvider 返回 PriceSnapshot
CoinGeckoPriceProvider 返回 API 原始 JSON
BinancePriceProvider 返回 number
OKXPriceProvider 返回另一个字段名
```

这样表面上都叫价格来源，但实际上不能互相替换，违反 LSP。

好的设计：

```text
所有 PriceProvider 都返回统一的 PriceQuote。
所有错误都转换成统一错误类型或统一失败结果。
所有价格数字都使用 DecimalString。
```

### 示例：保存方式可以替换

页面和 Service 不应该关心底层保存方式。

这些实现未来应该可以互换，但论文主线的第一版正式实现直接使用 IndexedDB：

```text
IndexedDBStorageAdapter
EncryptedIndexedDBStorageAdapter
FileStorageAdapter
CloudSyncStorageAdapter
```

它们都实现：

```ts
interface StorageAdapter {
  read(): Promise<LedgerData>;
  write(data: LedgerData): Promise<void>;
}
```

只要替换保存方式，不需要重写页面和业务逻辑，就说明 LSP 做得比较好。

### 本项目里的 LSP 规则

- 同一个接口的所有实现，输入和输出要稳定。
- 不要让某个实现偷偷要求额外字段，导致其他实现无法替换。
- 不要让页面根据 provider 名字写大量 `if else`。
- API 原始数据必须先转换成系统内部统一结构。
- 不同 StorageAdapter 都应该表现得像同一种保存层。

---

### 4. DIP：依赖倒置原则

DIP 全称是 Dependency Inversion Principle，中文常叫依赖倒置原则。

它的经典表述是：

```text
高层模块不应该依赖低层模块。
高层模块和低层模块都应该依赖抽象。
抽象不应该依赖细节。
细节应该依赖抽象。
```

用本项目的话说：

```text
Dashboard / TradeService / PositionService 是高层业务。
IndexedDB / Web Crypto / CoinGecko API / Binance API 是低层细节。

高层业务不应该直接 import IndexedDB 封装、Web Crypto 细节或某个 API client。
高层业务应该依赖 LedgerRepository、StorageAdapter、PriceProvider 这些抽象接口。
```

### 不好的依赖方向

```text
Dashboard
-> IndexedDB
-> CoinGecko API
-> Decimal 计算
-> Agent API
```

这样 Dashboard 直接依赖太多低层细节。

后果：

- 换保存方式要改页面。
- 换价格 API 要改页面。
- 加密保存要改页面。
- 做测试很困难。
- 页面文件越来越大。

### 好的依赖方向

```text
Dashboard
-> PositionService
-> LedgerRepository
-> StorageAdapter

Dashboard
-> PriceService
-> PriceProvider

Dashboard
-> TradeService
-> TradeValidator
-> TradeRepository
```

高层只依赖抽象服务。

低层实现这些抽象：

```text
IndexedDBStorageAdapter implements StorageAdapter
ManualPriceProvider implements PriceProvider
CoinGeckoPriceProvider implements PriceProvider
```

### 本项目里的 DIP 规则

- 页面不直接调用 IndexedDB、`localStorage` 或其他浏览器存储 API。
- 页面不直接调用 CoinGecko、Binance、OKX。
- 页面不直接处理加密。
- 页面不直接解析自然语言。
- Service 依赖 Repository / Provider / Calculator / Validator 的抽象接口。
- 具体实现可以在项目启动、配置层或工厂函数里组装。

### 简单依赖图

```text
UI Page
  -> Service
    -> Interface
      -> Concrete Implementation

例如：

Dashboard
  -> PriceService
    -> PriceProvider
      -> ManualPriceProvider
      -> CoinGeckoPriceProvider
      -> BinancePriceProvider
```

这样未来加新功能时，大多数情况是在低层新增实现，然后在配置层接进去，而不是改 Dashboard 主逻辑。

## 第一版稳定核心

未来功能再多，也应该围绕几个稳定核心对象扩展。

第一版稳定核心：

```text
Asset
Trade
PriceSnapshot
Position
FeeRule
```

它们的职责：

| 对象 | 职责 | 是否保存 |
| --- | --- | --- |
| Asset | 说明资产是什么 | 保存 |
| Trade | 记录一笔买入或卖出事实 | 保存 |
| PriceSnapshot | 记录某个时间点的价格事实 | 保存 |
| Position | 从交易和价格计算出来的持仓结果 | 第一版不保存 |
| FeeRule | 描述手续费规则 | 保存或配置 |

未来功能不应该绕过这些核心对象。

例如：

- K 线图可以新增 `Candle`，但不能破坏 `Trade`。
- 实时价格 API 可以生成 `PriceSnapshot`，但不能让页面直接依赖某个 API。
- Agent 问答可以读取 `Trade`、`PriceSnapshot`、`Position`，但不能自己偷偷改数据。

## 入口 1：价格来源 PriceProvider

第一版只有手动价格输入，但未来要接实时价格 API。

因此要先设计价格来源接口。

```ts
type PriceProviderName = "manual" | "coingecko" | "binance" | "custom";

type PriceQuote = {
  assetSymbol: string;
  price: string;
  currency: string;
  recordedAt: string;
  source: PriceProviderName;
};

interface PriceProvider {
  name: PriceProviderName;
  getLatestPrice(assetSymbol: string, currency: string): Promise<PriceQuote>;
}
```

### 第一版实现

```text
ManualPriceProvider
```

它不联网，只读取用户手动输入的最新价格。

### 未来实现

```text
CoinGeckoPriceProvider
BinancePriceProvider
OKXPriceProvider
YahooFinancePriceProvider
```

### 规则

- 页面不直接调用 CoinGecko、Binance 或 OKX。
- 页面只调用 `PriceService`。
- `PriceService` 根据配置选择具体 `PriceProvider`。
- API 返回的数据要先转换成 `PriceSnapshot` 或 `PriceQuote`，再进入系统。

## 入口 3：资产净值曲线 NetWorthSeries

K 线是资产本身的价格走势。

资产净值曲线是“我的组合价值”走势。

它们不是一回事，需要分开设计。

### 资产 K 线

```text
BTC 自己的价格怎么涨跌。
```

### 我的资产净值

```text
我持有 BTC、ETH、ADA 后，我的总资产价值怎么变化。
```

未来可以有一个专门服务：

```ts
type NetWorthPoint = {
  date: string;
  totalMarketValue: string;
  totalCostBasis: string;
  unrealizedPnl: string;
  currency: string;
};

interface NetWorthService {
  getSeries(params: {
    from: string;
    to: string;
    interval: "day" | "week" | "month";
  }): Promise<NetWorthPoint[]>;
}
```

### 规则

- 净值曲线从交易、持仓和价格快照计算。
- 不要把净值曲线数据直接手动写进页面。
- 未来可以缓存，但缓存是派生数据，不是真相来源。

## 入口 4：交易输入 TradeInputSource

第一版只做表单输入。

但未来会有自然语言、语音和导入。

所以要把“输入来源”和“交易保存”分开。

```ts
type TradeDraft = {
  occurredAt: string;
  timePrecision: "day" | "minute" | "second";
  type: "buy" | "sell";
  assetSymbol: string;
  quantity: string;
  price: string;
  totalValue: string;
  currency: string;
  fee: string;
  feeCurrency: string;
  note?: string;
  rawText?: string;
};

interface TradeInputSource {
  name: string;
  createDraft(input: unknown): Promise<TradeDraft>;
}
```

### 第一版实现

```text
ManualTradeFormSource
```

表单字段直接转成 `TradeDraft`。

### 未来实现

```text
NaturalLanguageTradeSource
VoiceTradeSource
CSVTradeImportSource
ExchangeTradeImportSource
```

### 统一流程

不管交易来自哪里，都必须走同一条流程：

```text
输入来源
-> TradeDraft
-> TradeValidator
-> TradeService
-> TradeRepository
-> StorageAdapter
```

这样未来新增 NLP 输入时，不需要重写保存和计算逻辑。

## 入口 5：自然语言解析 NaturalLanguageTradeParser

未来用户可能输入：

```text
以均价 67121.7 买入 BTC 0.00016388 个，价值 11 USD，26/04/02
```

解析器应该只负责解析，不负责保存。

```ts
interface NaturalLanguageTradeParser {
  parse(text: string): Promise<TradeDraft>;
}
```

### 解析器职责

- 识别买入 / 卖出。
- 识别资产。
- 识别数量。
- 识别均价。
- 识别总金额。
- 识别日期。
- 保留原始文本。

### 解析器不负责

- 不负责保存。
- 不负责计算持仓。
- 不负责判断超卖。
- 不负责写入 IndexedDB 或任何浏览器存储。

这些应该交给 `TradeValidator`、`TradeService` 和 `TradeRepository`。

## 入口 6：Agent 问答 AgentQueryService

未来可能会问：

```text
我现在 BTC 持仓是多少？
我 ADA 的平均成本是多少？
我今年卖出赚了多少钱？
如果 BTC 到 80000，我未实现盈亏是多少？
```

Agent 不能直接乱读乱改所有数据。

应该通过一个受控查询服务访问账本。

```ts
interface AgentQueryService {
  getPositions(): Promise<Position[]>;
  getTrades(filter?: TradeFilter): Promise<Trade[]>;
  getLatestPrices(): Promise<PriceSnapshot[]>;
  explainPosition(assetSymbol: string): Promise<string>;
}
```

### 规则

- Agent 默认只读。
- Agent 如果要写入交易，必须生成 `TradeDraft`，并经过用户确认。
- Agent 不能绕过 `TradeValidator`。
- Agent 不能直接写 storage。

```ts
type ReportResult = {

interface ReportGenerator {
```

```text

新增 FeeSummaryReportGenerator
```

而不是把所有报告逻辑塞进一个大页面。

```ts
interface FeeCalculator {
```

```text
ManualFeeCalculator
PercentageFeeCalculator
```

### 未来

```text
OKXFeeCalculator
BinanceFeeCalculator
StockBrokerFeeCalculator
```

## 入口 9：保存层 LedgerRepository 和 StorageAdapter

保存层是后续扩展最重要的地方之一。

页面不能直接操作：

```text
IndexedDB
SQLite
文件系统
云同步
```

统一从 Repository 进入。

```ts
interface LedgerRepository {
  getLedgerData(): Promise<LedgerData>;
  saveTrade(trade: Trade): Promise<void>;
  listTrades(): Promise<Trade[]>;
  savePriceSnapshot(snapshot: PriceSnapshot): Promise<void>;
  listPriceSnapshots(): Promise<PriceSnapshot[]>;
}

interface StorageAdapter {
  read(): Promise<LedgerData>;
  write(data: LedgerData): Promise<void>;
}
```

### 第一版

```text
IndexedDBStorageAdapter
```

### 未来

```text
EncryptedIndexedDBStorageAdapter
FileStorageAdapter
CloudSyncStorageAdapter
```

### 规则

- 页面只调用 Service。
- Service 调用 Repository。
- Repository 调用 StorageAdapter。
- 更换保存方式时，页面不应该大改。

## 入口 10：加密 EncryptionService

未来账本数据需要加密。

加密不应该散落在各个页面里。

```ts
interface EncryptionService {
  encrypt(plainText: string): Promise<string>;
  decrypt(cipherText: string): Promise<string>;
}
```

### 第一版

可以先用：

```text
NoopEncryptionService
```

意思是不加密，只保持接口存在。

### 未来

```text
WebCryptoEncryptionService
PasswordBasedEncryptionService
```

### 规则

- StorageAdapter 只负责保存字符串或结构化数据。
- EncryptionService 负责加密和解密。
- 不要让页面自己处理密钥和加密细节。

## 入口 11：导入导出 ImportExportAdapter

未来可能需要：

- 从 CSV 导入交易。
- 从交易所导出记录导入。
- 导出 JSON 备份。
- 导出 CSV。

## 不要写死的东西

第一版开发时要避免这些写法。

### 1. 不要把资产写死

不要只写：

```text
BTC
ETH
ADA
```

应该来自 `assets` 配置。

### 2. 不要把 USD 写死在所有地方

第一版默认 USD 可以，但字段里要保留 `currency`。

### 3. 不要把价格来源写死为手动

第一版手动输入，但数据结构里保留 `source`。

### 4. 不要把浏览器存储写进页面组件

页面不能直接：

```ts
indexedDB.open(...)
localStorage.setItem(...)
localStorage.getItem(...)
```

应该走 Repository。

### 5. 不要把计算写进 React 组件

组件不要到处写：

```ts
quantity * price
```

应该交给 `PositionCalculator` 或 `DecimalMath`。

### 6. 不要把 K 线和净值曲线混在一起

K 线是资产价格。

净值曲线是我的组合价值。

这两个未来要分开。

### 7. 不要让 Agent 直接改数据

Agent 只能生成建议或草稿。

写入前必须经过用户确认和统一校验。

## 未来功能接入方式

未来新增功能时，尽量按这个流程：

```text
1. 先判断它属于哪个入口。
2. 定义接口或复用已有接口。
3. 新增一个实现类或函数。
4. 接到 Service。
5. 最后让页面调用 Service。
```

例如接入 CoinGecko：

```text
1. 属于价格来源入口。
2. 使用 PriceProvider。
3. 新增 CoinGeckoPriceProvider。
4. PriceService 增加 provider 选择。
5. 页面仍然调用 PriceService。
```

例如接入自然语言输入：

```text
1. 属于交易输入入口。
2. 使用 TradeInputSource 和 NaturalLanguageTradeParser。
3. 新增 NaturalLanguageTradeSource。
4. 输出 TradeDraft。
5. 继续走 TradeValidator 和 TradeService。
```

## 完成标准

这份未来入口设计完成后，应该能回答：

- 未来实时价格 API 接在哪里？
- 未来 K 线接在哪里？
- 未来自然语言输入接在哪里？
- 未来 Agent 问答通过什么服务读数据？
- 未来加密保存放在哪一层？
- 页面为什么不能直接保存数据？
- 新增功能时如何尽量不改旧代码？

当前答案：

- 实时价格 API 接到 `PriceProvider`。
- K 线接到 `CandleProvider`，不要混进 `PriceSnapshot`。
- 自然语言输入接到 `NaturalLanguageTradeParser` 和 `TradeInputSource`。
- Agent 通过 `AgentQueryService` 读数据，默认只读。
- 加密放到 `EncryptionService`，保存层通过适配器组合使用。
- 页面只负责 UI，保存必须走 `Service -> Repository -> StorageAdapter`。
- 新功能优先新增 provider、adapter、parser、calculator 或 report generator，而不是重写页面。

## 当前结论

第一版可以很小，但边界要干净。

未来功能应该像积木一样接入：

```text
价格是一块积木。
K 线是一块积木。
自然语言输入是一块积木。
Agent 问答是一块积木。
报告是一块积木。
加密保存是一块积木。
```

只要第一版守住 SRP、OCP、LSP 和 DIP，后面扩展时就不会每次都推倒重来。
