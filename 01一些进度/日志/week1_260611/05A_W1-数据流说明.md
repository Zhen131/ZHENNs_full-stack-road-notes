# Day 5 数据流说明

日期：2026-06-15

状态：设计说明（不写实现代码）

目标：画清楚一笔数据从页面到本地保存，到底经过哪些层；让以后换存储方式或补加密时，不需要重写页面。

---

## 一句话结论

页面不直接碰本地存储，也不直接碰加密。

数据必须沿着一条固定的链路走，每一层只做自己那一件事：

```text
页面表单
  -> Service（校验 + 组织业务）
    -> Repository（账本总管：保存 / 读取 / 清空 / 导出 / 导入）
      -> StorageAdapter（存取工：屏蔽 IndexedDB 细节）
        -> EncryptionService（加密工：第一版 Noop，未来 AES）
          -> 浏览器存储（IndexedDB）
```

这条链路就是后面两份文档（保存层设计、加密路线）的骨架。

---

## 为什么要先画这条链路

第一版页面很简单：资产汇总、新增交易、交易列表、手动价格输入。

但如果把"读写本地存储、校验、计算、加密"都塞进页面组件里，未来每加一个功能都要拆页面，最后页面变成一个谁都不敢动的大文件。

所以第一版虽然功能少，但**层与层之间的边界今天就要定死**。这正是 Day3 扩展接口预留设计里反复强调的 SRP / OCP / LSP / DIP 四个原则的落地。

---

## 每一层负责什么（讲人话版）

| 层 | 角色比喻 | 只做这件事 | 绝对不做 |
| --- | --- | --- | --- |
| 页面 / 组件 | 柜台前台 | 展示数据、收集用户输入 | 不校验、不计算、不保存、不加密 |
| Service | 业务经理 | 校验输入、组织流程、调用下层 | 不直接碰存储、不直接碰加密 |
| Repository | 账本总管 | 提供保存 / 读取 / 清空 / 导出 / 导入等业务动作 | 不关心底层 IndexedDB 怎么打开、怎么建库、怎么事务提交 |
| StorageAdapter | 存取工 | 把一整坨数据读出来 / 写回去 | 不关心数据内容是什么业务 |
| EncryptionService | 加密工 | 写入前锁上、读出后解开 | 不关心数据从哪来、要存到哪 |
| 浏览器存储 | 仓库 | 真正存字节 | 什么都不懂 |

核心规则：**上面的层只知道下面紧挨着的那一层，不知道更底下发生了什么。** 页面不知道有没有加密，Service 不知道 IndexedDB 里有几个 object store、key 怎么设计。

---

## 三条主要数据流

### 流程一：新增一笔交易（写）

```text
TradeForm（页面填写：买/卖、资产、数量、价格、日期…）
  -> ManualTradeFormSource：把表单字段整理成统一的 TradeDraft
  -> TradeValidator：校验（数量>0、价格>0、卖出不超持仓、资产存在…）
  -> TradeService：校验通过后组织保存
  -> LedgerRepository.saveTrade(trade)
  -> StorageAdapter.write(ledgerData)
  -> EncryptionService.encrypt(...)（第一版 Noop，原样返回）
  -> 浏览器存储写入
  -> PositionCalculator 重新计算，页面刷新显示最新持仓
```

要点：页面只把"用户填了什么"交出去，后面校验、保存、加密、重算全是下层的事。校验失败会在 TradeValidator 这一层就被拦下，给页面返回错误信息，不会污染已保存的数据。

### 流程二：输入一条价格（写）

```text
PriceInputForm（页面填写：资产、价格、时间）
  -> PriceSnapshotService（组织成 PriceSnapshot，source = "manual"）
  -> LedgerRepository.savePriceSnapshot(snapshot)
  -> StorageAdapter.write(ledgerData)
  -> EncryptionService.encrypt(...)（第一版 Noop）
  -> 浏览器存储写入
  -> PositionCalculator 更新未实现盈亏，页面刷新
```

### 流程三：打开页面看资产汇总（读）

```text
Dashboard
  -> PositionService.getPositions()
  -> LedgerRepository.getLedgerData()
  -> StorageAdapter.read()
  -> EncryptionService.decrypt(...)（第一版 Noop，原样返回）
  -> 拿到明文 LedgerData（assets / trades / priceSnapshots / feeRules）
  -> PositionCalculator 从交易和价格算出 Position[]
  -> PositionSummaryTable 展示
```

读和写是同一条链路反着走：写的时候最后一道门加密，读的时候第一道门解密。中间所有层拿到的永远是明文对象。

---

## 导出 / 导入也要走这条链路

导出和导入不能绕过 Repository 直接读写存储。

```text
导出备份：
  Dashboard -> LedgerRepository.exportData() -> 得到明文 JSON -> 用户下载

导入备份：
  用户上传 JSON
  -> ImportAdapter 解析成 TradeDraft[] / LedgerData
  -> 每条交易仍然走 TradeValidator 校验
  -> LedgerRepository 保存
  -> StorageAdapter -> EncryptionService -> 浏览器存储
```

要点：

- **导出的 JSON 是明文**（这是 Day5 拍板的"导出明文备份"逃生口，详见加密路线说明）。它是用户忘记密码时的救命稻草，所以不能加密。
- 导入的数据**不能直接信任**，每一条仍然要过校验，避免坏数据污染账本。

---

## 这条链路守住的底线

1. **页面永远不写** `indexedDB.open`、`localStorage.setItem` / `getItem`。只调用 Service。
2. **计算不写在页面里**。`quantity * price` 这种永远交给 Calculator。
3. **加密只发生在最底层一道门**（StorageAdapter ↔ 浏览器存储之间），不散落在各处。
4. **IndexedDB 细节只放在 StorageAdapter**，上层不动。
5. **补加密只换 EncryptionService**，上层不动。

只要这五条守住，Day5 的完成标准就达成了：

- 能指出数据从页面到本地保存经过哪些层（就是上面那条链路）。
- 以后换存储方式或补加密，不需要重写页面。
