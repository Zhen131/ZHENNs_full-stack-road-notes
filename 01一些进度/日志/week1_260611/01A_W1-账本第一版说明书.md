# 账本第一版说明书

创建时间：2026-06-12

状态：第一版边界草稿

---

## 一句话说明

账本第一版是一个本地优先的个人投资交易账本，用来手动记录买入和卖出交易，并根据原始交易计算持仓、均价和基础盈亏。

第一版的重点不是做一个完整金融平台，而是先把最核心的交易记录链路做稳。

---

## 项目名称

暂定名称：

**Local-First Trading Ledger**

中文解释：

**本地优先的个人投资交易账本**

---

## 第一版解决什么问题

我现在的投资记录主要依赖手动文字记录，例如：

```text
以均价 67121.7 买入 BTC 0.00016388 个 价值 11$ 26/04/02
```

这种方式的优点是记录速度快、非常接近日常语言。

但问题是：

- 很难自动统计当前持仓。
- 很难自动计算平均成本。
- 很难自动计算盈亏。
- 很难检查记录是否完整。
- 很难为后续图表、语音输入、Agent 分析提供稳定数据。

所以第一版要做的事情是：

> 把自然语言式的交易记录，整理成结构化、可计算、可保存的交易账本。

---

## 第一版核心动作

第一版的核心动作是：

**记录交易 + 基础持仓盈亏**

其中优先级最高的是：

**记录交易**

因为交易记录是整个系统的原始数据。后续的持仓、均价、盈亏、图表、策略分析、自然语言输入和 Agent 问答，都必须建立在可靠的交易记录之上。

---

## 第一版用户流程

第一版用户打开页面后，应该能完成以下流程：

1. 添加一笔交易。
2. 填写日期、交易类型、资产、数量、均价、总金额。
3. 保存交易。
4. 在交易列表中看到这笔记录。
5. 手动输入某个资产的当前价格。
6. 在资产汇总中看到当前持仓、平均成本、市值和基础盈亏。

---

## 第一版必须包含的功能

### 1. 手动添加交易

支持手动记录：

- 买入
- 卖出

第一版暂时不处理复杂交易类型，例如转仓、空投、质押收益、借贷、合约和跨账户划转。

### 2. 交易列表

交易列表需要展示：

- 日期
- 交易类型
- 资产代码
- 数量
- 均价
- 总金额
- 手续费
- 备注

交易列表的作用是让用户确认每一笔原始交易是否记录正确。

### 3. 资产汇总

资产汇总需要展示：

- 资产代码
- 当前持仓数量
- 平均成本
- 当前价格
- 当前市值
- 已实现盈亏
- 未实现盈亏

资产汇总不是手动填写的数据，而是从交易记录和价格记录计算出来的结果。

### 4. 手动价格输入

第一版不接实时价格 API。

用户需要手动输入当前价格，例如：

```text
BTC 当前价格：70000 USD
ETH 当前价格：2500 USD
ADA 当前价格：0.25 USD
```

系统根据手动价格计算当前市值和未实现盈亏。

### 5. 本地保存

第一版的数据必须保存在用户本地。

第一阶段可以先用简单的浏览器本地存储实现，但代码结构上必须预留 Repository 和 Storage Adapter，避免页面直接操作底层存储。

未来可以升级为 IndexedDB 和加密存储。

## 第一版交易数据字段

第一版交易记录建议包含以下字段：

```text
id
occurredAt
timePrecision
type
asset
quantity
price
totalValue
currency
fee
note
rawText
createdAt
updatedAt
```

字段说明：

| 字段 | 含义 | 第一版要求 |
| --- | --- | --- |
| `id` | 交易唯一 ID | 必填 |
| `occurredAt` | 交易发生时间 | 必填 |
| `timePrecision` | 时间精度 | 必填，第一版多为 `day` |
| `type` | 交易类型 | 必填，`buy` 或 `sell` |
| `asset` | 资产代码 | 必填，例如 `BTC` |
| `quantity` | 买入或卖出数量 | 必填，必须大于 0 |
| `price` | 成交均价 | 必填，必须大于 0 |
| `totalValue` | 交易总金额 | 必填 |
| `currency` | 计价货币 | 第一版默认 `USD` |
| `fee` | 手续费 | 可选，默认为 0 |
| `note` | 用户备注 | 可选 |
| `rawText` | 原始输入文本 | 可选，但建议保留 |
| `createdAt` | 记录创建时间 | 系统生成 |
| `updatedAt` | 记录更新时间 | 系统生成 |

## 时间戳设计

当前交易记录通常只精确到日期，例如：

```text
26/04/02
```

第一版可以将其结构化为：

```text
occurredAt: "2026-04-02"
timePrecision: "day"
```

未来如果用户通过语音或自然语言输入更精确的时间，例如：

```text
2026年4月2日下午3点30分买入 BTC
```

可以升级为：

```text
occurredAt: "2026-04-02T15:30:00"
timePrecision: "minute"
```

这样第一版不需要实现复杂的时间解析，但数据结构不会挡住未来扩展。

## 参考 Frappe Books 的地方

Frappe Books 是完整会计软件，第一版账本不照搬它的业务规模，只参考它的结构思想。

第一版主要参考：

- Schema：先定义数据长什么样。
- Model / Service：把校验和计算放在业务层，不放在页面里。
- Repository：页面不直接操作本地存储。
- Report：资产汇总和盈亏结果从原始交易计算出来。
- UI 信息架构：页面分为汇总、表单、列表和报表区域。

第一版不参考：

- 发票
- 税务
- POS
- 会计科目表
- 复式记账
- 多语言
- 企业级权限系统

---

## 第一版不做什么

为了保证第一版可以完成，以下内容明确不做：

- 不做语音输入。
- 不做自然语言自动解析。
- 不接实时价格 API。
- 不做 K 线图。
- 不做 EMA、MACD、RSI、BOLL 等技术指标。
- 不做 DCA 回测。
- 不做策略建议。
- 不做 Agent 问答。
- 不做插件系统。
- 不做多端同步。
- 不做登录系统。
- 不做 Tauri 桌面端。
- 不做 Capacitor 移动端。
- 不做完整会计系统。

这些内容可以作为未来方向，但不能进入第一版开发范围。

## 第一版和毕业论文的关系

毕业论文项目方案强调：

- local-first
- browser-based
- encrypted storage
- transaction recording
- DCA cost basis
- holdings calculation
- realized / unrealized P&L
- import / export
- benchmark evaluation

第一版说明书只覆盖论文项目的最小地基：

- 交易记录
- 持仓计算
- 平均成本
- 基础盈亏
- 本地保存结构

加密、IndexedDB、导入导出、benchmark 和图表可以在后续阶段逐步加入。

---

## 第一版完成标准

第一版完成后，应该能做到：

- 用户能手动添加一笔买入或卖出交易。
- 用户能看到完整交易列表。
- 系统能根据交易计算每个资产的当前持仓。
- 系统能计算每个资产的平均成本。
- 用户能手动输入当前价格。
- 系统能计算当前市值和未实现盈亏。
- 系统能阻止明显错误的交易，例如数量小于等于 0、价格小于等于 0、卖出数量超过当前持仓。
- 数据保存在本地，不上传服务器。

## 3 分钟解释版本

Local-First Trading Ledger 第一版是一个本地优先的个人投资账本。它的核心不是预测市场，也不是做复杂策略，而是帮我把每一笔买入和卖出交易记录成结构化数据。

用户可以手动输入交易日期、资产、数量、均价和总金额。系统会保存这些原始交易，并从交易记录计算当前持仓、平均成本和基础盈亏。第一版不接实时价格 API，所以当前价格由用户手动输入。

这个版本的重点是把账本地基做稳：交易记录要清楚，数据结构要稳定，计算结果要可验证，保存层要直接面向 IndexedDB、加密、导入导出和论文 benchmark 留出空间。语音输入、AI Agent、图表、策略建议和插件系统都属于未来扩展，不进入第一版。
