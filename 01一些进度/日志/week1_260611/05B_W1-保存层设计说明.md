# Day 5 保存层设计说明

日期：2026-06-15

状态：设计说明（不写实现代码）

目标：定义保存层的两个核心角色 `LedgerRepository` 和 `StorageAdapter`，让页面和业务逻辑永远不直接碰本地存储；第一版正式实现直接使用 IndexedDB，同时保留以后替换存储实现时只改一个零件、不动上层的能力。

前置阅读：《05A_W1 数据流说明》。

---

## 保存层为什么要拆成两个零件

很多人会把"存数据"写成一个大函数，直接在里面又拼业务、又调浏览器存储 API。这样一来，换存储方式要改一大片，加密也无处下手。

第一版我们把保存层拆成上下两层，各管一件事：

- **LedgerRepository（账本总管）**：对上，提供"业务听得懂的动作"——保存交易、读取交易、清空、导出、导入。它不关心数据最终存到哪。
- **StorageAdapter（存取工）**：对下，只管"把一整坨数据读出来 / 写回去"。它不关心这坨数据是交易还是价格。

一句话：**Repository 懂业务不懂存储，StorageAdapter 懂存储不懂业务。** 中间用接口对话，互不干扰。

---

## 它们存的是什么：LedgerData

整个账本的本地数据是一个对象（沿用 Day2 数据结构草稿）：

```ts
type LedgerData = {
  schemaVersion: 1;
  assets: Asset[];
  trades: Trade[];
  priceSnapshots: PriceSnapshot[];
  feeRules: FeeRule[];
};
```

第一版 StorageAdapter 的最小做法：直接用 IndexedDB 保存整个 `LedgerData`，但先按一坨整体读写（whole-blob）。不做"按表分片""按字段加密"这种复杂事——那是后续性能优化和 Week6 加密阶段的活，不是 Week4 第一版持久化的活。

> 为什么在 IndexedDB 里先整坨读写：这样既能满足论文路线里"直接上 IndexedDB"的要求，又能把 Week4 的复杂度压住。等数据量大了（成千上万条），再在 StorageAdapter 内部升级成按 store 分片、索引或缓存，上层一行都不用改。

---

## StorageAdapter（存取工）

它是保存层最底下的一道门，只有两个动作：

```ts
interface StorageAdapter {
  read(): Promise<LedgerData>;
  write(data: LedgerData): Promise<void>;
}
```

- `read()`：从浏览器存储里把整个账本读出来，返回明文 `LedgerData`。
- `write(data)`：把整个账本写回浏览器存储。

### 第一版实现

```text
IndexedDBStorageAdapter
```

它内部做的事：打开浏览器 IndexedDB 数据库，在固定 object store 中保存一份 `LedgerData`。Week4 第一版可以只保存一个固定记录（例如 key = `ledger:v1`），读的时候取出整份账本；如果读不到数据，返回默认空账本。

> 第一版有没有加密，StorageAdapter 不自己决定。它把"要不要锁、怎么锁"交给 EncryptionService（见《05C_W1 加密路线说明》）。第一版的加密工是 Noop，所以等于明文存 IndexedDB——但接口已经留好，Week6 换真加密时 StorageAdapter 只改"调用加密工"这一小段。

### 未来可替换或增强的实现（今天不做，只留位）

```text
IndexedDBStorageAdapter          // Week4 正式第一版，whole-blob 读写
EncryptedIndexedDBStorageAdapter // 组合了真加密的版本
FileStorageAdapter               // 未来 Tauri 桌面端直接读写文件
```

它们都实现同一个 `StorageAdapter` 接口，所以可以互相替换——这就是 Day3 文档里说的 LSP（里氏替换原则）。

### StorageAdapter 的规矩

- 不关心数据内容的业务含义（它只看到"一坨要存的东西"）。
- 不在自己内部做业务校验（那是 Service 的事）。
- 读不到数据时（第一次打开、被清空），返回一个空的默认 `LedgerData`，不能直接崩。
- 所有实现的输入输出必须一致（都吃 `LedgerData`、都吐 `LedgerData`），不能某个实现偷偷要求多一个参数。

## LedgerRepository（账本总管）

它对上层（Service）提供业务动作。Day5 清单要求覆盖：保存、读取、清空、导出、导入。

```ts
interface LedgerRepository {
  // 读
  getLedgerData(): Promise<LedgerData>;
  listTrades(): Promise<Trade[]>;
  listPriceSnapshots(): Promise<PriceSnapshot[]>;

  // 写
  saveTrade(trade: Trade): Promise<void>;
  savePriceSnapshot(snapshot: PriceSnapshot): Promise<void>;

  // 清空
  clearAll(): Promise<void>;

  // 备份
  exportData(): Promise<string>;          // 导出明文 JSON 字符串
  importData(json: string): Promise<void>; // 导入并覆盖（导入前应已通过校验）
}
```

### 它内部怎么干活

Repository 自己不碰 IndexedDB API，它手里握着一个 `StorageAdapter`，所有动作都翻译成 `read` / `write`：

```text
saveTrade(trade):
  1. data = adapter.read()        // 读出当前整个账本
  2. data.trades.push(trade)      // 在内存里加上这笔
  3. adapter.write(data)          // 整坨写回

listTrades():
  1. data = adapter.read()
  2. return data.trades

clearAll():
  1. adapter.write(空的默认 LedgerData)

exportData():
  1. data = adapter.read()
  2. return JSON.stringify(data)  // 明文，给用户下载当备份
```

> 注意：第一版"整坨读出→改→整坨写回"完全够用。等数据量大了，可以在 Repository / Adapter 内部改成增量写，对上层接口不变。

### LedgerRepository 的规矩

- 它是上层进入存储的**唯一入口**。页面和 Service 不许绕过它直接碰 IndexedDB、`localStorage` 或任何浏览器存储 API。
- 它只组织"存取动作"，不做交易校验（校验在 Service / TradeValidator）。
- 它不做持仓/盈亏计算（那是 PositionCalculator）。
- `exportData` 导出的是**明文**，这是用户忘记密码时的逃生口，详见《05C_W1》。
- `importData` 假设传进来的数据已经过校验；不可信来源（用户上传的文件）必须先在 Service 层过一遍 TradeValidator 再调它。

## 依赖方向（谁拿着谁）

```text
Service
  └─ 拿着 → LedgerRepository（抽象接口）
                └─ 拿着 → StorageAdapter（抽象接口）
                              └─ 拿着 → EncryptionService（抽象接口）
```

关键：上层拿的都是**接口**，不是具体实现。具体用哪个实现（IndexedDBStorageAdapter？未来 FileStorageAdapter？Noop 加密？AES 加密？）在项目启动时的一个"组装点"（工厂函数 / 配置）里拼好，再传下去。

这就是 DIP（依赖倒置）：换实现只改组装点那一行，业务代码完全不知情。

举例——第一版组装：

```text
const encryption = new NoopEncryptionService();
const adapter    = new IndexedDBStorageAdapter(encryption);
const repository = new LedgerRepository(adapter);
// Service 拿到 repository 就能干活
```

未来某天要上真加密，只改加密实现：

```text
const encryption = new WebCryptoEncryptionService(userPassphrase);
const adapter    = new IndexedDBStorageAdapter(encryption);
const repository = new LedgerRepository(adapter);
```

上面所有页面、Service、计算逻辑一行都不用动。**这就是 Day5 完成标准"以后换存储方式或补加密，不需要重写页面"的具体兑现方式。**

## 完成自检（保存层部分）

这份文档完成后应该能回答：

- 页面为什么不能直接 `indexedDB.open` 或 `localStorage.setItem`？→ 因为存储只能从 LedgerRepository 进，Repository 再走 StorageAdapter。
- 换存储方式要改哪里？→ 只换 StorageAdapter 实现 + 组装点那一行。
- 补加密要改哪里？→ 只换 EncryptionService 实现 + 组装点那一行（详见 05C_W1）。
- 导出备份是明文还是密文？→ 明文，作为忘记密码的逃生口。
