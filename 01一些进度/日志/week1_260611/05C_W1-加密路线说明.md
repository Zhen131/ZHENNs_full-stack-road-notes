## EncryptionService 在哪一层

它是保存层最底下、紧贴浏览器存储的一道工序（见 05A 链路）：

```text
... -> StorageAdapter（存取工）
          -> EncryptionService（加密工）
            -> 浏览器存储
```

- **写**的时候：StorageAdapter 要往存储里写之前，先把数据交给加密工锁上。
- **读**的时候：StorageAdapter 从存储里拿到的是锁着的，先交给加密工解开，再往上交明文。

关键：**只有这一层碰加密。** 页面、Service、Repository 拿到的永远是明文对象，它们根本不知道有没有加密。这样未来开关加密，上层零改动。

## 接口长什么样

```ts
interface EncryptionService {
  encrypt(plainText: string): Promise<string>;  // 明文 -> 密文（或原样）
  decrypt(cipherText: string): Promise<string>; // 密文 -> 明文（或原样）
  isEnabled(): boolean;                          // 当前是否真的在加密
}
```

输入输出都是字符串（StorageAdapter 已经把 `LedgerData` 序列化成字符串了）。

> 设计要点：不管哪个实现，`encrypt` 后再 `decrypt`，必须**一字不差**拿回原文（byte-for-byte round-trip）。这是论文 Month 3 要专门验证的"往返完整性"。Noop 版本天然满足；AES 版本要靠测试保证。

## 第一版：NoopEncryptionService（占位，不加密）

```text
NoopEncryptionService：
  encrypt(x)  -> 原样返回 x
  decrypt(x)  -> 原样返回 x
  isEnabled() -> false
```

它的唯一作用：**让接口存在、让链路跑通**。第一版用它，等于明文存 IndexedDB，但代码结构已经为真加密留好了位置。

## 未来：真加密走什么技术路线

毕业论文方案已经把技术栈定死，我们照搬，不另起炉灶：

- **加密算法**：AES-256-GCM（Web Crypto API 原生支持，浏览器自带，不用装库）。GCM 模式的好处是加密的同时带"防篡改校验"——数据被人改过，解密会直接失败，而不是悄悄给你一份错数据。
- **钥匙怎么来**：用户输入一个密码（passphrase）→ 用 **PBKDF2** 把密码"熬"成一把 256 位的钥匙。PBKDF2 会故意算得很慢、加随机盐（salt），让别人没法靠字典暴力猜密码。
- **实现类**：

```text
WebCryptoEncryptionService / PasswordBasedEncryptionService：
  - 启动时：用户输密码 -> PBKDF2 派生密钥（带 salt + 迭代次数）
  - encrypt：AES-256-GCM 加密，输出 = salt + iv + 密文 一起打包
  - decrypt：拆出 salt + iv，重新派生密钥，解密；密码错或数据被改 -> 抛统一错误
  - isEnabled() -> true
```

换上它时，StorageAdapter 和上层全都不动——这正是第一版留接口的意义。

## 错误处理（必须现在就想清楚）

真加密上线后，会冒出第一版没有的新错误，文档里先把规矩定好：

| 情况 | 加密工怎么反应 | 给用户看什么 |
| --- | --- | --- |
| 密码输错 | decrypt 失败，抛统一的"解锁失败"错误 | "密码不对，请重试" |
| 数据被篡改/损坏 | GCM 校验失败，同样抛"解锁失败" | "数据无法读取，可能已损坏，请用备份恢复" |
| 浏览器不支持 Web Crypto | 启动时检测，降级或提示 | "当前环境不支持加密" |

规矩：**加密工只抛统一的失败结果，不把底层密码学细节漏给上层。** 上层只需要知道"成功 / 失败"，不需要知道是 iv 错了还是 tag 校验没过。

## ⚠️ 最大的产品风险：忘记密码 = 数据可能永久丢失

这是今天最重要的一段，不是技术问题，是产品取舍。

本地优先 + 加密意味着：**没有服务器、没有"找回密码"邮件、没有客服能帮你重置。** 用户密码一忘，加密的账本就永远打不开了。这跟需求总纲风险9（数据丢了 = 项目失去意义）正面冲突。

### 拍板的方案：留"导出明文备份"逃生口

（Day5 已确认采用此方案）

- 允许用户**随时导出一份未加密的 JSON 备份**，自己存到安全的地方（U 盘、私人网盘、本地另一个文件夹）。
- 这份备份是明文，所以**忘了密码也能从它恢复**。
- 恢复流程：导入这份 JSON → 校验 → 重新设一个新密码 → 重新加密保存。

### 这个取舍的两面，文档要对用户讲清楚

- **好处**：用户永远有救命稻草，不会因为忘密码就血本无归。符合"本地优先"里"用户拥有自己的数据"这条理想。
- **代价**：明文备份文件本身不加密，**安全责任转移给用户**——他得自己保管好这个文件，别随便丢公共网盘、别发给别人。
- **产品上必须做的提示**：导出时弹一句明确警告："这份备份未加密，请妥善保管，不要上传到不信任的地方。"

### 和导出/导入链路的关系

这条逃生口正好解释了 05A 里为什么"导出的 JSON 是明文"：

```text
正常使用：账本在浏览器里是加密存的（未来）
导出备份：导出时解密 -> 输出明文 JSON -> 用户自己保管
导入恢复：明文 JSON -> 校验 -> 重新加密存回
```

加密保护的是"放在浏览器里的日常数据"；明文备份保护的是"用户别把自己锁死在门外"。两者分工不同，缺一不可。

## 升级路线总表（从今天到 Month 3）

| 阶段 | EncryptionService 实现 | 实际效果 | 上层要改吗 |
| --- | --- | --- | --- |
| 第一版（现在） | NoopEncryptionService | 明文存 IndexedDB，接口已就位 | — |
| Month 3 / Phase 4 | WebCryptoEncryptionService | AES-256-GCM + PBKDF2 真加密 | **不用改**，只换组装点一行 |

```text
// 第一版

// Month 3
const encryption = new WebCryptoEncryptionService(userPassphrase);
```

就这一行。页面、Service、Repository、StorageAdapter 的接口全都不动。
