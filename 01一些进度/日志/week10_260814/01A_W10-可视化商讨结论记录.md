## 持仓市值饼图

- 饼图按当前市值计算，不按累计买入金额或持仓成本计算。
- 只统计非零持仓。
- 多币种不得直接相加；饼图与折线图共用币种选择器，一次只看一个 quote currency。
- 有 USD 时默认选 USD；没有 USD 时稳定选择账本已有的第一个币种。
- 单资产时显示 100%。
- 没有匹配价格的资产不得以 `0` 市值进入饼图，必须单独列为“未估值资产”。
- 只有部分资产可估值时，图中表达的是“已估值资产内部占比”，旁边同时说明哪些资产尚未进入占比。
- 全部非零持仓都缺价格时显示缺价状态，不画误导性空饼图。
- 交易或价格快照变化后，饼图从当前 `LedgerData` 重新派生并立即更新。

## 持仓市值与持仓成本曲线

### 名称含义

当前模型没有现金账户、充值和提现，因此这张图不能声称是完整账户净值：

- “持仓总市值”只表示当前仍持有资产按可证明价格计算的总价值。
- “持仓成本”是当前剩余持仓的 cost basis；部分卖出后会下降。
- 持仓成本不是历史累计投入金额。
- 卖出所得现金没有进入当前数据模型，图表必须标明“不含未记录现金”。

图表标题可以继续使用易懂的“资产增长曲线”，但图例和说明必须使用准确名称。

### 时间范围

折线图提供五个真实可用的范围按钮：

```text
1日 / 7日 / 30日 / 365日 / 全部
```

这些按钮控制显示范围，不是假导航。

- 7 日、30 日和 365 日使用连续日历日期。
- “全部”从账本第一笔相关事实开始显示到今天。
- 1 日范围没有可靠日内变化时，在区间起点和终点显示相同值，形成水平线。
- 本轮不增加滚轮、拖拽缩放或底部 dataZoom 滑块。

### 线条表达

- 使用阶梯线，不使用平滑曲线。
- 数值保持水平，直到新的交易或价格事实使它发生跳变。
- 没有价格变化且持仓未变化时，线保持水平，表现为用户所说的“心电图停跳”。
- 不能用斜线或平滑曲线制造两个事实点之间不存在的连续波动。
- 持仓总市值与持仓成本不堆叠，分别显示。

## 价格事实与准确性

市值估值只接受 `PriceSnapshot`：

- 当前手动录入的价格快照是合法来源。
- 未来 API 写入且标记为 `source: "api"` 的快照也是合法来源。
- 交易成交价不自动充当市场价格快照。
- 持仓成本不充当市场价格。
- 缺价格不显示为 `0`。

每个历史时间点只能读取该点及之前的价格快照，禁止未来信息泄漏。

已经存在真实价格快照、后来暂时离线或没有新价格时，可以沿用最后一次已验证价格：

- 图中必须显示价格截至时间。
- 沿用表示“按最后已知价格估值”，不表示实时行情。
- 新价格到来后重新派生图表。

某资产在目标时间点之前从未有合法价格快照时：

- 不使用最近成交价补齐。
- 不使用今天的价格倒灌昨天。
- 不要求图表猜测。
- 该资产必须等待用户手动补价，或等待未来联网后由行情 API 补充相应价格事实。

只要所选币种下任一非零持仓缺价，该时间点的“持仓总市值”就是不完整值：

- 不把其余已估值资产相加后冒充完整总市值。
- 市值线在该点保持缺失并断开。
- 持仓成本线仍可继续显示。
- 提示信息列出缺价资产。

图表不保存自己的历史结果；价格补齐、更正、交易新增或交易删除后，统一从账本事实重新派生。

## 交易活跃热力图

### 日历范围与统计

- 显示以今天为终点的最近 365 个自然日。
- 每个格子代表一个日期。
- 强度只按交易笔数计算，不按交易金额计算。
- 买入和卖出都计入总笔数。
- 未来日期的交易不进入最近 365 天统计。
- 日历结构、月份标签和亮度表达参照 GitHub contribution graph。

### 五档亮度

总共五种状态：

```text
level 0：0 笔交易
level 1-4：非零交易日的四个相对活跃等级
```

非零日期按照最近 365 天内的交易笔数做相对百分位排名：

- 由低到高对应四个非零亮度。
- 同样笔数必须得到相同等级。
- 当前最大笔数始终为 level 4。
- 最近 365 天只有一个非零交易日时，该日直接为 level 4。
- 新增或删除交易后，整个 365 天范围立即重新计算等级。

因此，只有 1 笔交易时该日可以最亮；以后出现 5 笔、30 笔等更活跃日期，旧格子的相对亮度会随整体分布变化。这种变化是实时派生结果，不写入 `LedgerData`。

### 日期筛选

- 悬停格子显示日期、总笔数、买入笔数和卖出笔数。
- 点击某一天后，下方交易列表只显示该日的全部交易。
- 交易列表标题显示所选日期。
- 页面提供明确的“清除日期筛选”入口。
- 再次点击同一天可以取消筛选。
- 新增或删除所选日期的交易时，格子、提示信息和过滤后的交易列表同步更新。
- 日期筛选属于页面临时状态，不写入账本或 IndexedDB。

## 图表技术选型

图表库正式选择 **Apache ECharts**，不再沿用旧文档中的 Recharts 候选。

选择原因：

- 后续明确可能出现 K 线、拖拽缩放和更多专业行情交互。
- 项目未来会面对 5 万、10 万甚至更多交易数据。
- ECharts 的 Canvas 渲染、渐进渲染和数据采样更符合长期方向。
- 当前优先级是产品完成度和后续扩展，不把“学习 React 图表组件组合”作为本轮选型目标。

使用方式已经确定：

- 通过 npm 安装并锁定 ECharts。
- 从 `echarts/core` 按需注册折线、饼图、日历热力图及必要组件，不导入无关图表。
- 三张图统一使用 Canvas renderer。
- 建立统一 React 适配层，集中处理 `init`、`setOption`、容器 resize、事件绑定/解绑和 `dispose`。
- Dashboard 不直接维护散落的 ECharts 实例。
- ECharts option 只负责展示，不能承担持仓、价格选择、缺价判断或热力等级等业务计算。
- 图表数据服务保持纯 TypeScript；以后即使更换渲染库，也不重写账本计算口径。
- 本轮关闭图表动画，视觉动画留到后续美化阶段。

## 大数据处理想法

选择 ECharts 不等于把 10 万条原始交易直接交给 Canvas。

历史曲线的数据处理方向确定为：

```text
交易与价格各排序一次
-> 单次向前重放持仓
-> 按日生成聚合点
-> ECharts 只绘制聚合后的日级序列
```

- 禁止每生成一个日期点就重新扫描全部交易。
- 禁止在 React 组件内复制 DCA、卖出成本或价格选择公式。
- 当前持仓和历史曲线应复用同一套持仓重放规则。
- Decimal 字符串继续作为业务精确值；只在 ECharts 坐标渲染边界转换为有限 JavaScript number。
- 提示框和明细继续显示原始精确字符串。
- 长时间范围可以使用 ECharts 折线采样，但采样只影响绘制，不改变账本事实和提示中的精确值。
- 10 万笔交易的正式性能指标和计时仍属于后续 benchmark；Week 10 只保证数据服务结构不采用明显的重复全量扫描。
