## P0：production 安全 Gate 缺少完整实证

### 专业问题定义

Week 9 的 production 安全验收证据不完整，当前不能证明最终 Go：

- 尚未在 DevTools Application / IndexedDB 中直接读取 `ledger:v1`，确认 record 为 `StoredLedgerEnvelopeV2`。
- 尚未对 production 原始 record 执行明文特征搜索，无法形成“IndexedDB 不含完整账本明文”的现场证据。
- 尚未在 production 中分别篡改 ciphertext 与 IV，验证认证解密失败、零写入和原密文不变。
- 尚未在保持 `localhost` 可访问的条件下完成硬离线打开、解锁、查询与导出。

自动化测试已覆盖真实 Web Crypto 加解密、错误密码和篡改拒绝，但自动化证据不能替代执行清单要求的 production 存储现场与硬离线证据。

影响：Week 9 Day 6 安全 Gate 未关闭，当前版本不能判定为 Week 9 Go，也不能进入合并。

### 通俗解释

现在代码测试已经证明“理论上会加密，也能识别被改坏的密文”，但我们还没有亲眼检查浏览器真正保存下来的那条数据。

因此目前还不能完全确认：

- 浏览器里存的是不是只有密文；
- 别人手动改密文后，网页是不是一定拒绝；
- 没有外网时，账本是不是仍然能正常使用。

这不是说加密已经失败，而是最后一组现场证据还没拿到。

## P1-01：首次设密写入成功但回读失败时形成重试死路

### 专业问题定义

`DefaultLedgerAccessController.setup()` 的执行顺序为：

```text
确认空库
-> 保存加密初始账本
-> 回读验证
-> 返回 Repository
```

如果 IndexedDB 写事务已经成功提交，但随后的回读因临时故障失败：

- `setup()` 返回 `LEDGER_SETUP_FAILED`；
- UI 保持在首次设置页并提示“请重试”；
- 用户直接重试时，控制器发现 V2 record 已存在，再次返回 `LEDGER_SETUP_FAILED`；
- 页面不会自动转入解锁路径，必须刷新后才能恢复。

现有测试覆盖“写入直接失败”，但未覆盖“写入成功、验证回读失败、用户再次提交”的状态转换。

影响：不会覆盖或丢失已经写入的密文，但首次设置页面可能陷入无法通过自身按钮恢复的状态。

### 通俗解释

可以把首次设密理解成：

1. 先把上锁后的账本放进柜子；
2. 再打开柜子检查一次；
3. 检查成功才告诉你完成。

现在的问题是：柜子可能已经放成功了，但检查时临时出错。页面却只告诉你“再试一次”。你再点一次，它又发现柜子里已经有东西，于是继续报错。

数据还在，也没有被覆盖，但页面自己走不出去，只能刷新后重新解锁。

## P1-02：Dashboard clear 加密主链缺少 production 闭环证据

### 专业问题定义

当前 production 已验证解锁页的 `resetEncryptedLedger()`，但尚未完整验证 Dashboard 内 `usePersistentLedger.clearLedger()` 的加密后主链：

```text
已解锁账本
-> 导出明文备份
-> Dashboard 清空本地账本
-> record 删除
-> 通过当前加密 Repository 导入备份
-> 新 record 仍为 V2 密文
-> 刷新并正确解锁恢复
```

这两个入口不等价：

- 解锁页重置直接调用 AccessController，只删除固定 record。
- Dashboard clear 必须继续服从既有写队列、pending save、operation mutex、dirty state 和 Repository generation 保护。

自动化测试覆盖队列与 clear 语义，但缺少真实 production IndexedDB 上的完整闭环证据。

影响：Week 7/8 的 clear 与备份恢复契约尚未取得“加密接入后仍无回归”的 production 级证明。

### 通俗解释

现在验证过的是“忘记密码时把整个加密账本删掉”，但还没有完整验证账本页面里面原来的“清空本地账本”按钮。

这两个按钮看起来都在删除数据，实际走的是两套不同通道：

- 忘记密码清空比较直接；
- Dashboard 清空还要排队，防止旧的自动保存突然把已经删除的数据写回来。

测试代码说这套排队逻辑没坏，但真实浏览器里还没有从清空、导入到刷新恢复完整走一遍。

### 通俗解释

计划表上有 55 个小方框，现在一个都没有打勾，但文件顶部又写着“已经执行”。

这就像一张体检表写着“检查完成”，下面所有项目却都是空白。后来的人看不出来：

- 哪些真的检查过；
- 哪些只是测试代码通过；
- 哪些还需要你亲自在浏览器确认。

代码不一定有问题，但文档现在不能准确告诉我们 Week 9 到底关了多少门。
