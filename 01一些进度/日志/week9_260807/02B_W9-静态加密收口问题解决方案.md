## P1-01：首次设密回读失败后的可恢复状态

### 决策

密文一旦已经写入，页面不得继续要求用户“创建新账本”。控制器必须返回“密文已经存在，需要转入解锁”的可恢复结果，页面自动切到解锁页。

新增稳定错误码：

```ts
LEDGER_SETUP_RECOVERY_REQUIRED
```

它表示：首次设密未完成当前页面内的回读验证，但存储中已存在结构合法的 V2 密文。它不是写入失败，也不是数据损坏。

### AccessController 修改

修改 `src/composition/ledgerAccessController.ts`：

1. `setup()` 开始时若发现合法 V2 record，不再返回通用 `LEDGER_SETUP_FAILED`，改为 `LEDGER_SETUP_RECOVERY_REQUIRED`；不得再次写入。
2. `repository.save(...)` 后的 `repository.load()` 失败时，只执行一次存储状态核对：
   - 合法 V2：返回 `LEDGER_SETUP_RECOVERY_REQUIRED`。
   - `null`：返回 `LEDGER_SETUP_FAILED`。
   - 读取失败：返回 `LEDGER_ACCESS_READ_FAILED`。
   - 非法 V2：返回 `LEDGER_ACCESS_INVALID_ENVELOPE`。
   - 旧/未知格式：返回 `LEDGER_ACCESS_UNSUPPORTED_FORMAT`。
3. 任何失败分支都不得 clear、覆盖或二次保存。
4. 不得使用首次设置页中的 passphrase 自动绕过解锁验证。

### Access Gate 修改

修改 `src/components/security/LedgerAccessGate.tsx`：

- 收到 `LEDGER_SETUP_RECOVERY_REQUIRED` 后清空两次密码输入。
- 自动进入 `unlock-required`。
- 显示非破坏性提示：`加密账本已写入，但本次验证未完成。请重新输入密码解锁。`
- 不挂载 Dashboard，不显示“创建失败，请重试”。
- 读取失败仍进入现有 error / 重新检查路径。

### 必须新增的测试

1. 写入成功、首次验证 read 失败、核对 read 返回合法 V2：
   - 返回 `LEDGER_SETUP_RECOVERY_REQUIRED`；
   - `write` 只调用一次；
   - V2 record 保留。
2. 用户在旧 setup 页面再次提交，但存储中已有合法 V2：
   - 不写入；
   - 返回 recovery-required。
3. Access Gate 收到 recovery-required：
   - 自动显示解锁页与恢复提示；
   - 不挂载 Dashboard。
4. 使用原密码解锁成功，账本内容完整。
5. 核对 read 仍失败：
   - 返回 `LEDGER_ACCESS_READ_FAILED`；
   - 不删除、不覆盖。
6. 既有正常 setup、错密码、损坏 V2、reset 测试继续通过。

### 通俗解释

现在相当于“东西已经放进保险柜，但关门后的检查临时失败”。修复后页面不会让你继续创建第二个保险柜，而是直接告诉你：保险柜已经存在，请用刚才的密码重新打开确认。数据保留，也不需要刷新页面。

## P1-02：Dashboard 全账本 clear 与加密恢复闭环

### 产品边界

Dashboard 的“清空本地账本”会永久删除当前 origin 下的完整账本 record，包括自定义资产、交易、价格和手续费规则。

它不是：

- 删除某一行交易；
- 撤回上一步；
- 回收站或软删除。

单行交易删除走 `trade/delete` 安全校验；当前没有撤回功能。完整 clear 的恢复入口只有事先导出的明文备份。

### production 执行

1. 正确解锁，准备至少两笔交易和一条价格，等待“已保存到本地”。
2. 导出完整明文备份，确认页面继续显示明文风险提示。
3. 点击 Dashboard“清空本地账本”，输入固定文本 `清空本地账本`。
4. 等待“账本已清空”，直读 IndexedDB，确认 `ledger:v1` 不存在。
5. 不刷新页面，导入第 2 步备份。
6. 等待“已保存到本地”，直读 `ledger:v1`：
   - `formatVersion = 2`；
   - 原始 record 无账本明文特征。
7. 刷新页面，输入原密码，确认交易、价格、持仓完整恢复。

通过线：

- clear 前的 pending save 必须先结束，旧异步 save 不得在 clear 后写回。
- clear 成功后 record 不存在，页面为新的初始账本。
- 导入后只生成 V2 密文。
- 刷新解锁后数据与导出备份一致。
- 全流程无未处理应用错误。

若全部通过：只补验收记录，不修改源码。若任一项失败：停止收口，记录失败步骤、页面状态、record 前后值，再单独制定源码修复，不得现场扩大修改范围。
