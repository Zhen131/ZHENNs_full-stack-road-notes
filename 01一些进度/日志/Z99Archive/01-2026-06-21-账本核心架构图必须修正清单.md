## 总结论

当前 Canvas 的主方向是对的：`UI -> Draft -> Validator -> Service -> Repository -> StorageAdapter -> EncryptionService -> IndexedDB` 这条写入链路成立，`Trade / PriceSnapshot` 作为原始事实保存、`Position` 作为计算结果不保存的设计也成立。

但是这张图还不能当作最终运行协议图。下面 8 条属于必须修正或必须补充的内容，否则后面实现时容易出现数据污染、安全边界被绕过、论文范围不一致或性能评审被追问的问题。

## 必修清单

- [x] **1. 修正读取链路，避免绕过加密层**

  当前 Canvas 里存在 `IndexedDB -> read-repository` 这类表达，容易误解成 Repository 可以直接读 IndexedDB。正确读取链路应该是：

  ```text
  IndexedDB
    -> EncryptionService.decrypt
    -> StorageAdapter.read
    -> LedgerRepository.getLedgerData
    -> PositionService
    -> PositionCalculator
    -> DisplayFormatter / ViewModel
    -> UI
  ```

  修图要求：把读路径也画成和写路径同一套边界，不允许 Repository 或 Service 直接碰 IndexedDB。

  处理状态：已在 `000-账本核心架构与数据流图-横向泳道版.canvas` 中把读取路径改为 `IndexedDB -> EncryptedEnvelope -> EncryptionService.decrypt -> StorageAdapter.read -> schema validation / migration -> LedgerRepository.getLedgerData`。

- [x] **2. 补上导入 / 导出链路**

  导入导出不是普通附加功能，而是论文要求、备份恢复、忘记密码逃生口的核心链路。Canvas 目前只在 Repository 节点里写了 `exportData()` / `importData(data)`，但没有画出完整流程。

  需要补：

  ```text
  Export:
  UI -> LedgerRepository.exportData -> 明文 JSON -> 用户下载 -> 明文备份警告

  Import:
  用户上传 JSON
    -> ImportAdapter / JSON parser
    -> schema validation
    -> TradeValidator / PriceSnapshotValidator
    -> 用户确认覆盖
    -> Repository
    -> StorageAdapter
    -> EncryptionService
    -> IndexedDB
  ```

  修图要求：明确导出是明文 JSON；导入不能直接写库，必须先校验。

  处理状态：已在副本图中补 `Export 明文备份链路`、`Import 导入链路`、`明文备份警告`、`导入覆盖确认` 节点，并标出导入必须先过 schema 与业务校验。

- [x] **3. 细化加密和解锁节点**

  Canvas 现在只写了 `EncryptionService`、`AES-256-GCM`、`PBKDF2`，还不够作为安全运行图。后面实现时必须明确：

  - `UnlockScreen / PasswordInput`
  - `PBKDF2 key derivation`
  - `salt`
  - `iv`
  - `ciphertext`
  - `authentication tag`
  - `in-memory key`
  - 密码错误 / 密文损坏的统一失败结果

  修图要求：补一个加密包格式，例如：

  ```text
  EncryptedEnvelope = {
    version,
    kdf: "PBKDF2",
    salt,
    iterations,
    algorithm: "AES-256-GCM",
    iv,
    ciphertext
  }
  ```

  安全要求：IV 必须每次随机；派生密钥不能落盘；上层只拿明文 `LedgerData`。

  处理状态：已在副本图中补 `Unlock / key lifecycle` 与 `EncryptedEnvelope` 节点，标注 PBKDF2、AES-256-GCM、salt、iv、ciphertext、in-memory key 与统一失败结果。

- [x] **4. 统一论文范围里的 `position-adjustment` 和代码类型**

  毕业论文方案写了系统记录 `buy, sell, and position-adjustment transactions`，但当前 Canvas 和代码里的 `TradeType` 只有：

  ```ts
  type TradeType = "buy" | "sell";
  ```

  必须二选一：

  - 方案 A：把论文里的 `position-adjustment` 改成 future work，不进入暑假核心。
  - 方案 B：现在设计 `adjustment` 类型，并明确它如何影响 `quantity / costBasis / realizedPnl`。

  建议：为了保护 Week2-W10 主线，优先选方案 A，把 adjustment 推迟到 future work。

  处理状态：已在副本图中补 `Scope decision` 节点，明确 `TradeType` 在 Week2-W10 保持 `"buy" | "sell"`，`position-adjustment` 进入 future work。

- [x] **5. 补上确定性 ID 策略和重复交易识别**

  论文 Month 2 要求 deterministic ID，需求总纲也提到自动识别重复交易。Canvas 目前只有 `id` 字段，没有说明 ID 怎么来，也没有重复交易检测。

  需要补：

  ```text
  TradeDraft
    -> normalize
    -> deterministicId / fingerprint
    -> duplicateDetector
    -> TradeValidator
    -> saveTrade
  ```

  至少要防：

  - 用户双击保存导致重复写入。
  - JSON / CSV 导入重复交易。
  - 同一条交易因为格式差异生成多个记录。

  建议先做一个 `tradeFingerprint`：资产、类型、日期、数量、价格、总金额、币种、手续费这些字段规范化后生成 hash。

  处理状态：已在副本图中补 `normalize / ID / duplicateDetector` 节点，并把 Draft 到 Validator 的路径改成先规范化、生成 fingerprint、再校验。

- [x] **6. 补上 schema 校验和迁移机制**

  `LedgerData` 已经有 `schemaVersion: 1`，但 Canvas 没有画 `schemaValidator` 或 `migrator`。这会影响导入旧数据、读取旧版本 IndexedDB、未来字段升级。

  需要补：

  ```text
  StorageAdapter.read / ImportAdapter
    -> parse JSON
    -> validate schema
    -> migrate if needed
    -> LedgerData vCurrent
  ```

  修图要求：在 IndexedDB 读出和 JSON 导入之后，都必须经过 schema validation / migration，不能把未知结构直接交给业务层。

  处理状态：已在副本图中补 `schemaValidator / migrator` 节点，并连接到安全读取路径与导入路径。

- [x] **7. 标注 whole-blob IndexedDB 只是第一阶段策略，并补性能升级路线**

  当前 Canvas 写的是 whole-blob 读写。Week4 第一版这样做合理，但论文 benchmark 要测 100 / 1,000 / 10,000 / 100,000 笔交易，如果图上看起来永远整坨读写，性能评审容易被追问。

  需要补一条虚线未来路线：

  ```text
  Phase 1:
  whole-blob LedgerData

  Future performance path:
  IndexedDB object stores
    -> trades store
    -> priceSnapshots store
    -> assets store
    -> indexes
    -> PositionCache / checkpoints
  ```

  注意：这不是现在马上实现分片，而是在图里承认性能升级路径。

  处理状态：已在副本图中补 `IndexedDB 性能路线` 节点，明确 Phase 1 是 whole-blob，未来路线是 object stores、indexes、PositionCache / checkpoints。

- [x] **8. 补上 PriceSnapshotDraft 和价格校验**

  Canvas 对交易有 `tradeValidator`，但价格输入只写了 `PriceInput` 和 `PriceSnapshot`，缺少 `PriceSnapshotDraft` 的校验流程。

  需要补：

  ```text
  PriceInput
    -> PriceSnapshotDraft
    -> priceSnapshotValidator
    -> priceSnapshotService
    -> repository.savePriceSnapshot
  ```

  至少校验：

  - asset 必须存在。
  - price > 0。
  - currency 第一版必须符合 MVP 口径，建议默认 USD。
  - recordedAt 必须是合法日期 / 时间。
  - source 第一版为 `manual`。

  处理状态：已在副本图中补 `priceSnapshotValidator.ts` 节点，并把 `PriceSnapshotDraft` 从 normalize / ID / duplicateDetector 接到价格校验和 service。

## 安全提醒

- 加密只能保护本地密文，不能自动防 XSS 或恶意前端脚本。
- 明文 JSON 备份必须有明显警告。
- 导入文件永远不能信任。
- 外部 API 和实时价格不在论文核心范围内，未来接入时必须默认关闭，并且不能上传用户交易数据。
