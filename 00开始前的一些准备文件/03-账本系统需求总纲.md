> > [!note]+ ## 二、 商业级系统架构设计 (Architecture)
> > 为了实现 **模块7：扩展与插件生态** 7，我们不能把代码写死。我们需要采用 **“微内核 + 插件化” (Micro-kernel + Plugin)** 架构。
> > 
> > ### 1. 架构分层 (Layered Structure)
> > 
> > 我们将系统分为四层，层与层之间通过“接口”对话，互不干扰：
> > 
> > - **L1 - 基础设施层 (Infrastructure)**
> >     - 负责读写数据库、加密解密（对应 **模块6** 8）。
> >     - *原则：这一层不管业务，只管存数据。*
> > - **L2 - 核心业务层 (The Kernel)**
> >     - 这里是系统的“大脑”。包含 **模块1：核心交易账本** 9 的逻辑。
> >     - 比如：`计算平均持仓成本` 10、`资金池管理` 11 的算法就在这里。
> > +1
> >     - *原则：这是系统最稳固的部分，绝对不能因为装了一个插件就崩了。*
> > - **L3 - 插件/扩展层 (The Plugins)**
> >     - 这里是体现你“高拓展性”的地方。
> >     - 我们将 **策略系统 (模块3)**、**情绪系统 (模块4)**、**趣味系统 (模块8)** 全部视为“插件”。
> >     - 比如：你想增加一个“RSI指标” 12，就在这里写一个插件，而不是去改核心代码。
> > - **L4 - 表现层 (Presentation)**
> >     - 用户看到的界面。Web 页面、手机 App 界面。
> >     - 它负责把 L2 和 L3 的数据画出来。

> > 2. 本地加密（至少：敏感字段 + API Key）
> > 3. 自动备份 + 一键恢复（最小实现）
> > 4. 网络访问权限控制（外部 API 默认关闭）

> > - 备份/恢复流程稳定可重复
>

> > [!note]+ # Phase 5（第 7–8 月）：模块④ 行为金融 & 情绪系统（Emotion v1）

> > 
> > - 情绪系统不影响核心账本（完全可关闭）

> > [!note]+ # Phase 7（第 10 月）：模块⑦ 插件生态系统（Plugin Framework v1）

> > 1. 插件注册与加载机制
> > 2. 插件隔离运行（最小沙盒）
> > 3. 一个示例插件：RSI 或一个策略插件
> > 4. 插件安装/移除（本地）

> > ## 验收标准（Gate 7）

> > - 插件崩溃不能影响内核
> > - 插件只能拿到受限接口（不能随便读写数据库）

> [!note]+ # 风险清单与应对策略
> ### 本章节目的
> 
> - **老板**：看清项目会在哪些地方翻车、怎么防止延期/烂尾/返工/安全事故。
> - **程序员**：知道每个风险对应的“工程动作”和“验收门槛（Gate）”。
> 
> > 约定：风险分级
> 
> - **P0（致命）**：一旦发生，项目可能需要大规模返工或无法上线
> - **P1（严重）**：会明显拖慢进度、影响质量，必须提前管
> - **P2（可控）**：常见小坑，按流程可规避
>

> 
> > [!note]+ ## 风险 1：DG 不稳导致“数据结构僵化”，后期迁移地狱（P0）
> > <!-- Column 1 -->
> > ### 风险是什么（老板版）
> > 
> > 前期字段没设计全，后期要加情绪/外部数据/插件时发现存不下，只能推倒重来或做痛苦迁移。
> > 
> > 
> > <!-- Column 2 -->
> > ### 为什么会发生（程序员版）
> > 
> > Trade/Asset 等核心结构没冻结、没有扩展机制、没有 schemaVersion/迁移策略。
> > 
> > ### 应对策略（必须做）
> > 
> > - Phase 0 建立 **Data Dictionary v1（数据字典）**并冻结
> > - 采用 **“主结构稳定 + 扩展信息外置”**：
> >     - 核心交易字段只放事实
> >     - 情绪/新闻/策略通过 `meta` 或关联表挂载
> > - 数据结构带 `schemaVersion`，未来变更必须提供默认值/兼容读取
> > 
> > ### 验收标准（Gate-DG）
> > 
> > - 不改 Trade 主字段，也能挂上情绪与新闻引用
> > - 旧数据导入后仍能被新版本读取
> 
> ---
> 
> > [!note]+ ## 风险 2：过早追求插件系统（模块⑦）导致烂尾（P0）
> > ### 风险是什么
> > 
> > 为了“未来生态”，早期设计复杂的插件权限/市场/隔离，导致内核一直做不出来。
> > 
> > ### 应对策略
> > 
> > - **先写死，再抽离**（你提出的蛋糕胚策略，确认正确）
> > - Phase 1–6：核心功能写死 + 只预留接口
> > - Phase 7：只做插件系统最小闭环：
> >     - 加载/卸载
> >     - 隔离
> >     - 受限 API
> >     - 1 个 demo 插件
> > 
> > ### 验收标准（Gate-Plugin v1）
> > 
> > - 插件崩溃不影响内核
> > - 插件不能直接写核心数据（v1 禁止）
> > - 可新增一个指标/策略而不改内核代码
> 
> ---
> 
> > [!note]+ ## 风险 3：单人四角色“串台”，导致决策混乱与内耗（P0）
> > ### 风险是什么
> > 
> > 你一会儿当老板要功能、一会儿当用户想爽、一会儿当工程师想省事、一会儿当架构师想完美，结果天天改需求、天天推翻。
> > 
> > ### 应对策略
> > 
> > - 固定每周“角色时间块”：
> >     1. PM 时间：只排优先级/砍需求
> >     2. Architect 时间：只审结构/接口/变更
> >     3. Dev 时间：只写代码不改需求
> >     4. User 时间：只体验提问题不当场改
> > - **任何新增需求进入 Backlog，不插队**
> > 
> > ### 验收标准
> > 
> > - 每个 Sprint（2周）必须交付可运行版本
> > - 需求变更只能在 Sprint 开始时进入，不允许中途无限插单
> 
> ---
> 
> > [!note]+ ## 风险 4：范围膨胀（152 条目“诱惑”）导致延期（P1）
> > ### 风险是什么
> > 
> > 看到功能清单就想做全，导致每个阶段都超出范围，最终没有一个版本稳定可用。
> > 
> > ### 应对策略
> > 
> > - 每个 Phase 只做**最小可验收子集（MVS）**
> > - 用“红线规则”：
> >     - 本阶段未写入计划的功能 = 一律不做
> > - 每月一次“砍需求会议”：删掉 10% 可选功能（保持可控）
> > 
> > ### 验收标准
> > 
> > - 每阶段 Gate 清单固定，达到就进入下一阶段
> > - Backlog 只增不做，除非明确替换（A 做不了就删A做B）
> 
> ---
> 
> > [!note]+ ## 风险 5：账本计算错误（均价/盈亏/手续费/卖出逻辑）导致“不可信”（P0）
> > ### 风险是什么
> > 
> > 这类产品最大风险不是 UI，而是**算错一次就再也不敢用**。
> > 
> > ### 应对策略
> > 
> > - Phase 1 必须建立“黄金用例（Golden Cases）”
> >     - 10~20 组固定交易样例
> >     - 对应人工结果（均价/持仓/盈亏）
> > - 计算逻辑必须集中在 `calculations.ts`，禁止散落在 UI
> > - 引入精度策略（至少金额 2 位、小数数量 8 位，统一 round 规则）
> > 
> > ### 验收标准（Gate-Calc）
> > 
> > - 所有黄金用例 100% 通过
> > - 卖出超持仓、负数输入等必须提示错误且不破坏数据
> 
> ---
> 
> > [!note]+ ## 风险 6：数值精度与货币单位混乱（P1）
> > ### 风险是什么
> > 
> > 不同币种、不同 quote（USD/CNY/HUF）、手续费、价格精度不统一，会导致“看起来对、细算错”。
> > 
> > ### 应对策略
> > 
> > - Phase 0 冻结“金额/数量/精度规则”
> > - 统一 Money/Quantity 表达：
> >     - 金额：按最小单位或统一 round
> >     - 数量：8 位小数
> > - 先固定一种计价货币（例如 USD），多币种换算延后
> > 
> > ### 验收标准
> > 
> > - 同一笔交易在 UI/导出/计算中金额一致
> > - 汇率引入前后不影响历史记录的原始数据
> 
> ---
> 
> > [!note]+ ## 风险 7：本地数据库选型过重导致开发卡死（P1）
> > ### 风险是什么
> > 
> > 一上来就 SQLite(WASM)/复杂同步，会把你困在“搭数据库”而不是“做产品”。
> > 
> > ### 应对策略（分阶段）
> > 
> > - Phase 1：IndexedDB/localStorage（先跑通闭环）
> > - Phase 4：再升级到 PouchDB 或 SQLite
> > - 数据访问统一走 `database.ts`，避免未来迁移改全项目
> > 
> > ### 验收标准
> > 
> > - Phase 1 的数据层可被替换（只改 database.ts 不动业务）
> 
> ---
> 
> > [!note]+ ## 风险 8：安全与隐私未落地，外部数据接入变成“泄漏点”（P0）
> > ### 风险是什么
> > 
> > 如果 API Key/交易数据没加密、权限没管控，一旦接外部数据就是安全洞。
> > 
> > ### 应对策略
> > 
> > - 外部访问默认关闭，必须用户手动授权（开关）
> > - API Key 加密存储（最小可用即可）
> > - “离线模式”必须可完全使用核心功能
> > 
> > ### 验收标准（Gate-Security v1）
> > 
> > - 断网仍可用账本核心
> > - 未授权时系统不会发起任何外部请求
> > - Key 不以明文存储
> 
> ---
> 
> > [!note]+ ## 风险 9：数据丢失（浏览器缓存清理/设备更换/误操作）（P0）
> > ### 风险是什么
> > 
> > 你辛苦记账几个月，数据没了——项目直接失去意义。
> > 
> > ### 应对策略
> > 
> > - Phase 1 就必须有“手动备份/恢复”（导出文件）
> > - Phase 4 加“自动备份”（本地多份 + 提醒）
> > - 恢复流程必须傻瓜：导入文件→校验→恢复
> > 
> > ### 验收标准（Gate-Backup）
> > 
> > - 任意时间点都能导出并在新环境恢复
> > - 恢复后均价/持仓/图表一致
> 
> ---
> 
> > [!note]+ ## 风险 10：性能退化（交易多了卡、图表卡）（P2→P1）
> > ### 风险是什么
> > 
> > 交易记录一多、图表一开就卡顿，使用体验崩溃。
> > 
> > ### 应对策略
> > 
> > - Phase 1/2 先做“分页/虚拟列表”（交易表）
> > - 图表计算走 memo/cache，不在渲染时重复算
> > - 大计算（回测/模拟）延后或放到 worker（后期）
> > 
> > ### 验收标准
> > 
> > - 1万条交易仍可打开账本页（可稍慢但不崩溃）
> > - 图表切换资产不卡死
> 
> ---
> 
> > [!note]+ ## 风险 11：UI 复杂度过早追求“金融级质感”拖慢开发（P1）
> > ### 风险是什么
> > 
> > 为了好看/动效/完美排版消耗大量时间，核心功能没完成。
> > 
> > ### 应对策略
> > 
> > - Phase 1：只用 shadcn 默认组件 + 简单布局
> > - Phase 8：统一做 UI/动效/视觉升级
> > - UI 优化不得阻塞核心 Gate
> > 
> > ### 验收标准
> > 
> > - Phase 1 只看流程是否顺畅，不看“像不像 TradingView”
> 
> ---
> 
> > [!note]+ ## 风险 12：需求变更没有记录，导致“反复横跳”返工（P1）
> > ### 风险是什么
> > 
> > 今天想做 A，明天觉得 B 更好，后天又回 A；最终代码变成拼凑。
> > 
> > ### 应对策略
> > 
> > - 建立一个简单的变更日志（CHANGELOG/DECISIONS）
> >     - 记录：做了什么决定、为什么、替代方案
> > - 任何重大变更都必须写 3 行：
> >     1. 变更内容
> >     2. 原因
> >     3. 影响范围
> > 
> > ### 验收标准
> > 
> > - 关键决策可追溯（否则下次还会重复纠结）
> 
> ---
> 
> > [!note]+ ## 风险 13：测试缺失导致“越修越坏”（P0）
> > ### 风险是什么
> > 
> > 功能越来越多，改一个地方坏三处，最后不敢动代码。
> > 
> > ### 应对策略
> > 
> > - 最低成本测试策略：
> >     - 计算逻辑：黄金用例（固定输入→固定输出）
> >     - UI 核心流程：手工回归清单（10 条）
> > - 每次发版本必须过一遍回归清单
> > 
> > ### 验收标准
> > 
> > - 每次合并代码前，黄金用例全通过
> > - 回归清单无红项
> 
> ---
> 
> > [!note]+ ## 风险 14：打包与多端（Tauri/Capacitor）过早介入（P2→P1）
> > ### 风险是什么
> > 
> > 早期去搞桌面端/移动端打包，会把精力从产品内核拖走。
> > 
> > ### 应对策略
> > 
> > - 先 Web/桌面二选一（建议先 Web）
> > - 打包只在 Phase 8 进入正式工作
> > - Phase 0 只做“适配预留”：不写死浏览器 API
> > 
> > ### 验收标准
> > 
> > - 核心功能在 Web 稳定后再谈多端
>
