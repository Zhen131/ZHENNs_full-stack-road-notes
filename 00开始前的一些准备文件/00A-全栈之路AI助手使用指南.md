# 全栈之路 AI 助手使用指南

更新时间：2026-06-25
状态：长期项目规则
用途：让新的 AI 会话快速进入 Local-First Trading Ledger，不要求用户反复解释目录、进度和文件位置。

## 结论

本工作区使用三层入口：

| 层级 | 文件 | 作用 |
| --- | --- | --- |
| 固定规则 | `AGENTS.md` | Codex 自动读取的目录、隐私、Git 和执行边界 |
| 任务流程 | `.agents/skills/ledger-workflow/SKILL.md` | 按写代码、Git、讨论、继续、日志选择上下文 |
| 动态进度 | `01一些进度/日志/00-当前开发状态.md` | 当前里程碑、下一步、风险和最近验证 |

设计、实施计划和用户速查统一放在 `99_tool/`。

AI 不再按固定顺序读取大量文件。先判断任务模式，再读取最小必要材料。

## 用户日常用法

推荐说法：

```text
账本写代码：今天实现未实现盈亏
账本 Git：检查当前改动并整理版本
账本讨论：IndexedDB 应该放在哪一层
账本继续
账本日志：总结今天完成的代码和测试
```

这些口令不是强制格式。用户可以直接说：

```text
帮我把 Validator 和 Calculator 接到页面内存状态。
看看账本源码现在适不适合提交。
我们讨论一下最新价格应该怎样选。
```

AI 应根据具体目标识别模式，不要求用户重新介绍项目。

## 任务路由

| 模式 | 默认读取 | 按需读取 |
| --- | --- | --- |
| 写代码 | 当前状态、源码 README、源码 Git 状态、相关代码和测试 | 当前周计划、相关设计 |
| Git | 目标仓库状态、分支、近期提交、diff | 当前状态、任务相关文档 |
| 讨论 | 当前状态、相关需求或设计 | 相关代码、Frappe Books、Canvas |
| 继续 | 当前状态、当前周计划、README、两个仓库状态 | 最近日志和相关代码 |
| 日志 | 当前状态、当前周计划、实际变更和测试结果 | 文档产出规范 |

默认不要读取所有历史日志、整个参考仓库或全部 Canvas。

## 目录语义

### 用户账本源码

`01一些进度/产出/LocalFirstTradingLedger/`

- 真正的 Local-First Trading Ledger 源码。
- 是独立 Git 仓库。
- 写代码、测试和源码版本管理以这里为准。

### 进度与日志

`01一些进度/日志/`

- `00-当前开发状态.md`：跨聊天框的当前入口。
- `00-W2-W12总路线图与执行说明.md`：跨周主线。
- `week*_*/`：每周计划、设计、验收和开发日志。
- 历史 Week 文件用于追溯，不在每次启动时全部读取。

### 前期规划

`00开始前的一些准备文件/`

- 暑假总览、需求总纲、论文方案和长期项目材料。
- 用于范围判断，不替代当前开发状态。
- 新功能与当前计划冲突时，优先守住 P0 主线。

### Frappe Books

`00frappe-books-typescript/`

- 外部参考项目，不是用户源码。
- 只用于定向参考 schema、model/service、repository/storage、report 和 UI 分层。
- 默认不读取整个仓库，不照搬 POS、税务、复式记账、插件、多语言或 Electron 复杂度。

### NLP

`02_NLP/`

- 后续要接入账本的模块。
- 目前不移动、不改写。
- 只有 NLP 学习、自然语言录入或后期账本接入任务才读取。

### 工具

`99_tool/`

- 保存账本上下文系统的设计、实施计划、技能清单和使用说明。
- 它是管理入口，不保存动态开发进度。

### Canvas 与私人资料

- `*.canvas` 只在架构图或数据流任务中读取。
- `99-私人网络环境说明.md` 永不主动读取、引用或提交，除非用户明确要求。

## 推进原则

1. 先确认当前任务和产出物。
2. 只读取能支持当前任务的材料。
3. 写代码时先定位实现和测试，再做最小可运行改动。
4. 页面只负责输入和展示；计算归 calculator/service，保存归 repository，数据形状归 models。
5. 完成后运行与风险相称的测试、lint 或 build。
6. 只有实质进展改变里程碑、下一步、风险或验证结果时，才更新当前开发状态。
7. 明显阶段性成果完成后，再询问是否进行 Git 提交或推送。

## 范围原则

- P0：数据模型、计算正确性、IndexedDB、导入导出、加密、benchmark、论文素材。
- 先能跑，再变好。
- NLP 和 Agent 必须服务账本，不得挤占核心交付。
- 新想法先判断是否属于当前周、是否服务论文主线、是否可以进入 backlog。
- 不为了架构完整而提前实现未来功能。

## Git 规则

本工作区包含两个独立仓库：

```text
根目录                                  文档、日志、计划和工具
01一些进度/产出/LocalFirstTradingLedger  账本源码
```

Git 操作必须先确认目标仓库。不要混用状态、分支、提交或推送。

默认不提交：

- `.DS_Store`
- `.obsidian/workspace.json`
- `00frappe-books-typescript/`
- 私人网络环境说明
- 构建产物和依赖目录

## 回答与文档风格

- 简洁、直接、先给结论。
- 少讲基础概念，多给文件、规则、边界和验证证据。
- 文档遵循 `01一些进度/日志/00-文档产出规范.md`。
- 如果项目显得太大，回到当前状态文件中的“下一步”。

## 一句话总纲

> 用户只负责说今天要完成什么；系统负责找到正确上下文、守住边界、执行验证并留下可继续的位置。
