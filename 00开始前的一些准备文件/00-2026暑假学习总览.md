# 2026 暑假学习总览

时间：2026-06-12 至 2026-08-20  
节奏：每周 6 天，每天 4 小时以上  
总量：10 周，约 60 个学习日，至少 240 小时

---

## 一、总原则

这个暑假不按“我要学很多技术”来排，而按“我要交付什么东西”来排。

核心目标只有一个：

> 做出论文项目的可运行 MVP，并让 NLP 与 Agent 学习都服务于这个项目。

暑假结束时，理想状态不是“学完 NLP、学完 Agent、学完全栈”，而是：

1. 有一个能演示的 Local-First Trading Ledger。
2. 有清楚的论文实验方案与初步 benchmark。
3. 有一个最小 NLP 输入模块。
4. 有一个最小 Agent 问答模块。
5. 有一套项目文档、学习记录和后续路线。

---

## 二、优先级排序

### P0：论文项目主线

对应文件：

- `04-毕业论文项目方案.md`
- `03-账本系统需求总纲.md`

必须完成：

1. 数据模型：Trade / Asset / Position / schemaVersion。
2. 核心计算：DCA 均价、持仓、已实现盈亏、未实现盈亏、卖出超持仓校验。
3. 价格快照：暑假版先手动输入当前价格，但设计时预留未来接实时价格 API 的位置。
4. 本地存储：从一开始就经过可替换的本地 repository 层，不让页面直接碰浏览器存储。
5. 加密：尽早进入数据保存流程，避免后期硬塞加密导致返工。
6. 导入导出：JSON backup / restore。
7. 可视化：持仓饼图、资产增长曲线、交易点标注三选二或三者最小版。
8. benchmark：100、1,000、10,000、100,000 笔合成交易测试。

判断标准：

> 这个项目能支撑毕业论文的 System Design、Implementation、Evaluation 三章。

### P1：NLP

只做“账本输入理解”，不泛学 NLP。

暑假版边界：

- NLP 只允许新增交易。
- 不允许通过 NLP 修改交易。
- 不允许通过 NLP 删除交易。
- 不允许通过 NLP 生成投资建议。

暑假目标：

用户输入：

> 今天买入 0.1 BTC，价格 65000 美元，手续费 5 美元。

系统解析成结构化交易：

```json
{
  "type": "buy",
  "asset": "BTC",
  "quantity": 0.1,
  "price": 65000,
  "fee": 5
}
```

允许从规则解析开始，不必一开始就训练模型。

判断标准：

> NLP 模块能作为课程项目雏形，也能变成账本系统的输入增强功能。

### P2：AI Agent 开发

只做“账本查询 Agent”，不做通用 Agent。

暑假版边界：

- Agent 只允许查账。
- Agent 不允许给投资建议。
- Agent 不允许替用户下判断，例如“你应该买/卖”。
- Agent 回答数字时，必须来自账本计算结果。

暑假目标：

用户可以问：

- 我的 BTC 平均成本是多少？
- 本月收益是多少？
- 如果 BTC 到 80000，我大概浮盈多少？

Agent 通过 tool calling 调用账本计算函数，而不是凭空回答。

判断标准：

> Agent 是项目的自然延伸，而不是另开一条战线。

### P3：RAG / 插件 / 外部数据 / 高级图表 / 自媒体

全部降级为素材和未来 backlog。

可以记录灵感，但不插队。

## 三、每周计划

## Week 1：6月12日 - 6月18日

主题：先想清楚账本长什么样，并做出一个空壳

这一周要做成什么：

- 不急着写复杂功能，先把“我要做的账本到底是什么”说清楚。
- 做出一个能打开的空网页，让项目正式开始。
- 提前想清楚数据保存和加密路线，避免后面返工。

这一周结束时，我应该看见：

1. 一份“账本第一版说明书”：这个账本解决什么问题，第一版只做哪些功能。
2. 一份“交易记录清单”：一笔交易需要记录日期、买入/卖出、资产名、数量、价格、手续费、备注。
3. 一份“价格记录清单”：当前价格先手动输入，未来再接实时价格 API。
4. 一个能打开的网页项目，首页能看到项目名字。
5. 页面草图：添加交易、交易列表、资产汇总、价格输入这四块先画出来。
6. 一条技术规则：页面不能直接保存数据，所有数据都必须经过统一的保存层。
7. 5 条假交易样例，用来以后检查系统算得对不对。

不要做：

- 插件系统
- 多端同步
- 外部 API
- 复杂 UI

详细每日执行清单见：`01-Week1每日执行清单.md`

---

## Week 2：6月19日 - 6月25日

主题：先让账本算得对，并把价格来源说清楚

这一周要做成什么：

- 不管页面好不好看，先让系统会算买入、卖出、持仓、均价和盈亏。
- 所有计算都要能用手算检查。
- 暑假版不接实时价格 API，但要能手动输入当前价格来计算浮盈/浮亏。

这一周结束时，我应该看见：

1. 输入几条交易后，系统能算出当前还持有多少资产。
2. 系统能算出平均买入成本。
3. 系统能算出已经卖出赚/亏了多少。
4. 一个手动价格快照：比如 BTC 当前价格是多少、ETH 当前价格是多少。
5. 系统能根据手动价格估算还没卖出的浮盈/浮亏。
6. 如果我卖出的数量超过持仓，系统会报错。
7. 至少用 5 条假交易做检查，结果要和手算一致。

验收：

- 给一组交易，系统结果与手算一致。
- 就算没有漂亮页面，计算功能也能单独跑通。
- 不接实时价格 API，但代码结构不能把未来接 API 的路堵死。

## Week 3：6月26日 - 7月2日

主题：做出第一个能保存、能加密雏形的账本

这一周要做成什么：

- 把 Week 2 的计算功能放进网页里。
- 我可以在页面上新增交易、查看交易、刷新后数据还在。
- 数据保存从这一周就开始走统一保存层，为加密做准备。

这一周结束时，我应该看见：

1. 一个“添加交易”的表单。
2. 一个“交易列表”，能看到我输入过的每一笔交易。
3. 一个“价格输入”的小表单，可以手动填当前价格。
4. 一个“资产汇总”，能看到当前持仓、均价、盈亏。
5. 数据能保存在本地，刷新网页后不会消失。
6. 数据保存必须经过统一保存层，不能在页面里随手保存。
7. 加密流程的最小雏形：至少明确密码从哪里输入、数据从哪里加密。
8. 一个“导出备份”按钮，可以导出 JSON 文件。

验收：

- 刷新页面后数据仍在。
- 导出的文件我能看懂，也能作为备份保存。
- 以后要改加密时，只改保存层，不大改页面。

## Week 4：7月3日 - 7月9日

主题：把加密、备份、恢复做成完整闭环

这一周要做成什么：

- 让账本不只是能用，还符合“本地优先”和“加密保存”的论文主题。
- 我应该能把数据导出、清空、再导回来，而且数据不乱。
- Week 3 先做加密雏形，这一周把它做完整。

这一周结束时，我应该看见：

1. 一个“导入备份”按钮，可以把之前导出的文件恢复回来。
2. 一个密码输入流程：打开或导出时需要输入密码。
3. 本地保存的数据不是明文交易记录。
4. 导出、清空、导入之后，交易和统计结果完全一致。
5. 一份简单记录：断网时账本仍然能打开、能使用。

验收：

- 导出后清空，再导入，状态一致。
- 存储层不直接保存明文交易数据。

主题：把账本结果变成看得懂的图

这一周要做成什么：

- 不只看数字，还要能用图看懂资产分布和变化。

1. 一个 Dashboard 页面。

3. 一个资产变化图：能看到总资产随时间怎么变化。

- 图表数字与账本汇总一致。

## Week 6：7月17日 - 7月23日

主题：测试账本在大量交易下跑得快不快

这一周要做成什么：

- 不只是说“这个系统能用”，而是用数据证明它在 100、1,000、10,000、100,000 笔交易下表现如何。

这一周结束时，我应该看见：

1. 一个自动生成假交易的工具。
2. 100、1,000、10,000、100,000 笔四组测试数据。
3. 一张表：不同数据量下，保存交易要多久。
4. 一张表：不同数据量下，查询持仓和盈亏要多久。
5. 一张表：不同数据量下，重新计算全部结果要多久。
6. 初版实验结果，可以先粗糙，但必须能重复跑。

验收：

- 这些速度测试可以重复运行。
- 每个规模至少跑 3 次，后续再补到 10 次。

## 四、跳票周机制

真实开发一定会延期，所以计划允许“跳票周”，但不能放纵跳票。

触发条件：

- 如果某一周结束时，有 4 个或以上本周成果没有完成，就启动跳票周判断。
- 这里的“没完成”只算计划里明确写出的成果，不算临时冒出来的新想法。

跳票周怎么用：

- 启动跳票周后，下一周不做新主题，只用来补齐欠下的核心成果。
- 补完后，后面的周计划整体顺延一周。
- 跳票周优先补 P0 主线：账本计算、保存、加密、备份、benchmark。
- 跳票周不拿来加新功能。

限制：

- 每个自然月最多使用 1 次跳票周。
- 8 月 7 日之后原则上不再启动跳票周，因为 Week 9 和 Week 10 要留给演示、整理、复盘。
- 如果已经用完当月跳票周，仍然延期，就必须砍功能，而不是继续顺延。

砍功能顺序：

1. 先砍高级图表，只保留两个核心图。
2. 再砍 benchmark 的最大规模，100,000 笔可以延后。
3. 再砍 NLP，只保留中文新增交易。
4. 再砍 Agent，只保留 3 个固定查账问题。
5. 不砍核心账本、加密、导入导出。

## 七、范围控制规则

暑假内允许做：

- 核心账本
- 本地加密
- 导入导出
- 手动价格快照 Market Price Snapshot
- 未来实时价格 API 的接口预留
- 基础图表
- benchmark
- NLP 交易输入
- Agent 账本问答

暑假内不做：

- 插件市场
- 多端同步
- Tauri / Capacitor 打包
- 实时价格 API 完整接入
- 新闻抓取
- K 线高级指标
- 复杂策略系统
- 游戏化
- 完整 RAG 知识库

这些不是不要，而是放到 8 月 20 日之后。

关于实时价格 API：

- 产品长期一定需要实时价格 API，否则对短线交易者和经常查看收益的用户不现实。
- 暑假版暂时不接 API，是为了保护主线进度。
- 暑假版用手动价格快照代替实时价格。
- 代码设计时必须预留价格来源接口，未来可以把手动价格替换成 API 价格。

## 九、一句话版本

这个暑假不是“三线并行学完所有东西”，而是：

> 用论文账本项目做主线，把 NLP 做成输入能力，把 Agent 做成查询能力，最后形成一个能演示、能写论文、能继续扩展的产品原型。
