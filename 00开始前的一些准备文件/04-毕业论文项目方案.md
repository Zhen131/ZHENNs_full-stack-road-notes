***A Local-First Personal Trading Ledger: Design, Implementation, and Performance Evaluation of an Encrypted Browser-Based Financial Tool***
**Student:** Zhu Zhen
**Supervisor:** Dr. Attila Tiba
**Date:** April 2026

---

The project will be implemented in TypeScript using Next.js 14 (App Router) as the application framework, Tailwind CSS and Shadcn/UI for the user interface, IndexedDB as the local transactional database, and the Web Crypto API (AES-256-GCM) for client-side encryption. Charts are rendered with Recharts, automated tests are written in Vitest, and end-to-end performance benchmarks are executed with Playwright. Development takes place in a local environment (VS Code, Node.js LTS), with version control through Git and a public GitHub repository. The system runs entirely in the browser; no user data is transmitted to any external server, and all core features remain functional without an internet connection. There is no external dataset; instead, a deterministic synthetic-trade generator produces benchmark workloads of 100, 1,000, 10,000, and 100,000 transactions for the evaluation experiments.

The program functions as a single-page web application centered on a personal trading ledger. It records buy, sell, and position-adjustment transactions with timestamps and fees, computes the dollar-cost-average (DCA) basis, current holdings, and real-time profit-and-loss without manual input, and visualizes portfolio composition and growth through three core charts: a holdings pie chart, an asset-growth time-series curve, and a transaction-marker overlay. All records are encrypted at rest in IndexedDB using AES-256-GCM and can be exported to and re-imported from a portable JSON format. The main research contribution is an empirical feasibility study of the local-first software paradigm proposed by Kleppmann et al. (Onward! 2019) in the specific domain of personal financial tooling: the system is evaluated against the seven local-first ideals, and a benchmark suite measures write latency, query latency, full portfolio P&L computation time, and cold-start time across four data scales (10², 10³, 10⁴, 10⁵ transactions), with each measurement averaged over ten runs.

**Condensed Work Plan (9 Months)**

**Month 1 — Setup and literature review.** Study the foundational local-first software paper (Kleppmann et al., 2019) and recent literature on browser-based encrypted storage, IndexedDB performance characteristics, and CRDT-based architectures. Set up the TypeScript and Next.js environment, create the public GitHub repository, and define the data model for transactions, holdings, and assets in a versioned schema.

**Month 2 — Core ledger module.** Implement transaction recording (buy, sell, adjustment) with field validation and a deterministic ID strategy. Implement the calculations layer covering DCA cost basis, current holdings, realized and unrealized P&L, and anomaly detection (e.g., over-selling guards). Cover the calculations module with a golden-case unit test suite and establish CI on GitHub Actions.

**Month 3 — Encrypted local storage.** Integrate IndexedDB through a typed repository layer and implement client-side AES-256-GCM encryption via the Web Crypto API, with a key-derivation flow based on PBKDF2 and a user passphrase. Implement JSON export and import for full backup and restore, and verify byte-for-byte round-trip integrity.

**Month 4 — Visualization and user interface.** Implement the three core charts (holdings pie chart, asset-growth curve, transaction-marker overlay) bound to the same single source of truth as the ledger. Build the primary ledger page, transaction-entry forms, and import/export controls. Confirm that the application remains fully functional in a hard-offline browser session.

**Month 5 — Benchmark design and execution.** Formalize the performance evaluation protocol: workloads of 10², 10³, 10⁴, and 10⁵ synthetic transactions; metrics covering write latency, query latency, full-portfolio P&L computation time, and cold-start time; ten runs per configuration with median and interquartile range reported. Implement the synthetic-trade generator and the Playwright-based benchmark harness, and execute the full sweep.

**Month 6 — Evaluation against local-first ideals.** Analyze benchmark results, produce risk-of-misuse and performance-degradation curves across data scales, and complete a structured assessment of the system against the seven local-first ideals (Kleppmann et al., 2019), explicitly distinguishing ideals realized in this thesis (no-spinner responsiveness, optional network, the long now, security and privacy by default, user ownership) from those deferred to future work (multi-device sync, real-time collaboration).

**Month 7 — Final evaluation and freeze.** Lock the implementation, run a small structured user-walkthrough study (3–5 participants) covering trade entry, import/export, and offline use. Produce publication-quality figures for all benchmark results, complete an optional ablation comparing encrypted versus plaintext storage cost, and freeze the codebase under a tagged release.

**Month 8 — Writing.** Draft the full thesis in LaTeX, covering Introduction, Related Work, System Design, Implementation, Evaluation, Discussion, and Conclusion. Submit chapters to the supervisor on a rolling weekly basis and revise based on feedback.

**Month 9 — Defense and submission.** Prepare and rehearse the defense presentation, anticipate likely examiner questions, submit the final thesis ahead of the faculty deadline, and release the project repository publicly with a clean README, reproducible benchmark command, and licensed source code.
