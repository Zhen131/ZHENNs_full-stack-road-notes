## 0. 怎么读这份计划

这份计划做了两个和原总览不同的关键调整，先讲清楚原因：

**调整一：周期定为 12 周，核心在 Week 10 冻结。**
论文核心（数据模型 → 计算 → IndexedDB+加密 → 导入导出 → 图表 → benchmark → 论文素材）放在 Week 2–10，到 8 月 20 日**冻结成一个能演示、能写论文的完整原型**。NLP（Week 11）和 Agent（Week 12）作为**可砍的延伸周**——因为它们根本不在你的毕业论文方案里，是独立的学习目标。这样 Week 11–12 天然成为「缓冲带」：前面哪周延期了，就吃掉这两周，而不会动摇论文核心。

**调整二：把 IndexedDB 明确排进计划（Week 4）。**
你的论文 benchmark 要测到 **10 万笔交易**，localStorage 装不下这个量，所以 IndexedDB 是论文硬需求。Week1 保存层文档已统一为 `Repository -> StorageAdapter -> IndexedDB` 的正式路线，本计划在 Week 4 落地，并且**直接上 IndexedDB**（你已确认），不绕 localStorage 中间步骤。

其余沿用你总览里已经定好的东西：6 天节奏、每天 4 小时切分、跳票周机制、砍功能顺序。

## 1. 先回答几个范围判断问题（产品经理视角）

### Q1：这个项目 10 周（核心）的目标到底是什么？

一句话：**做出一个浏览器内、本地优先、加密存储、算得对、能导入导出、有图表、有性能数据、能支撑论文三章（System Design / Implementation / Evaluation）的个人交易账本原型。**

注意它**不是**「学完全栈 + NLP + Agent」。NLP 和 Agent 是锦上添花，论文不依赖它们。判断你暑假是否成功，只看一条：**8 月 20 日能不能从零启动这个账本、演示完整链路、并且有 benchmark 数据可以写进论文。**

### Q2：哪些内容必须进入核心计划（不可砍）？

| 必做 | 为什么不可砍 | 落在 |
| --- | --- | --- |
| 数据模型落地成代码 | 一切的地基 | W2 |
| 核心计算：持仓 / DCA 均价 / 已实现+未实现盈亏 / 超卖校验 | 论文 Implementation 的心脏，且必须手算可验证 | W2 |
| 计算接入页面，能真正添加交易并看到结果 | 否则只是一堆函数，不是「可演示原型」 | W3 |
| IndexedDB 本地持久化（走统一保存层） | 论文标题里的核心，benchmark 的前提 | W4 |
| 导入导出 JSON（含往返一致性） | 论文要求 + 忘记密码的逃生口 | W5 |
| AES-256-GCM + PBKDF2 加密 | 论文标题里的「encrypted」，安全章节的核心 | W6 |
| 至少两个图表（持仓饼图 + 资产增长曲线） | 论文 Visualization，演示必备 | W7 |
| 合成交易生成器 + benchmark（4 个量级） | 论文 Evaluation 的唯一实验数据，研究贡献所在 | W8 |
| 论文素材沉淀 + 冻结 + 演示脚本 | 把「做了东西」变成「能写论文」 | W9–10 |

### Q3：哪些内容应该推迟到 10 周之后（不进核心）？

- **NLP 自然语言输入** → 放 Week 11（延伸，可砍）。论文不依赖。
- **Agent 查账问答** → 放 Week 12（延伸，可砍）。论文不依赖。
- **实时价格 API、K 线、技术指标、策略回测、插件、多端同步、Tauri/Capacitor 打包** → 全部进 8 月 20 日之后的 backlog。暑假期间**只预留接口、不写实现**（接口 Week1 已设计好）。

这里有个反直觉但重要的点：**你越早承认 NLP/Agent 是可砍的，论文核心就越安全。** 原总览把它们排在 W7–8（演示之前），等于把两个最不确定的新功能放在最危险的位置——一旦前面延期，它们会反过来挤掉 benchmark 和论文整理的时间。本计划把它们挪到最后，就是为了保护核心。

### Q4：主线怎么拆？（Week 2–12 一览）

每一周只攻一个能独立验收的里程碑，后一周建立在前一周「已经能跑」的基础上：

```
W2 算得对（纯逻辑+测试）
  → W3 页面活起来（能加交易、看结果，内存态）
    → W4 数据存得住（IndexedDB，刷新还在）
      → W5 备份导得出导得回（JSON 往返一致）
        → W6 数据锁得住（AES 加密 + 解锁流程）
          → W7 看得见（持仓饼图 + 资产曲线）
            → W8 量得出（合成数据 + benchmark）
              → W9 讲得清（论文素材 + 修演示 bug）
                → W10 冻得住（冻结版本 + 演示脚本 + 复盘 + 路线图）★ 论文就绪里程碑
———————————————— 以下为可砍延伸 ————————————————
                  → W11 一句话变一笔交易（NLP 规则解析）
                    → W12 Agent 帮我查账（tool calling）
```

## 2. 节奏与时间模板

### 每周 6 天节奏（沿用总览）

| 日 | 角色 | 主要做什么 |
| --- | --- | --- |
| Day 1 | 计划与设计 | 明确本周交付物、拆任务、写验收标准、读一点要用到的资料 |
| Day 2–4 | 开发主战场 | 每天只攻一个核心任务，写完就测，不临时加需求 |

> Day 5 是每周的「缓冲日」：如果 Day 2–4 有任务没做完，Day 5 先补完再测。真正补不完的，才触发跳票周判断（见第 4 章）。

### 每日 4 小时模板（沿用总览，按周微调）

| 时段 | 默认 | 说明 |
| --- | --- | --- |
| 0.5h 启动 | 回看昨天日志，明确「今天只完成这一件事」 | 防止漫无目的 |
| 2.0h 深度开发 / 学习 | 写当天核心代码，或集中学一个马上要用的知识点 | 主战场 |
| 1.0h 测试 / 验证 | 跑测试、手算对比、真实输入验证 | 写完立刻验 |
| 0.5h 记录 | 今天做了什么、卡在哪、明天第一步 | 顺便攒论文素材 |

**学习 vs 开发的比例随阶段变化**（你是非技术背景，前期学得多很正常）：

| 阶段 | 学习占比 | 开发占比 | 说明 |
| --- | --- | --- | --- |
| W2–W3 | ~40% | ~60% | TS 模块化、测试框架、React 状态——新东西最多 |
| W4–W6 | ~30% | ~70% | IndexedDB、Web Crypto 要学，但有 Week1 设计兜底 |
| W7–W8 | ~25% | ~75% | Recharts、Playwright 上手快 |
| W9–W10 | ~15% | ~85% | 几乎全是整理、写作、修 bug |
| W11–W12 | ~40% | ~60% | NLP/Agent 是新领域，回到学习密集 |

> Day 1 的「读资料」和 2 小时里的「学知识点」算学习；其余算开发/测试/文档。每天那 0.5h 记录，本身就是在攒论文 Implementation 章的一手材料。

### Week 2（6/19–6/25）· 先让账本算得对

**目标**：把 Week1 的数据结构和计算口径，第一次变成能跑、能测、手算可验证的 TypeScript 代码。**这一周完全不碰页面好不好看**，只做纯逻辑 + 单元测试。

**起点衔接**：Week1 已写好 `Trade/Asset/PriceSnapshot/FeeRule/Position` 的字段设计和加权平均成本口径（见 `02-核心数据结构设计草稿.md`），也有 5 条样例交易（见 `01-5条假交易样例.md`）。本周把这些「纸上的类型」变成 `src/models` 里的真代码，把「纸上的算法」变成 `src/calculators` 里的真函数。

| 日 | 任务 |
| --- | --- |
| Day 1 | 〔学〕〔码〕按 Week1 扩展接口文档建好目录骨架：`src/{models,services,repositories,adapters,calculators,validators,utils}`。〔学〕选定 decimal 库（推荐 `decimal.js` 或 `big.js`），想清楚「保存用 DecimalString、计算用 decimal、展示再格式化」三段式。写本周验收标准。 |
| Day 2 | 〔码〕把 `Trade/Asset/PriceSnapshot/FeeRule/Position/LedgerData/TradeDraft` 类型落进 `src/models`。〔码〕写 `utils/decimalMath.ts`：加减乘除、比较、格式化，全部基于 decimal 库，业务代码里**不再出现裸 `quantity * price`**。 |
| Day 3 | 〔码〕写 `calculators/positionCalculator.ts` 的买入逻辑：从一组 `Trade` 算出每个资产的持仓数量、持仓成本 `costBasis`、加权平均成本 `averageCost`。**口径锁定**：`costBasis = Σ totalValue`（用记录的总额，因为真实成交会四舍五入），手续费第一版不计入成本。 |
| Day 4 | 〔码〕补卖出逻辑：卖出按当前平均成本结转，算 `realizedPnl`；补未实现盈亏：`marketValue = quantity × 最新价`，`unrealizedPnl = marketValue − costBasis`，最新价取该资产最新的 `PriceSnapshot`。〔码〕写 `validators/tradeValidator.ts`：数量>0、价格>0、fee≥0、资产存在、**卖出不超持仓**。 |
| Day 5 | 〔学〕〔测〕装 Vitest，写 golden 测试：输入 5 条样例交易，断言 ADA 持仓=251.2006、均价≈0.2468、成本=62 USD，BTC/ETH 各自数量与均价正确；加一条「卖出超持仓应报错」的用例；加一条「卖一部分后 realizedPnl 正确」的用例。全部跑绿。 |
| Day 6 | 〔文〕写 Week2 开发日志：计算口径的每个决定（为什么用 totalValue 当成本、为什么手续费不进成本、为什么平均成本法不用 FIFO）——**这段直接进论文 Implementation**。整理 Week3 任务。 |

📦 **产出物**：`models/` 类型代码、`decimalMath`、`positionCalculator`、`tradeValidator`、一套能跑绿的 Vitest 测试、Week2 日志。
✅ **验收**：给一组交易，代码算出的持仓/均价/盈亏与**手算完全一致**；卖出超持仓会被拦截；测试可重复运行。就算没有页面，计算也能独立跑通。
📄 **论文素材**：计算口径与取舍（DCA 加权平均 vs FIFO、手续费处理、DecimalString 精度策略）、golden 测试方法。
✂️ **落后时可砍**：未实现盈亏可推迟到 W3 和价格一起做；本周死守「买入/卖出/均价/超卖校验 + 测试」。

### Week 4（7/3–7/9）· 数据存得住（IndexedDB）

**目标**：让数据**刷新后还在**。按 Week1 保存层设计实现 `Repository → StorageAdapter`，存储后端**直接用 IndexedDB**，加密层先用 Noop 占位（接口立好、暂不真加密）。这是论文「typed repository layer over IndexedDB」的落地。

**起点衔接**：Week1 已设计好 `LedgerRepository` 和 `StorageAdapter` 接口、以及「组装点一行切换实现」的 DIP 思路（见 `05B_W1-保存层设计说明.md`）。本周把内存状态换成走这条保存层。

| 日 | 任务 |
| --- | --- |
| Day 1 | 〔学〕学 IndexedDB 基本模型（database/objectStore/事务/异步）。**建议用轻量封装库 `idb`** 降低异步复杂度。规划第一版「整坨 LedgerData 读写」（whole-blob）。 |
| Day 2 | 〔码〕实现 `adapters/indexedDbStorageAdapter.ts`：`read()` 返回整个 `LedgerData`（空库返回默认空结构、不崩），`write(data)` 整坨写回。 |
| Day 3 | 〔码〕实现 `repositories/ledgerRepository.ts`：`getLedgerData/listTrades/saveTrade/savePriceSnapshot/clearAll`，内部全部翻译成 adapter 的 read/write。建「组装点」工厂：`Noop 加密 + IndexedDB adapter + repository`。 |
| Day 4 | 〔码〕把 W3 的内存状态切换成「启动时从 repository 读、每次变更后写」。页面/Service 仍然不知道底层是 IndexedDB。 |
| Day 5 | 〔测〕验收：加几笔交易→刷新→数据还在；`clearAll` 后变空；用浏览器 DevTools 的 Application 面板看到 IndexedDB 里确实有数据。修 bug。 |
| Day 6 | 〔文〕日志：保存层三件套如何分工、为什么页面碰不到存储、换存储为什么只改组装点一行——**进论文 Implementation 的存储架构**。整理下周。 |

📦 **产出物**：`indexedDbStorageAdapter`、`ledgerRepository`、`noopEncryptionService`、组装点工厂；持久化后的可用应用。
✅ **验收**：刷新页面数据仍在；页面和 Service 没有任何一行直接操作 IndexedDB；交易/统计结果与持久化前一致。
📄 **论文素材**：可替换存储层的接口设计、whole-blob 第一版策略及其未来分片优化路线。
✂️ **落后时可砍**：本周是论文硬需求，**不可砍**。若 IndexedDB 卡壳，缩小 IndexedDB 实现范围，只保留 whole-blob 读写和最小 repository；必要时保留 W3 内存态演示，但不把 localStorage 纳入正式路线。

### Week 5（7/10–7/16）· 备份导得出、导得回

**目标**：实现 JSON 备份的**导出 / 导入 / 往返一致**。这既是论文要求的 import/export，也是 Week1 拍板的「忘记密码逃生口」——导出的是**明文** JSON。

**起点衔接**：保存层已就位，导出导入只需在 Repository 上加 `exportData/importData`，并接两个页面按钮；导入要复用 W2 的 `tradeValidator`。

| 日 | 任务 |
| --- | --- |
| Day 1 | 〔码〕`ledgerRepository.exportData()`：读出 `LedgerData` → `JSON.stringify` → 触发浏览器下载一个 `.json` 文件（学 Blob + 下载）。 |
| Day 2 | 〔码〕「导出备份」按钮接通；导出时弹明确警告：「此备份未加密，请妥善保管」。 |
| Day 3 | 〔码〕`importData(json)`：解析 → 逐条过 `tradeValidator` → 覆盖写回。坏数据/格式错误要给清晰提示，不写入。 |

### Week 6（7/17–7/23）· 数据锁得住（真加密）

**目标**：把 Noop 换成**真加密**：AES-256-GCM + PBKDF2 口令派生，打开账本要输密码，本地存的是密文。验证「换加密只改组装点一行、上层零改动」。**这是核心里最难的一周**，所以前面留了余量、后面有缓冲带。

**起点衔接**：Week1 加密路线（`05C`）已把算法、密钥派生、错误处理、忘记密码兜底全部设计好。本周是照着设计实现 `WebCryptoEncryptionService`。

| 日 | 任务 |
| --- | --- |
| Day 1 | 〔学〕学 Web Crypto API：`crypto.subtle` 的 `importKey/deriveKey(PBKDF2)/encrypt/decrypt(AES-GCM)`、salt 与 iv 的作用。 |
| Day 2 | 〔码〕实现 PBKDF2：口令 + 随机 salt + 迭代次数 → 256 位密钥。 |
| Day 3 | 〔码〕实现 `encrypt/decrypt`：输出打包 `salt + iv + 密文`；`decrypt` 拆包重派生密钥；密码错/数据被改 → 抛统一「解锁失败」。 |
| Day 4 | 〔码〕加「输入密码解锁」流程；组装点把 `Noop` 换成 `WebCryptoEncryptionService`，确认页面/Service/Repository/Adapter **一行都不用改**。 |
| Day 5 | 〔测〕验收：DevTools 里看到 IndexedDB 存的是密文不是明文；密码对能解、错会提示；改坏密文解密失败；**导出仍是明文**逃生口可用。顺手测离线（DevTools 切 offline）账本仍能用。 |
| Day 6 | 〔文〕日志：加密分层（只有最底一道门碰加密）、GCM 防篡改、密钥不落盘、错误统一化——**进论文 Implementation 安全部分 + Evaluation 的「security by default」理想**。整理下周。 |

📦 **产出物**：`webCryptoEncryptionService`、解锁流程、加密后的可用应用、离线可用记录。
✅ **验收**：本地存储为密文；正确密码可解锁、错误密码被拒；篡改数据解密失败；导出明文备份仍可恢复；断网仍可用。
📄 **论文素材**：加密方案与密钥派生设计、往返完整性（byte-for-byte）、离线能力证据、对照 7 条 local-first 理想中的「security & privacy by default / the long now / optional network」。
✂️ **落后时可砍**：若 Web Crypto 卡死，可先交「明文 IndexedDB + 完整加密设计文档」保住演示与论文骨架，加密实现挪到缓冲带 W11 前补；但加密是论文标题词，尽量不砍。

## 4. 风险、优先级与压缩预案

### 4.1 优先级（落后时按这个保）

- **P0（命脉，绝不砍）**：数据模型、核心计算（持仓/均价/盈亏/超卖）、IndexedDB 持久化、导入导出、加密、benchmark、论文素材沉淀。少了任何一个，毕业论文都不成立。
- **P1（重要，能压缩但尽量留）**：第二/第三个图表、benchmark 的 100,000 量级、UI 美化。
- **P2（加分，可整块砍）**：NLP（W11）、Agent（W12）、交易点标注图、实时价格 API 等一切 backlog。

### 4.2 主要风险与对策

| 风险 | 触发点 | 对策 |
| --- | --- | --- |
| 加密最难、易卡死 | W6 Web Crypto | 已把它排在中段、后有缓冲带；卡死则先交「明文+加密设计文档」，实现挪后补 |
| IndexedDB 异步难 | W4 | 用 `idb` 封装库；实在不行缩小为 whole-blob 最小读写，保留内存态演示，不引入 localStorage 正式路线 |
| 范围膨胀（想多做） | 全程 | NLP/Agent 已隔离到最后；backlog 只记不做；Day 2–4 不临时加需求 |
| 进度落后 | 任意周 | Day 5 缓冲 → 跳票周 → 砍功能，三级处理（见 4.3） |
| UI 完美主义 | W3/W7/W9 | UI 属 P1，「能看懂、能演示」即可，先功能后好看 |
| 小数精度 bug | W2 起 | 全程 DecimalString + decimal 库，业务层禁止裸算 |
| 10 万笔内存/卡顿 | W8 | 100k 是 P1，可先砍；whole-blob 若太慢，按 Week1 设计上分片缓存 |
| 非技术背景上手慢 | 前期 | 前 3 周学习占比高是正常的；每天只攻一件事，写完即测 |

### 4.3 落后了怎么办：三级处理

**第一级——Day 5 缓冲**：当周 Day 2–4 没做完的，Day 5 先补再测。多数小延期到这里就消化了。

**第二级——跳票周**（沿用总览）：某周结束时若有 **4 项及以上**本周计划成果未完成，下一周不做新主题，专门补欠下的 P0。补完后整体顺延一周。每个自然月最多 1 次；8 月 7 日后不再启动（要留给 W9–10 收尾）。

**第三级——砍功能**（用完跳票周仍延期，按此顺序砍）：

1. 先砍 NLP（W11）和 Agent（W12）——它们本就是缓冲带。
2. 再砍高级/第三个图表，只保留持仓饼图 + 资产曲线两张核心图。
3. 再砍 benchmark 最大规模，100,000 笔延后，保 100/1k/10k。
4. 再砍 NLP 只剩中文新增交易、Agent 只剩 3 个固定问题（如果还没整块砍掉的话）。
5. **绝不砍**：核心账本计算、IndexedDB 持久化、加密、导入导出、benchmark 基本盘、论文素材。

### 4.4 「如果计划还是太满」——主动压缩成 9 周核心

如果到 W5 末就发现明显吃力，立即启用压缩版（牺牲打磨、保住论文）：

- W5 导入导出**只做整库覆盖**，砍合并逻辑。
- W7 图表**只做一张饼图**，资产曲线留到 W9 有余量再补。
- W9 三章**只写提纲 + 图**，正文留到 9 月写作期。
- W11–W12 **直接不做**，NLP/Agent 进 9 月课程项目。

这样最差情况下，Week 2–10 仍交付一个「算得对 + 存得住 + 锁得住 + 导得出 + 一张图 + benchmark + 论文素材」的完整原型——**毕业论文照样成立**。
