# 编程学习整理助手

当 Ethan 在其他项目中学习、写代码、调试、读代码或使用 AI 编程时，使用这个 skill。

这个 skill 不负责直接把内容写入 `编程宝典`。

它的目标是：把别的项目中的学习过程整理成一个清晰的「待入库学习包」，之后再交给 `programming-knowledge-ingest` 导入 `编程宝典`。

## 使用场景

当用户说类似下面的话时，使用这个 skill：

- 帮我整理一下我刚刚学到的东西
- 把这个项目里的学习过程总结一下
- 我刚刚调试这个 bug 学到了什么？
- 这段 AI 写的代码我终于看懂了，帮我整理成笔记
- 帮我生成一个之后可以导入编程宝典的学习包
- 我在这个项目里学到了某个技术，帮我沉淀一下

## 工作流程

### 1. 确认学习背景

先明确这次学习发生在哪里。

需要提取：

- 项目名称
- 涉及的功能、bug、文件、命令、对话或代码路径
- Ethan 一开始哪里不懂
- 这次学习是通过什么发生的：调试、读代码、问 AI、查文档、重构、写功能等

### 2. 提取真实痛点

不要只总结“学了什么技术名词”，要先找到背后的问题。

需要回答：

- 在理解这个技术之前，遇到了什么问题？
- 为什么这段代码或这个项目会让人觉得像 black box（黑盒）？
- 如果只是复制 AI 给出的代码，而不理解它，会带来什么维护风险？
- 这个知识点为什么值得进入 `编程宝典`？

### 3. 提取候选卡片

把学习内容拆成三类候选内容。

#### 技术卡片候选

稳定、可复用、以后还会遇到的技术概念。

例如：

- `Closure（闭包）`
- `Promise（承诺）`
- `Virtual DOM（虚拟DOM）`
- `Tokenization（分词）`
- `Dependency Injection（依赖注入）`

#### 问题卡片候选

从痛点出发的问题。

例如：

- 为什么需要 X？
- X 解决了什么问题？
- 没有 X 之前怎么做？
- 为什么某段代码会变成 black box？
- 为什么这个 bug 很难调试？

#### 项目学习记录候选

记录这次学习发生在哪个真实项目里。

例如：

- `26_06_29 账本项目 - 理解 Zustand 状态管理`
- `26_07_01 Todo 项目 - 理解 React useEffect 的依赖数组`

### 4. 使用中文逻辑 + 英文术语

整理时遵守 `编程宝典` 的语言规则：

- 用中文解释背景、痛点、推理和学习反思
- 技术名词优先保留英文
- 第一次出现技术名词时使用 `English（中文）` 格式
- 后续可以直接使用英文术语

例如：

`Tokenization（分词）` 是把文本切成 token 的过程。LLM 不直接处理自然语言字符串，而是处理 token 序列。
